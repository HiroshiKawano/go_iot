## アプリケーション内部アーキテクチャ（パッケージ層）

> 上図はデプロイ階層（デバイス／バックエンド／フロント）の構成。ここでは **バックエンド内部の Go パッケージ設計方針** を示す。**詳細ルールの正（single source of truth）は [steering/structure.md「依存方向ルール」](../.kiro/steering/structure.md) と [steering/tech.md「データアクセス方針」](../.kiro/steering/tech.md)** であり、本節はその要約（2026-06-03 決定）。

- **採用方針: 実務的 Layered-lite**（layer-first + 画面=feature ファイル分割）。**厳格 Clean Architecture は不採用**。理由: SSR+templ で JSON API を持たず Presenter/DTO 層が空振りする・CRUD のボイラープレートが過多・ほぼ個人開発／スピード重視／cc-sdd と不整合なため。一方 `domain` は既に純粋で DB 境界（Querier）は無償でモック可能＝Clean の主要メリットは安価に回収できる中立点。
- **依存は下向き一方向**（逆流・循環禁止）:
  ```
  cmd（合成ルート） → handler / middleware / auth → service → repository → infra
                                                 └→ domain（純粋・無依存。handler/auth/service など上位から参照のみ）
  ```
  下向きなら隣接層スキップ可（単純 CRUD は `handler → repository` 直行）。`cmd/server/main.go` が唯一の合成ルート。
- **DIP は2点限定**（むやみに interface を増やさない）: ① **DB ポート = `repository.Querier`**（sqlc `emit_interface=true` の生成 interface。handler/auth は具象 `*repository.Queries` ではなく Querier に依存し、テストでモック可）。② service の consumer interface は必要時のみ。
- **view（templ）→ repository / service を呼ばない**（handler が渡した表示用データのみ描画。`domain` の表示メソッド `Label()` / `Unit()` 等のみ許可）。
- いつ `service` 層を挟むか（＝層の厚みの線引き基準）は、最初の1-2画面で SSR+HTMX 依存フローを検証してから steering に追記する（現時点は意図的に未定義）。

## 構成の詳細

### 🔷 デバイス層（ESP8266 / ESP-WROOM-02 + 温湿度センサー）

> **2026-06-22 仕様変更**: マイコンを **ESP32 → ESP8266（ESP-WROOM-02 系）** に変更（実機の刻印・構成から確定）。ESP8266（Xtensa lx106）は TinyGo が実質非対応のため、ファームウェアは **Arduino C++（`ESP8266WiFi`）** で実装する（言語統一の前提が変わった点は [実装計画.md](実装計画.md) の「デバイス連携・現地実証フェーズ」を参照）。センサー型番（SHT31 / DHT22 / BME280 等）は **要確認**。

**技術スタック:**
- Arduino C++（`ESP8266WiFi` / `ESP8266HTTPClient` / `WiFiClientSecure`）
- I2C 通信プロトコル（温湿度センサー）
- HTTPS クライアント

**役割:**
- 温湿度センサーから I2C 通信で温湿度データを取得
- REST API（HTTPS POST + JSON）で Go バックエンドへデータ送信
- Bearer トークンで認証

**通信方式:**
- Device → Server: REST API (HTTPS POST / JSON)

---

### 🔶 バックエンド（Go 1.26 + Gin + templ - モノリス構成）

**技術スタック:**
- Go 1.26
- Gin (`github.com/gin-gonic/gin` v1)
- templ v0.3 (`github.com/a-h/templ`)
- PostgreSQL 16 + pgx/v5 (`github.com/jackc/pgx/v5`)
- sqlc v1.30 (SQL → Go コード生成)
- goose v3 (DB マイグレーション)
- alexedwards/scs v2 (Web UI Session)
- go-playground/validator v10 (バリデーション)
- Scalar UI + OpenAPI 3.0 (API ドキュメント)

**役割:**
- デバイスからのデータ受信・保存
- センサーデータのグラフを SVG としてサーバーサイドで生成
- templ テンプレートへのデータ供給（Handler からコンポーネント関数を呼び出し）
- HTMX 向け部分 HTML レスポンスの生成（templ コンポーネント関数を直接 `.Render()`）
- デバイス API 専用のドキュメント化（Scalar + OpenAPI）

**通信方式:**
- Device → Server: REST API（自作 Bearer ミドルウェアで認証）
- Server → Frontend: グラフ SVG を含む templ レンダリング済み HTML（scs Session で認証）
- Frontend → Server: HTMX による HTTP GET/POST/PUT/DELETE/PATCH（部分 HTML 返却）

**特徴:**
- フロントエンド用 JSON API は不要（templ が HTML を直接返す）
- グラフ描画も含めてすべてサーバーサイドで完結
- デバイスが叩くエンドポイントのみをドキュメント化

---

### 🔸 フロントエンド（templ + HTMX + Alpine.js）

**技術スタック:**
- templ（サーバーサイド HTML テンプレート、コンパイル済みの Go 関数）
- HTMX
- Alpine.js（UI の開閉・タブ切替等の軽量インタラクション）

**役割:**
- HTML モックを templ に組み込み、サーバーから受け取った HTML を表示
- HTMX の `hx-get` / `hx-post` / `hx-put` / `hx-delete` / `hx-patch` でデバイス制御・部分更新
- Alpine.js による軽量な UI 制御（メニュー開閉、タブ切替、削除確認モーダル等）

**特徴:**
- JavaScript フレームワーク不要（HTMX と Alpine.js の属性ベース記述のみ）
- グラフ含むすべての表示ロジックをサーバーサイドに集約
- フロント担当者は HTML モック作成のみ、実装はすべて Go 側で実施
- templ コンポーネント関数分割により、部分テンプレートファイルを分けず、フルページテンプレート内で部分更新領域を定義

## データフロー

### センサーデータ収集フロー

```
温湿度センサー → (I2C) → ESP8266 → (HTTPS POST + Bearer Token) → Gin Handler → CreateSensorReading → PostgreSQL
```

### データ表示フロー

```
PostgreSQL → sqlc クエリ → Handler → サーバサイド SVG 生成 → templ コンポーネント → (HTML + SVG) → ブラウザ
```

### デバイス制御フロー

```
templ (hx-post) → (HTMX HTTP POST) → Gin Handler → PostgreSQL → templ 部分コンポーネント → DOM 更新
```

## 認証方式

| 接続元 | 認証方式 | 用途 |
|--------|----------|------|
| ESP8266 | 自作 Bearer トークンミドルウェア (SHA-256 ハッシュ照合) | センサーデータ送信 |
| ブラウザ | Session（alexedwards/scs + PostgreSQL セッションストア） | Web UI 操作 |

## ルートと Middleware のマッピング

Gin のルーターグループ機能でルートとミドルウェアを分離する。

| ルーターグループ | 適用ミドルウェア | 対象 | 認証ミドルウェア |
|---|---|---|---|
| `/api/*` (`r.Group("/api", ...)`) | Logger, Recovery, **DeviceAuth** | ESP8266 デバイス API（Scalar ドキュメント対象） | 自作 Bearer (`internal/auth/device_auth.go`) |
| `/ (non-api)` | Logger, Recovery, **SessionLoad**, **CSRF**, （http層）**MethodOverride** | ブラウザ UI・HTMX リクエスト | scs Session (`internal/auth/session_auth.go`) + gorilla/csrf。保護ルートは `RequireAuth` |
| `/docs`, `/docs/openapi.yaml` | Logger, Recovery のみ | Scalar API ドキュメント公開 | なし |
| `/health` | Logger, Recovery | 外形監視 | なし |

> HTMX の `hx-get` / `hx-post` 等のリクエストも Web グループを経由し、Session 認証が適用される。
> デバイス API と Web UI で Gin のルーターグループが分離されているため、ミドルウェアの適用範囲が明確に分かれる。
> Logger と Recovery は `r.Use()` でグローバル適用するため、`/docs` や `/health` を含む全ルートに適用される（グループ固有のミドルウェアは認証・MethodOverride 等）。

## エラーレスポンス方針

| リクエスト種別 | エラー応答形式 | 実装方式 |
|-------------|-------------|---------|
| デバイス API (`/api/*`) | JSON | `c.AbortWithStatusJSON(4xx/5xx, gin.H{"message": ...})` で直接 JSON 返却 |
| HTMX リクエスト（部分更新） | 部分 HTML（エラーメッセージコンポーネント） | templ のエラー用コンポーネントを `.Render()` してステータス 4xx で返す |
| フルページリクエスト（初回ロード） | Go 製エラーページ | `internal/view/page/Error.templ` をレンダリング |

> HTMX リクエストか否かは `c.GetHeader("HX-Request")` で判定可能。
> 認証エラー（401）は API は JSON 返却、Web はログインページへリダイレクト (`c.Redirect(http.StatusFound, "/login")`)。
