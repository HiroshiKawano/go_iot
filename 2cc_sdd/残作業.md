## B. ESP8266 実機への書き込み（物理作業＝人手／周辺は AI 支援）

ハードウェア（ESP8266 + SHT31）に参考ファームウェアを書き込み、自前サーバの `POST /api/sensor-data`（HTTPS + Bearer）へ実データを送れる状態にする。**A セクション（Lightsail への HTTPS 公開・デバイス登録・トークン発行）が完了していることが前提**であり、A の成果物（本番 FQDN・`DEVICE_ID`・平文 Bearer トークン）を本セクションの入力として使う。

> **このセクションの「AI 主導」の限界（最重要・誠実に明記）**
> 本書全体は「人がコマンドを直接打たず、AI と対話して AI が MCP で実行する」運用が主眼だが、**B は本質的に物理作業を含むため AI に全部は任せられない**。次の切り分けを前提とする。
>
> | 区分 | 担当 | 具体 |
> |---|---|---|
> | **物理作業（AI 不可・人手）** | **河野さん（現地/書込担当）** | SHT31 とボードの**配線**、USB ケーブル接続、Arduino IDE での**コンパイル＆書き込み（マイコンへの書込）**、FLASH/BOOT + RESET のボタン操作、シリアルモニタを開く操作 |
> | **AI が支援できる範囲** | AI（対話＋ローカル実行） | ① `config.h` の値生成（`API_ENDPOINT` / `API_BEARER` / `DEVICE_ID` の組み立て）、② スケッチ・`config.h` の**コンパイル前チェック**（マクロ・型・URL 形式の机上点検）、③ シリアルログ／HTTP ステータス／コンパイルエラーの**解釈と切り分け**、④ 書込前に **`cmd/sensor-sim` で本番疎通を AI に確認させる**（実機を焼く前にサーバ側を黒判定で除外） |
>
> ⚠️ **AWS MCP（AWS API MCP Server）は B では使わない**。B の操作対象は AWS リソースでも Lightsail インスタンス内でもなく、**手元 PC の Arduino IDE と物理デバイス**だからである。AI が B で「実行」できるのは、ローカルシェル経由の `go run ./cmd/sensor-sim`（疎通テスト送信）程度に限られる（後述 B-4。これは AWS MCP ではなくローカルの Go ツールチェイン実行であり、人の承認下で行う）。本セクションの大半は「AI が値を作り・点検し・読み解く」対話支援であって、書込そのものは人が行う。

> **▶ 実機実績で判明（2026-06-23・付録F）— 実際の分担はもっと AI 寄りにできた**
> 上表は着手前の見積りだが、実際には **AI が次まで実行できた**: ① 実機 MAC 取得（`esptool read-mac`）② デバイス登録（chrome-devtools MCP でブラウザ操作）③ トークン発行（SSH でサーバ上 `gen-token`）④ ビルド（`arduino-cli compile`）⑤ **書き込み（`esptool` v5.3.0 で `write-flash`）** ⑥ I2C アドレス特定（`i2c_scanner` 書込＋スキャン）⑦ シリアル採取（`firmware/tools/serial_capture.py`）⑧ `sensor-sim` 疎通。
> **人（河野さん）に残るのは物理操作のみ**＝配線・USB/FTDI の接続・**書込モード/起動のボタン操作（GPIO0 長押し+RST ／ 物理 RST）**・FTDI の抜き差し。**Arduino IDE GUI は必須ではない**（むしろ GUI/arduino-cli 同梱の esptool.py v3.0 は書込が不安定なので、**CLI の esptool v5.3.0 を推奨**＝B-5・付録F-7）。

対象ファイル: `firmware/esp8266_sht31/esp8266_sht31.ino` / `firmware/esp8266_sht31/config.example.h`（詳細は `firmware/README.md`）。なお実機で確定した値・実出力・つまづきの記録は**付録F**（B章はそちらを正とする）。

> **前提の念押し**: `API_ENDPOINT` は `https://`（A で Caddy が 80/443 経由で張った本番 FQDN）であること。アプリは平文 HTTP の `:8080` でしか listen しておらず、`8080` / `5432` はパブリック非公開のため、ESP8266 から直接到達できるのは Caddy の `443` のみ（A で確立した不変条件）。

### B-1. 必要なもの・配線（SHT31 I2C）— **人手（河野さん）**

**準備するもの**:
- **ESP8266 / ESP-WROOM-02 系ボード**（NodeMCU・WeMos D1 mini 等。基板に `3V3`/`GND`/`D1`/`D2` のシルク印刷があるもの）。
- **SHT31 温湿度センサーモジュール**（ピンに `VIN`(または`VCC`)/`GND`/`SCL`/`SDA`/`ADDR` のシルク印刷。汎用/Adafruit 製はプルアップ抵抗内蔵が多い）。
- **ジャンパワイヤ 4 本**: ボード／モジュールのピンが**オス（突起）なら メス–メス**、ブレッドボードに挿すなら オス–オス。**実物のピン形状を見て選ぶ**。
- （任意）ブレッドボード（半田付けを避けたい場合）、テスター（電圧確認用）。
- **データ通信対応の Micro USB ケーブル**（充電専用だと PC がポートを認識しない＝最頻トラブル）。

SHT31 を I2C で配線する（NodeMCU 既定ピン。基板の刻印に合わせて後述 `config.h` で調整）。**両基板のシルク印刷（ピン名）を必ず確認**してから挿す。

| SHT31 側 | ESP8266 側 | 備考 |
|---|---|---|
| VIN / VCC | **3V3** | 5V を繋がない（3.3V 系） |
| GND | **GND** | 共通グラウンド |
| SDA | **GPIO4（D2）** | `I2C_SDA=4` |
| SCL | **GPIO5（D1）** | `I2C_SCL=5` |

SHT31 の既定 I2C アドレスは `0x44`（`ADDR` ピンを High=3V3 に繋いだ場合のみ `0x45`。**未接続なら `0x44`**）。モジュールは通常 SDA/SCL にプルアップ抵抗を内蔵するが、認識しない場合は 4.7kΩ プルアップが要ることもある（要確認）。
> ▶ **実機実績（付録F-8）**: 本案件の実機は **`0x45`** だった（`[SHT31] not found at 0x44` で発覚）。未検出時は I2C スキャナ `firmware/i2c_scanner/`（全ピン×全アドレス総当たり）で実アドレス/ピンを特定し、`config.h` の `SHT31_ADDR` を合わせる。

#### 🙋 人（河野さん）が行う結線手順
1. **USB を PC に挿す前（通電前）**に配線する。両基板のシルク印刷（ピン名）を確認し、形状に合うジャンパワイヤを用意。
2. SHT31 の **`VIN`/`VCC` → ESP8266 の `3V3`**。**`5V`(VU/5V 系ピン) には絶対に繋がない**（SHT31 は 3.3V 系・5V でセンサーが壊れる）。
3. SHT31 の **`GND` → ESP8266 の `GND`**（共通グラウンド。GND ならどのピンでもよい）。
4. SHT31 の **`SDA` → ESP8266 の `GPIO4`（基板刻印 `D2`）**。
5. SHT31 の **`SCL` → ESP8266 の `GPIO5`（基板刻印 `D1`）**。
6. 4 本が奥まで挿さっているか、隣ピンと触れていないか・緩みが無いかを目視。`ADDR` は未接続のまま（=`0x44`）。
7. （任意・テスターがあれば）USB 接続後に **`3V3`↔`GND` 間が約 3.3V** か、`5V`↔`GND` を誤接続・ショートしていないかを確認。
8. 既定ピン（D2/D1）以外に繋いだ場合は、実際に使った SDA/SCL の GPIO 番号を AI に伝える（B-3 の `config.h` に反映）。

#### 🤖 AI が支援できること
- 配線表の確認チェックリスト（VCC=3V3 か／5V を誤接続していないか／SDA・SCL の取り違えがないか）を提示する。
- 河野さんから「SDA/SCL を別ピンに繋いだ」と報告があれば、B-3 の `I2C_SDA` / `I2C_SCL` の値を**それに合わせて生成し直す**（配線と `config.h` の整合を AI が担保）。

#### ✅ 完了判定
- 4 本の結線完了（VCC=3V3・GND・SDA=D2・SCL=D1、または申告した代替ピン）、ショート無し。AI 側に「実際に使った SDA/SCL のピン番号」が共有されている（B-3 の値生成に必要）。

### B-2. Arduino IDE のセットアップ（ボード定義・ポート・ライブラリ）— **人手（河野さん）**

> 前提: **Arduino IDE 本体**を [arduino.cc](https://www.arduino.cc/en/software) から入手・インストール済み（バージョンは要確認。2.x 系推奨）。

**0. （初回のみ）USB シリアルドライバを入れる**
- ESP8266 ボードは USB–シリアル変換チップ（基板により **CP2102（Silicon Labs）** / **CH340（WCH）** / **FTDI** のいずれか）を載せている。**基板上のチップの刻印**で見分ける。
- 該当ドライバを OS に入れる（**「チップ名 + driver」で公式を検索**して入手・URL は要確認）:
  - CP210x → Silicon Labs「CP210x USB to UART Bridge VCP Drivers」
  - CH340/CH341 → WCH 公式
  - FTDI → FTDI「VCP Drivers」
- **macOS** は近年の版で CH340/CP210x が OS 標準対応のこともある（要確認）。**Windows** は基本ドライバ導入が要る。入れたら PC を再起動。

1. **ESP8266 ボード定義を追加**。`ファイル → 環境設定 → 追加のボードマネージャURL` に次を貼る:

   ```
   http://arduino.esp8266.com/stable/package_esp8266com_index.json
   ```

   続けて `ツール → ボード → ボードマネージャ` で **esp8266 by ESP8266 Community** を検索・インストール → `ツール → ボード` で **Generic ESP8266 Module**（実機に合うものがあればそれ）を選ぶ。

2. **ポートを選ぶ**。USB 接続後、`ツール → ポート` で対象を選択:

   ```
   Windows: ツール → ポート → COM7 等（デバイスマネージャの「ポート(COM & LPT)」で番号を確認）
   macOS:   ツール → ポート → /dev/cu.usbserial-xxxx
            （または /dev/cu.SLAB_USBtoUART, /dev/cu.wchusbserial-xxxx）
   ```

   **ポートが出てこない時の対処**（上から順に試す）:
   - **充電専用ケーブルを疑う** → データ通信対応に交換（最頻原因）。
   - **ドライバ未導入** → 手順0。基板のチップを確認して入れ直す。
   - **USB ポートを変える**（ハブ経由を避け PC 本体に直挿し）。
   - **IDE を再起動／ケーブルを挿し直す**。
   - Windows はデバイスマネージャで「不明なデバイス」「！」マークが出ていないか確認。

3. **必要ライブラリをインストール**。`スケッチ → ライブラリをインクルード → ライブラリを管理` から次を入れる:

   ```
   Adafruit SHT31 Library
   （依存の Adafruit BusIO / Adafruit Unified Sensor も一緒にインストール）
   ```

   JSON は標準 `snprintf` で組み立てるため ArduinoJson は不要。

#### 🤖 AI が支援できること
- 河野さんから「ポートが出てこない」「ボードマネージャでインストールに失敗」等のエラー文・症状を聞き、**原因の切り分けと対処を提示**する（例: ポート不可視→充電専用ケーブル疑い／ドライバ未導入、ライブラリ依存欠落→ BusIO・Unified Sensor の同時導入を案内）。
- 必要ライブラリの抜けがないかを `esp8266_sht31.ino` の `#include` から逆算して列挙する（Read で `.ino` を読んで確認）。`.ino` の include は `ESP8266WiFi.h` / `ESP8266HTTPClient.h` / `WiFiClientSecure.h` / `Wire.h`（いずれも esp8266 ボードパッケージ同梱）と `Adafruit_SHT31.h`（要ライブラリ導入）。

#### 🙋 人（河野さん）が行う点
- 上記 0〜3 の操作すべて（ドライバ導入は OS 管理者権限が要る場合あり）。

#### ✅ 完了判定
- ボードが選べ、`ツール → ポート` に対象ポートが見え、`Adafruit SHT31 Library`（+依存）が導入済み。

### B-2.5. MAC アドレス取得 → デバイス登録 → device_id・トークン確定（**順序が最重要**）

B-3 の `config.h` に書く `DEVICE_ID` と `API_BEARER` は、**この実機の MAC アドレスで登録したデバイス**に紐づく。MAC は物理デバイスからしか得られないので、**B-3 の前に必ずこの順で確定**させる（鶏卵関係。A-4 で仮 MAC で登録済みなら実機 MAC に更新）。実出力・詳細は付録F-1〜F-3。

1. **MAC アドレスを取得**（スケッチ書込不要・付録F-1）。AI が esptool で読む:
   ```bash
   python3 -m venv esptool-venv && esptool-venv/bin/pip install esptool   # 一度だけ（v5.3.0）
   # 【人】GPIO0 ボタン長押し → RST 一度押して離す → GPIO0 離す（＝書込モード）
   esptool-venv/bin/esptool --port /dev/cu.usbserial-XXXX --before no-reset --chip esp8266 read-mac
   # → 出力 "MAC: 24:62:ab:1c:e7:65" が登録に使う値（登録フォームは大文字へ正規化）
   ```
   - `Invalid head of packet (0x2E)` が出たら**書込モードに入っていない**（チップが通常起動中）→ ボタン操作をやり直す。
   - ポート名は `ls /dev/cu.*` で確認（FTDI=`usbserial-` / CP210x=`SLAB_` / CH340=`wchusbserial`）。
   - （別法）ファーム書込後なら起動ログ `[WiFi] MAC: XX:XX:...` でも読める（B-5 後）。
2. **Web UI でデバイス登録 → device_id 発番**。本番 `https://<FQDN>/devices/create`（ログイン要・一覧は `/dashboard`・`/devices` 単体は 404）で MAC・名前・場所を入力。AI が chrome-devtools MCP で代行可（ブラウザが本番ログイン済みであること）。登録後にリダイレクトされる URL `…/devices/<N>` の `<N>` が **device_id**。
3. **トークン発行**（本番 DB に対して・付録F-3）。AI が SSH でサーバ上の `gen-token` を実行（手元の Go では本番 DB に届かない）:
   ```bash
   ssh ubuntu@<静的IP> 'sudo bash -c '"'"'
     DBURL=$(grep -E "^DATABASE_URL=" /opt/go_iot/go_iot.env | cut -d= -f2-)
     DATABASE_URL="$DBURL" SESSION_SECRET=dummy /opt/go_iot/go_iot_gen-token -user=<登録ユーザーID> -name="<デバイス名>"
   '"'"''
   ```
   - **device_id の所有者 = トークンの user** を一致させる（不一致は B-5 で 403）。**平文トークンは発行時 1 回のみ表示** → ファイルへ保存（チャット/コミットに出さない）。

→ ここで確定した `device_id` と平文トークンを、次の B-3（config.h）に入れる。

#### ✅ 完了判定
- 実機 MAC を取得済み・その MAC で登録した `device_id` が判明・その所有ユーザーで平文トークンを発行済み（3 点が揃って B-3 へ）。

### B-3. config.h の値を AI に生成させる（**AI 支援の中核**）

ここが B における AI 支援の主役。`config.example.h` を `config.h` にコピーするのは河野さん（または AI が文面を提示し人が貼る）だが、**埋める値（`API_ENDPOINT` / `API_BEARER` / `DEVICE_ID` ほか）は AI に生成させる**。`config.h` は機密（Wi-Fi パスワード・Bearer トークン）を含むため `.gitignore` 済み（`firmware/**/config.h`）で、**コミットしてはいけない**。

#### ▶ AI への依頼（プロンプト例）

**今回の実機で確実だったのはこちら**（GUI で `Invalid head of packet` / stub 失敗 / ブートローダに入れない時はこれに切替。最初からこちらでも可）。**AI がコマンドを実行し、人はボタン操作だけ**行う。詳細・実出力は付録F-5/F-7/F-9。

- **書込中にリセット・電源落ち**: USB 給電不足 → PC 本体ポートに直挿し／セルフパワー USB ハブを使う。
- **書けたがシリアルが文字化け**: シリアルモニタ右下のボーレートが **115200** か確認。

> 「シリアルモニタにこう出た（ログを貼る）。何が原因でどう直す？」

#### 🤖 AI が行うこと（ログ解釈・MCP 操作なし）
- 河野さんが貼ったシリアルログ／HTTP ステータスから原因を切り分ける（HTTP エラー時の実ログ行は `[HTTP] <code> (error). resp=...` 形式）:
  - **`[HTTP] 201 Created OK`** = 成功（B-5 の判定）。
  - `[HTTP] 401 (error)` = Bearer トークン不正/未設定 → `API_BEARER` を確認（1 文字違いも疑う。B-3 に戻す）。
  - `[HTTP] 403 (error)` = 他ユーザーのデバイス → `DEVICE_ID` とトークン発行ユーザーの対応を確認。
  - `[HTTP] 422 (error)` = 存在しない/削除済み `device_id`、またはバリデーションエラー。
  - `[HTTP] 400 (error)` = JSON 不正、`[HTTP] 500 (error)` = サーバ DB。
  - `[SHT31] not found at 0x..` = 配線/アドレス（B-1, `SHT31_ADDR`）。
  - `[WiFi] connect failed (timeout)` = Wi-Fi 設定（SSID/パスワード）または電波。
  - `[HTTP] transport error: ...` = `API_ENDPOINT` の HTTPS/証明書まわり（`TLS_INSECURE` 設定・FQDN・Caddy の証明書・名前解決）。
  - `[loop] skip: time not synced` = NTP 未同期（時刻同期できない周期は送信スキップ。`[NTP] synced` が出るまで待つ）。
- **B-4 を通過済みなら、HTTP 系の失敗（401/403/422）が出た場合でも「サーバは正常／実機の値か config.h の不一致」と確度高く切り分けられる**（サーバ起因は B-4 で除外済みのため）。

- 書込・シリアルモニタ操作・Web UI 確認。失敗時はログ全文を AI に渡す。

- シリアルに `[HTTP] 201 Created OK`、かつ Web UI に送信値が反映される。これで実データ送信パイプライン（デバイス → API → DB → 表示）が一気通貫。
- 既定の送信間隔は 5 分（`SEND_INTERVAL_MS`）。`nextSendAt` の初期値 0 で**起動直後に 1 回**、以降 5 分ごと。確認中に頻繁に見たい場合は一時的に短くしてもよいが、本番では 5 分に戻す。

### B-6. 証明書の注意（本番運用での見直し）— 方針は維持

参考ファームは初回疎通用に **`setInsecure()`（TLS 証明書検証を無効化）**で動く（`config.h` の `TLS_INSECURE` で切替。`1`=検証なし、`0`=検証あり）。検証無効のままだと**中間者攻撃（MITM）の余地**が残るため、本番運用では次のいずれかを検討する（実装計画 §8-4 の要決定事項）。

- **証明書フィンガープリント固定**: 接続先サーバ証明書のフィンガープリントを `config.h`/スケッチに焼き込み、`WiFiClientSecure::setFingerprint()` で一致時のみ接続（ESP8266 Arduino core の BearSSL で `setFingerprint()` が受け付けるのは SHA-1 ハッシュ（要確認・core バージョン依存））。
- **ルート CA 証明書の同梱**: `WiFiClientSecure::setTrustAnchors()` に CA を設定して検証（Let's Encrypt なら ISRG Root X1 を同梱）。NTP 同期で時刻が正しいことが検証の前提（本ファームは送信前に NTP 同期済み）。

```c
// 本番でTLS検証を有効化する場合（config.h）
#define TLS_INSECURE   0
// ↑にした上で esp8266_sht31.ino 側にフィンガープリント固定 or ルートCA設定を追加する。
// 現行スケッチは #if TLS_INSECURE の中で client.setInsecure() を呼ぶだけで、
// 0 にしても検証コードが無く（trust anchor 未設定で）接続できない。
// よって 0 化には上記いずれかの実装追加が必須。
```

- フィンガープリント固定は**証明書更新（Let's Encrypt の更新周期・自動更新の有無は要確認）のたびに値が変わり、ファーム側の再書き込み（＝河野さんの物理作業）が必要**になる点に注意。運用負荷とのトレードオフで、当面はルート CA 同梱方式の方が更新に強い（中間証明書の切り替えがあり得る点だけは要監視）。
- いずれを選ぶにせよ、**最低限 Bearer トークン認証は必須**（A で発行・B-3 で設定済み）であり、TLS 設定に関わらず常に有効にしておく。トークンの失効・再発行運用も §8-4 の検討事項。発行時の有効期限は `gen-token` の `-expire-days=<N>`（既定 0 = 無期限）で設定でき、`expires_at` カラムも存在する。**本番デバイスのトークンは無期限のまま放置せず、有効期限と再発行手順を決めること。**

#### 🤖 AI が支援できること（B-6）
- `TLS_INSECURE=0` 化に伴う `esp8266_sht31.ino` の改修案（`setFingerprint()` or `setTrustAnchors()` の追記コード）を生成・提示する。
- A の Caddy が使う証明書のフィンガープリント取得方法（A 側で取得した値）を `config.h` 用に整形する。**ただしスケッチへの反映後の再コンパイル・再書込は河野さんの物理作業**。
- トークン失効・再発行は A の `gen-token`（本番側で発行）と連動。新トークン発行後の `config.h` 差し替え値生成は AI、再書込は河野さん。

#### 🙋 人が判断/承認する点
- フィンガープリント固定 vs ルート CA 同梱の方針選択（§8-4 の要決定事項）。
- トークン有効期限ポリシーの決定。

#### ✅ 完了判定（本番強化時）
- `TLS_INSECURE=0` で検証コードを実装したファームが書込済みで、シリアルに `[HTTP] 201 Created OK` が継続して出る。トークンに有効期限が設定され、再発行手順が文書化されている。

## C. 現地実証と「AI主導の継続運用」

> **このセクションの位置づけ**
> 現地実証と運用を **「人がコマンドを直接打たず、AI と対話して AI が MCP / SSH で実行する」**形で扱う。狙いは持続可能な運用＝サーバの面倒を AI に任せ、人は「依頼」と「承認」だけに集中すること。
>
> **前提（A/B が完了していること）**: ローカル(Mac)でクロスコンパイル → バイナリのみ scp、PostgreSQL は native apt（Docker 不使用）+ swap、アプリは平文 8080 を `0.0.0.0` bind、前段 Caddy が 80/443 で自動TLS（Let's Encrypt）終端、環境変数は systemd の `EnvironmentFile=/opt/go_iot/go_iot.env`（権限 600）で供給、`go_iot.service`（`User=go_iot` / `Restart=always`）で常駐。`https://<本番FQDN>/health` が 200 を返す状態から始める。
>
> **セキュリティ境界（再掲・MCP化しても厳守）**: 公開してよいのは **80/443（および接続元IP限定の 22）のみ。アプリ 8080 と PostgreSQL 5432 はパブリック非公開**（Lightsail ファイアウォール + OS ufw + listen を localhost 限定）。本セクションの psql/pg_dump は同一インスタンス内（localhost）からのみ実行する。

> ⚠️ **前提**: 本 C 章は §S（AWS MCP 接続・最小権限 IAM 発行）と §A（デプロイ完了・`https://<本番FQDN>/health` 200）が済んでいる前提。人手が残る作業の全体像は付録D-2。MCP の env 名・ツール名は `@latest` で変わりうるため接続前に STABLE 版 README で再確認（要確認）。

### C-0. AI運用のための MCP セットアップ（一度だけ・人手）

AI に AWS と本番サーバを触らせるための土台を作る。ここは **AI に任せられない初回ブートストラップ**（末尾チェックリスト「フェーズ1」参照）。

> **MCP 接続・最小権限 IAM・安全策の正本は §S**: 使う MCP サーバ（**既定=self-host・実証済み**／マネージド版は未検証）、`claude mcp add` / `.mcp.json` の登録、`mcp-security-policy.json`（denyList/elicitList）、承認ゲート（`READ_OPERATIONS_ONLY` / `REQUIRE_MUTATION_CONSENT`）、CloudTrail 監査の詳細は **§S（S-1 接続 / S-3 最小権限 IAM / S-5 安全策 / S-7 デプロイ反復経路）に集約**してある。**A 章を実施した時点で S の接続・認証は完了済み**なので、本 C-0 では運用フェーズ特有の差分だけを示す。

#### C-0-1. 運用フェーズの最小権限 IAM（S-3-2 から作成系を外したもの）

S-3-2 の**構築フェーズ用**ポリシーは `CreateInstances`/`AllocateStaticIp`/`CreateDomainEntry` 等の作成系を含む。運用フェーズはインスタンスが既にあるため、**作成系を外し、参照系＋当該インスタンスの変更系（再起動/スナップショット/ポート変更）だけ**に絞る（破壊系は引き続き非付与＝一次防御）。`<ACCOUNT_ID>`/`<INSTANCE_GUID>` は `GetInstance` の結果で確定。

```jsonc
// 運用フェーズ用 最小権限ポリシー（作成系を含まない。S-3-2 の構築フェーズ版から create-* を外したもの）
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "LightsailReadOnly",
      "Effect": "Allow",
      "Action": [
        "lightsail:GetInstances", "lightsail:GetInstance",
        "lightsail:GetInstanceState", "lightsail:GetInstancePortStates",
        "lightsail:GetInstanceMetricData", "lightsail:GetInstanceAccessDetails",
        "lightsail:GetStaticIps", "lightsail:GetStaticIp",
        "lightsail:GetInstanceSnapshots", "lightsail:GetInstanceSnapshot",
        "lightsail:GetBundles", "lightsail:GetBlueprints", "lightsail:GetRegions",
        "lightsail:GetOperations", "lightsail:GetOperation"
      ],
      "Resource": "*"
    },
    {
      "Sid": "LightsailManageThisInstance",
      "Effect": "Allow",
      "Action": [
        "lightsail:RebootInstance", "lightsail:StartInstance", "lightsail:StopInstance",
        "lightsail:CreateInstanceSnapshot",
        "lightsail:PutInstancePublicPorts", "lightsail:CloseInstancePublicPorts"
      ],
      "Resource": "arn:aws:lightsail:ap-northeast-1:<ACCOUNT_ID>:Instance/<INSTANCE_GUID>"
    }
  ]
}
```

> DNS（`CreateDomainEntry`）系は A-1/A-4 で設定済み前提のため運用フェーズ IAM には含めない（変更時のみ別途付与・**DNS API は us-east-1 限定**で `--region us-east-1` 明示が要る）。`GetInstanceAccessDetails`（一時SSH鍵）が MCP 既定 denylist か未確認＝接続後に疎通確認する（詳細は §S-5-1）。

#### C-0-2. 運用フェーズの denyList / elicitList（S-5-1 の運用サブセット）

運用で AI に絶対やらせない/毎回承認させる操作を `~/.aws/aws-api-mcp/mcp-security-policy.json` に列挙する（表記・制約は §S-5-1 と同じ＝`"aws <service> <operation>"` の完全一致・ワイルドカード不可）。

```json
{
  "version": "1.0",
  "policy": {
    "denyList": [
      "aws lightsail delete-instance",
      "aws lightsail release-static-ip",
      "aws lightsail delete-instance-snapshot"
    ],
    "elicitList": [
      "aws lightsail create-instance-snapshot",
      "aws lightsail reboot-instance",
      "aws lightsail stop-instance",
      "aws lightsail put-instance-public-ports"
    ]
  }
}
```

> **承認ゲートの使い分けは §S-5-2 と同じ**: ①「調子どう?」等の調査・状態確認は `READ_OPERATIONS_ONLY=true`、②再起動/スナップショット/ポート変更等の書込フェーズは false + `REQUIRE_MUTATION_CONSENT=true` で都度承認（elicitation 未対応なら付け外し＋IAM＋人承認で代替）。本サーバは**マルチテナント非対応**＝1ユーザー専用・専用 AWS 認証情報で動かす。

#### C-0-3. SSH 経路（インスタンス内コマンドは AI が SSH で実行・§S-7 と同じ）

`call_aws` は **AWS API のみ**で SSH/scp を扱わないため、インスタンス内の反復操作（systemd 再起動・ログ確認・pg_dump・gen-token）は **SSH 経由を既定**にする（経路の考え方は §S-7）。AI が SSH を実行する経路は次のどちらか:

- **(a) 既定 = ローカル端末の `ssh`/`scp`**（A で `~/.ssh/config` に `Host go-iot-prod` を定義済み）を、AI がシェル実行できる汎用ツール経由で呼ぶ。鍵は人の端末にのみ置く。
- **(b) 任意 = `call_aws` で `aws lightsail get-instance-access-details` を毎回呼び一時SSH鍵を取得**して接続（鍵を端末に常置しない・有効期限あり）。ただし当該 API が MCP 既定 denylist か未確認（要確認）。

> 透明性のため、AI が裏で叩く SSH コマンドの実体は本セクション各所のコードブロックに明示する。人はそれを読んで監査できる。AWS CLI の `date` 例は **macOS（BSD date）構文**（`date -u -v-6H`）で、`call_aws` を Mac 上の MCP から実行する前提。Linux 上で実行する場合は `date -u -d '6 hours ago'` に置換する。

#### C-1-2. 実機連続送信 → グラフ/アラート/最終受信/通信遅延を AI に確認・要約させる

実機書き込み（B・人手）後、5分間隔で自走する。**「届いているか」の判定を AI に任せる**。

```text
▶ AIへの依頼（プロンプト例）
  「眞境名圃場ハウスA（device_id=<id>）の実証状況を要約して。
   ①直近24hで何点受信しているか ②最終受信時刻と現在との開き
   ③recorded_at と created_at の通信遅延 ④アラートは出ているか
   を本番 DB から読んでまとめて。」
```

```text
🤖 AIが行う操作（裏で実行）
  DB は localhost:5432 で外部非公開のため、AI は SSH で本番に入り psql を read-only で叩く。
  パスワードは ~/.pgpass（chmod 600）に解決させ、接続文字列・ps・履歴に平文を出さない。
  （sensor_readings は recorded_at=デバイス計測日時 / created_at=サーバ受信日時 を保持・スナップショット準拠）
```

```bash
# AI が SSH 経由で実行（最終受信・通信遅延・受信点数）。SELECT のみ＝破壊しない
ssh go-iot-prod "psql 'postgres://<DBユーザー>@localhost:5432/go_iot' -c \
\"SELECT device_id, count(*) AS pts_24h
    FROM sensor_readings
   WHERE device_id=<id> AND created_at > now() - interval '24 hours'
   GROUP BY device_id;\" -c \
\"SELECT recorded_at, created_at,
         EXTRACT(EPOCH FROM (created_at - recorded_at)) AS delay_sec
    FROM sensor_readings WHERE device_id=<id>
   ORDER BY created_at DESC LIMIT 10;\""
```

```text
🙋 人が確認/承認する点
  - read-only 確認なので承認は軽い（READ_OPERATIONS_ONLY=true 起動なら AWS 側書込は物理的に不可。
    ただし SSH 越しの SQL は MCP の制御外なので、参照クエリに限定する運用ルールを守る）。
  - delay_sec が極端に大/負 → NTP 時刻ずれを疑う（C-4・参考ファーム仕様）。
  - 最終 created_at が現在から大きく開く → 送信停止（Wi-Fi断/電源/トークン失効）を現地確認。
✅ 完了判定: AI の要約で「直近24hに5分間隔の点が並び、delay_sec が常識的範囲、最終受信が直近」を確認。
   Web UI のデバイス詳細グラフ（期間=直近24h）でも点が時系列に並ぶこと（最小ゴール達成）。
```

### C-2. AI主導の継続運用（本書の主眼）

ここからが本セクションの中心。**「サーバ調子どう?」のような曖昧な一言から、AI が状態確認・原因切り分け・必要な操作（要承認）まで対話で完結**させる。各操作は「▶ 依頼 → 🤖 MCP/SSH操作 → 🙋 承認点 → ✅ 完了判定」で示す。AI が裏で叩く実コマンドはコードブロックで明示（＝監査可能）。

#### C-2-1. 「サーバ調子どう?」— ヘルス総点検（read-only）

```text
▶ AIへの依頼（プロンプト例）
  「go-iot 本番サーバの調子どう? /health・サービス稼働・ディスク/メモリ/swap・直近のエラーログを見て要約して。」
```

```text
🤖 AIが行う操作（裏で実行）
  この依頼は read-only。安全のため AWS API MCP は READ_OPERATIONS_ONLY=true で起動した状態が望ましい。
  (a) Lightsail のインスタンス状態とメトリクスを call_aws で取得（AWS API）
  (b) アプリ/OS の状態は SSH 経由で確認（インスタンス内・参照のみ）
```

```bash
# (a) AI が call_aws で実行（AWS API・read-only）。インスタンスが running か / CPU メトリクス
#     ※ date は macOS(BSD) 構文。Linux 実行なら -d '6 hours ago' に置換。period は60の倍数。
```

#### C-2-4. 「このトークン、失効して再発行して」— トークン失効/再発行（要承認）

トークンには有効期限（`expires_at`）を付けられる（`gen-token -expire-days=N`、0=無期限）。認証は `token_hash` 一致かつ `(expires_at IS NULL OR expires_at > NOW())` で行うため、`expires_at` を現在時刻以下にすると即失効する。紛失・退役・期限切れ時は新トークンを発行して実機の `API_BEARER` を更新し、旧トークンを失効させる。**トークンは `user_id` に紐づく**（`device_tokens` に `device_id` 列は無い）ため、特定する際は name/user_id で突き合わせる。

```text
▶ AIへの依頼: 「眞境名圃場ハウスA のトークンを再発行（期限365日）して、旧トークンを失効させて。
   どのトークンが現行か先に見せて。」
```

```bash
# (1) AI が SSH で現行トークンを確認（参照のみ・平文は DB に無くハッシュのみ）
ssh go-iot-prod "psql 'postgres://<DBユーザー>@localhost:5432/go_iot' -c \
\"SELECT id, name, user_id, expires_at, created_at, last_used_at FROM device_tokens
   WHERE user_id=<user_id> ORDER BY created_at DESC LIMIT 10;\""
```

```bash
# (2) AI が SSH で再発行（クロスコンパイル済み gen-token バイナリを使用・サーバに Go 不要）
ssh go-iot-prod 'sudo -u go_iot bash -c "set -a; . /opt/go_iot/go_iot.env; set +a; \
  /opt/go_iot/go_iot_gen-token -user=<user_id> -name=\"眞境名圃場ハウスA(再発行)\" -expire-days=365"'
# → 新平文トークンを控える（再表示不可）。実機 config.h の API_BEARER に焼き直す（B・人手）。
```

```bash
# (3) AI が SSH で旧トークンを失効（expires_at を現在時刻に。id は (1) で特定）
#     ※ device_tokens に削除用クエリ/論理削除列が無いため失効は expires_at の更新で行う。

ssh go-iot-prod "psql 'postgres://<DBユーザー>@localhost:5432/go_iot' -c \
```

### F-7. 書き込み（最重要のつまづき）

- ❌ **arduino-cli upload（＝同梱 esptool.py v3.0）が `Invalid head of packet` で失敗**（接続・MAC 読取までは成功するが stub 転送で死ぬ）。
- ✅ **`esptool` v5.3.0（pip venv）で `write-flash`**:
  ```bash
  esptool-venv/bin/esptool --port /dev/cu.usbserial-XXXX --chip esp8266 \
    --before no-reset --after no-reset \
    write-flash --flash-mode dout --flash-freq 40m --flash-size detect --erase-all 0x0 \
    <build>/esp8266_sht31.ino.bin
  ```
- **リセット配線（この基板・実測からの確定範囲）**: 自動リセット（`default_reset`＝DTR/RTS トグル）を投げても**チップは通常 RUN で起動**した（＝RST 線は配線されているが、**GPIO0 を Low に落とす制御線が未配線**で書込モードに入れない）。どちらの線（DTR/RTS）が RST かは**未確定**。加えて**シリアル open 時の DTR/RTS 極性が不定で自動リセット起動も不安定**だったため、結論として**物理 RST/GPIO0 ボタンが最も確実**。
  - **書込モード（手動）**: GPIO0 ボタンを押したまま → RST ボタンを一度押して離す → GPIO0 を離す。
  - **書込後の起動**: `--after no-reset` で bootloader 待機 → **RST ボタンを物理押下**して新ファーム起動が確実（自動パルスは空振りしうる）。
- **シリアルキャプチャ（pyserial）の罠**: ポート open 時に DTR が RST を握ると無出力になる → RST ボタン押下 or RTS/DTR パルスで起動させる。**日本語マーカーは `"文字列".encode()`**（`b"日本語"` は SyntaxError＝bytes リテラルに非 ASCII 不可）。

### F-8. センサー検出のつまづき → I2C `0x45` 発見（核心・調査方法を再現可能に）

- **症状**: センサー配線済みなのに初回起動ログが **`[SHT31] not found at 0x44`**（WiFi/NTP/MAC は OK＝ESP は正常）。→ アドレス/ピン/種類のいずれかが想定と不一致。引継ぎメモでセンサー型番は未確定だった（DHT11/DHT22/SHT31/BME280…）。
- **調査方法（推測せず総当たりで実測）**: **I2C スキャナ `firmware/i2c_scanner/`** を焼いて探索。スキャナは ESP8266 で I2C 可能な **10 通りのピン組合せ**（`{4,5}`既定・`{5,4}`逆・`{0,2}`・`{2,0}`・`{12,14}`・`{14,12}`・`{13,12}`・`{12,13}`・`{2,14}`・`{4,14}`）× 全アドレス `0x01..0x7F` を `Wire.beginTransmission`→`endTransmission()==0` で走査し、SHT3x/BME280/AHT/HTU の候補名も付す。
  - **手順**: ① スキャナを esptool で書込（F-7 と同じ `write-flash`）→ ② 物理 RST で起動 → ③ シリアル採取（F-9 の `serial_capture.py --until "スキャン完了"`）。
- **実スキャン結果（本案件で取得した実データ）**:
  ```
  ================ I2C スキャナ開始 ================
  (全ピン組合せ x 全アドレスを総当たり)
  >>> SDA=GPIO4(D2)  SCL=GPIO5(D1)  ★既定
      ✓ 応答 0x45  (SHT3x の可能性)
  ================ スキャン完了 ================
  ```
  → **配線・ピンは想定どおり GPIO4(SDA)/GPIO5(SCL)。アドレスだけ 0x44 ではなく `0x45`**（SHT31 の ADDR ピンが High に結線）。
- **修正**: `config.h` の `SHT31_ADDR=0x45` → 再ビルド → 再書込（1行で解決）。
- **横展開**: 別センサーでも同手順でアドレス特定可（SHT3x=`0x44/0x45`・BME280/BMP280=`0x76/0x77`・AHT10/20=`0x38`・HTU21/Si70xx=`0x40`）。**どのピン組合せにも応答が無ければ** I2C 以外（DHT22 等の 1-Wire）か、配線/電源/プルアップ抵抗を疑う。

### F-9. 実機シリアルの採取（USB チップ非搭載の基板）＋ 汎用ツール `serial_capture.py`

- **配線**: 実機基板は USB-シリアル非搭載 → **外付け FTDI を UART に接続**: `FTDI GND↔基板 GND` / `FTDI RXD↔ESP TXD`（**受信だけなら 2 本**）。**FTDI VCC は繋がない**（基板は自前 AC アダプタ給電・2 電源衝突回避）。接続で `/dev/cu.usbserial-XXXX` が復活。**FTDI は監視用なので、確認後は抜いてよい**（基板は自前電源で 5 分間隔送信を継続）。
- **採取は再利用ツール [`firmware/tools/serial_capture.py`](../firmware/tools/serial_capture.py) を使う**（Arduino Serial Monitor の代替・AI が直接実行可。今セッションで MAC/I2Cスキャン/201 採取に繰返し使用したものを汎用化して同梱）:
  ```bash
  esptool-venv/bin/python firmware/tools/serial_capture.py \
    --port /dev/cu.usbserial-XXXX --seconds 80 --reset \
    --until "201 Created OK" --until "not found" --until "スキャン完了"
  ```
- **罠（このツールが吸収している点）**:
  - ポート open 時に DTR/RTS が RST 線を握ると**無出力 or リセット保持**になる → ツールは両線を解放しリセットパルスを打つ（`--reset`）。ただし極性が不定なので、**物理 RST ボタン押下が最確実**（その場合 `--reset` 無しで起動 → ボタンを押す）。
  - 終了マーカーに日本語を使うときは内部で `.encode()`（`b"日本語"` は **SyntaxError**＝bytes リテラルに非 ASCII 不可）。
  - `--until` は複数指定可。いずれか出現で早期終了（最大は `--seconds`）。

## 付録G. 査読で残ったリスク（センサー・基盤）

ESP8266 実機・基板・物理書き込み、および運用の後始末に関する残リスク（旧・付録B から分離）。「接続前の調査（2026-06 時点）ベースで接続後に実機検証する」前提は付録B と同じ。

**B（ESP8266）・運用の後始末**
- **A→B 直列依存**: A（FQDN/トークン/device_id 確定）が未完成のまま B に着手すると config.h が埋まらない。B-0 でゲートするが、A 側の完了判定(/health 200)と本書前提がずれた場合に検知できない。
- **B-3.5 机上点検は静的レビュー**: 実機固有のヒープ不足・BearSSL バッファ(`setBufferSizes(512,512)`)起因の TLS ハンドシェイク失敗は事前検出不可。`TLS_INSECURE=0`（ルートCA検証）に切り替えると BearSSL のメモリ消費が増え 512/512 で落ちる可能性あり、B-6 実装時に実機再検証。
- **B-6 ルートCA同梱も完全な再書込フリーではない**: Let's Encrypt の中間/ルート切替（DST Root → ISRG Root X1 の実績）が起きると同梱 CA 更新＝再書込が要る。`setFingerprint()` が SHA-1 のみか等は core 版で変わるため、フィンガープリント方式採用なら実機 core 版で要再確認。
- **テスト/受け入れ行の後始末が未定**: B-4 本番テスト送信は本番 DB に1行残す。受け入れ用 readings/alert_histories/ユーザー/トークンの DELETE 手順・対象 device_id 確定も未定（止め忘れ・削除漏れリスク）。「seed は本番禁止」とするのみで掃除ポリシーは別途決める。
