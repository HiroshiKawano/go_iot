## 1. エグゼクティブサマリ

**ESP8266（ESP-WROOM-02）+ 温湿度センサー（SHT31・I2C `0x45`）のデータを Go バックエンドで受信・蓄積し、ブラウザで可視化・アラート管理する農業IoTシステム。** cc-sdd（仕様駆動開発）で進行中。

**全体到達度:** **「サーバ側（バックエンド + Web UI 層 S1〜S8）は実装完了」。** 残るは実デバイスからの実データ送信（ファームウェア）と現地実証。

- バックエンド基盤（DB 6テーブル・sqlc 全37クエリ・設定/起動/Graceful shutdown・デバイス Bearer 認証・センサーデータ受信 API・CLI ツール・OpenAPI ドキュメント）は **実装完了**。
- Web UI 層（scs Session 認証・CSRF・MethodOverride・templ 画面/レイアウト/コンポーネント・HTMX 動的化・サーバサイド SVG グラフ・アラート判定の本番接続）も **S1〜S8 として実装完了**（2026-06-08 までに全 spec 完了）。
- つまり **「デバイス → API → DB → Web 表示・アラート」のサーバ側パイプラインは一通り通っている**。
- **残課題はデバイス側**: ESP8266 から実センサー値を読み取り、自前 Go API（`POST /api/sensor-data`、JSON + Bearer）へ送信するファームウェアと、眞境名さん圃場での現地実証（→ [実装計画.md](実装計画.md) §8）。

## 2. アーキテクチャ概要

3層モノリス構成。フロント用 JSON API を持たず、templ が HTML を直接返す（グラフも含め全表示ロジックをサーバサイドに集約）。

```
[デバイス層]   温湿度センサー →(I2C)→ ESP8266 →(HTTPS POST + Bearer SHA-256)→   ★ファーム未実装
[バックエンド]  Go 1.26 + Gin + templ + PostgreSQL 16 (pgx/v5 + sqlc)
                ├─ デバイスAPI (/api/*)        : 自作 Bearer ミドルウェア     ✅
                ├─ Web UI (/ non-api)         : scs Session + gorilla/csrf    ✅
                ├─ サーバサイド SVG グラフ生成   : self-built（internal/chart）  ✅
                └─ Scalar + OpenAPI (/docs)    : go:embed                     ✅
[フロント層]    templ（28ファイル）✅ + HTMX ✅ + Alpine.js（一部）✅
```

### アプリ内部アーキテクチャ（パッケージ層）

実務的 **Layered-lite**（layer-first + 画面=feature 分割）を採用、**厳格 Clean は不採用**（2026-06-03 決定）。依存は下向き一方向（`cmd → handler/middleware/auth → service → repository → infra`、`domain`・`chart`・`timefmt` は無依存の純粋層。下向きなら隣接層スキップ可）。DIP は2点で、① DB ポート=sqlc 生成の `repository.Querier` interface（handler/auth/service は具象 `*Queries` ではなく Querier 依存＝テストでモック可）、② consumer interface は必要時のみ（`SensorAPI.Evaluator`・各 handler の最小 Repo interface 等）。所有者認可（BOLA 防止）は `internal/authz` に集約。詳細ルールの正は [steering/structure.md](../.kiro/steering/structure.md) / [steering/tech.md](../.kiro/steering/tech.md)。

### 認証方式（接続元別）

| 接続元 | 認証方式 | 状態 |
|--------|----------|------|
| ESP8266（デバイス） | 自作 Bearer トークン（SHA-256 ハッシュ照合。`internal/auth/device_auth.go`）| ✅ 実装済み |
| ブラウザ | Session（alexedwards/scs/v2 + scs/pgxstore。`internal/auth/session_auth.go`）+ CSRF（gorilla/csrf）| ✅ 実装済み |

### ルーティング（実装の現状）

`cmd/server/main.go` の `newHTTPHandler` が全ルートを配線。http.Handler 層で `MethodOverride(sm.LoadAndSave(engine))` の順に外側から包む（メソッド上書きとセッション load/save は Gin のメソッドルーティングより外側で適用）。

| グループ | メソッド/パス | ハンドラ | 主なミドルウェア |
|---|---|---|---|
| ヘルス/ドキュメント | `GET /health` | inline（DB Ping 込み 200/503）| Logger, Recovery |
| | `GET /docs`・`GET /docs/openapi.yaml` | docs（Scalar / OpenAPI）| Logger, Recovery |
| 静的 | `GET /static/*` | `view.MountStatic`（go:embed）| Logger, Recovery |
| デバイス API | `POST /api/sensor-data` | `SensorAPI.Create`（+ AlertEvaluator 注入）| **DeviceAuth**（Bearer）|
| Web 認証 | `GET /`・`GET/POST /login`・`GET/POST /register`・`POST /logout` | `AuthHandler` | **SessionLoad**, **CSRF** |
| Web ダッシュボード | `GET /dashboard` | `AuthHandler.Dashboard` | + **RequireAuth** |
| デバイス登録・編集 | `GET /devices/create`・`POST /devices`・`GET /devices/:device/edit`・`PUT /devices/:device` | `DeviceHandler` | + RequireAuth |
| デバイス詳細 | `GET /devices/:device`・`GET /devices/:device/chart`・`DELETE /devices/:device` | `DeviceHandler` | + RequireAuth |
| センサー履歴 | `GET /devices/:device/readings` | `ReadingsHandler.Index` | + RequireAuth |
| アラートルール | `GET/POST /alerts/rules`・`GET /alerts/rules/:rule/edit`・`PUT /alerts/rules/:rule`・`PATCH /alerts/rules/:rule/toggle`・`DELETE /alerts/rules/:rule` | `AlertRuleHandler` | + RequireAuth |
| アラート履歴 | `GET /alerts/history` | `AlertHistoryHandler.Index` | + RequireAuth |

> グローバルに `engine.Use(gin.Logger(), gin.Recovery())`。Web グループは `SessionLoad` + `CSRF`、各保護ルートに `RequireAuth`。デバイス API は CSRF 対象外（Bearer のみ）。GET のみの履歴系（readings / alerts/history / chart）は CSRF 検証対象外。

## 5. 実装済みサブシステム詳細

### 5.1 ドメイン層 Enum ✅

文字列ベースの型付き Enum。DB には VARCHAR 格納、CHECK 制約で値域保証。マスターデータは DB テーブルではなく Go 定数で管理。

**`internal/domain/metric.go`** — `type Metric string`
- 定数: `MetricTemperature="temperature"` / `MetricHumidity="humidity"`
- メソッド: `Label()`（温度/湿度）、`Unit()`（℃/%）、`Valid()`、`ParseMetric()`、`AllMetrics()`

**`internal/domain/comparison_operator.go`** — `type ComparisonOperator string`
- 定数: `OpGreaterThan=">"` / `OpLessThan="<"` / `OpGreaterThanOrEqual=">="` / `OpLessThanOrEqual="<="`
- メソッド: `Label()`、**`Evaluate(actual, threshold float64) bool`（アラート判定の中核）**、`Valid()` 他
- テスト: 境界値（threshold 35.00/30.00 で ±0.01 と等値）を重点検証

### 5.2 インフラユーティリティ ✅

**`internal/infra/pgconv/pgconv.go`** — pgtype ↔ Go 型変換
- `Numeric2(f float64) pgtype.Numeric`（NUMERIC(5,2) 相当）/ `NumericToFloat(n) float64`（無効値は 0 を返す防御的実装）/ `Timestamptz` 系

**`internal/infra/token/token.go`** — Bearer トークン
- `Generate() (plaintext, hash, err)`: crypto/rand 32バイト → hex 64文字平文 + SHA-256 ハッシュ
- 方針: 平文は発行時のみ表示、DB には SHA-256 ハッシュのみ保存

### 5.3 デバイスAPI認証ミドルウェア ✅

**`internal/auth/device_auth.go`** — `DeviceAuth(DeviceAuthConfig{Repo})`
- `Authorization: Bearer <token>` を SHA-256 化 → `GetDeviceTokenByHash` で照合。期限切れは SQL 側で除外
- 成功時 `c.Set` で `user_id` / `token_id` 格納。`last_used_at` は best-effort 更新
- ゲッター: `auth.UserID(c)` / `auth.DeviceTokenID(c)`

### 5.4 センサーデータ受信ハンドラ ✅（アラート判定 接続済み）

**`internal/handler/sensor_api.go`** — `SensorAPI.Create`（`POST /api/sensor-data`）
- フロー: `ShouldBindJSON` → `authz.RequireDeviceOwner`（存在/所有者確認）→ `CreateSensorReading` → `UpdateDeviceLastCommunicated`（失敗無視）→ **`Evaluator.EvaluateAndNotify`（アラート同期判定・ベストエフォート）** → **201**
- エラー振り分け: JSON構文 → 400 / 未認証 → 401 / 他ユーザー所有 → 403 / binding違反・存在しない device_id → 422 / DB失敗 → 500
- レスポンスに `alerts_fired`（当該受信で発火したアラート件数）を含む

リクエストスキーマ:
```go
type CreateSensorReadingRequest struct {
    DeviceID    int64     `json:"device_id"    binding:"required,min=1"`
    Temperature float64   `json:"temperature"  binding:"gte=-40,lte=125"`
    Humidity    float64   `json:"humidity"     binding:"gte=0,lte=100"`
    RecordedAt  time.Time `json:"recorded_at"  binding:"required"`
}
```

### 5.5 エントリポイント・設定・DB接続 ✅

**`cmd/server/main.go`** — `run()` で `config.Load` → `infradb.NewPool` → `repository.New` → `auth.NewSessionManager` → `newHTTPHandler`（全ルート配線）→ goroutine で `ListenAndServe` → SIGINT/SIGTERM → `srv.Shutdown(10秒)`。`newHTTPHandler` は §2 ルーティング表のとおり API/Web/静的/docs/health を配線し、`MethodOverride(sm.LoadAndSave(engine))` で包む。

**`internal/config/config.go`** — `os.Getenv` 直読み。必須 `DATABASE_URL` / `SESSION_SECRET`、デフォルト `APP_ENV=development` / `APP_PORT=8080`。production かつ `SESSION_SECRET` < 32文字でエラー。**`SESSION_SECRET` は scs SessionManager と gorilla/csrf 鍵導出で実利用**。

**`internal/infra/db/pool.go`** — `MaxConns=10` / `MinConns=2` 他。起動時 5秒 Ping で疎通確認。

### 5.6 APIドキュメント ✅

`internal/docs/`（go:embed）で `openapi.yaml`（OpenAPI 3.0.3、`POST /api/sensor-data` 1本、Bearer、201/400/401/403/422/500）と Scalar `index.html`（CDN ロード）を同梱。

### 5.7 CLI ツール ✅

**`cmd/seed/main.go`** — 冪等な開発用シード（`make seed`）。6テーブルを `TRUNCATE RESTART IDENTITY CASCADE`、ユーザー1名・デバイス2台・各288件/計576件のセンサーデータ（5分間隔・決定論的）・アラートルール4件・履歴3件を投入（device_tokens は gen-token で別途発行）。
**`cmd/gen-token/main.go`** — Bearer トークン発行（`make gen-token user=<id> name="..."`）。平文は標準出力に1回だけ表示。
**`cmd/db-snapshot/main.go`** — 実DBを内省し `docs/database_snapshot/` を再生成（`make db-snapshot`）。
**`cmd/sensor-sim/main.go`** — ESP8266 ファームと同形の JSON を `POST /api/sensor-data` へ送る実機不要のテスト送信 CLI（`make sensor-sim token=<平文> device=1`、連続送信は `-count 0 -interval 5m -random`）。受信〜画面反映〜アラート判定を実機を待たず検証。payload 生成は純粋関数で単体テスト済み。

### 5.8 Web UI 認証・ミドルウェア（S1）✅

- **`internal/auth/session_auth.go`** — `NewSessionManager(pool, cfg)`：scs/v2 + pgxstore で PostgreSQL セッションストアを構築。ログインユーザー ID の格納/取得ヘルパ。
- **`internal/middleware/session_load.go`** — `SessionLoad(sm)`：Web グループに scs を適用。
- **`internal/middleware/method_override.go`** — `MethodOverride`：hidden `_method`（put/patch/delete）を実 HTTP メソッドへ昇格。http.Handler 層（Gin ルーティングより外側）で適用。
- **`internal/middleware/csrf.go`** — `CSRF(cfg)`：gorilla/csrf を Gin に適応。`SESSION_SECRET` を SHA-256 で 32バイト鍵化。dev（HTTP）は `csrf.Secure(false)`。Web グループ限定（デバイス API は対象外）。
- **`internal/middleware/require_auth.go`** — `RequireAuth`：未ログインは `/login` へリダイレクト。

### 5.9 Web UI ハンドラ群（S3〜S8）✅

| ハンドラ | 主な責務 | 対象画面 |
|---|---|---|
| `handler/auth.go`・`auth_validation.go` | login/register/logout・Root・Dashboard・入力検証 | login / register / dashboard |
| `handler/device.go`・`device_form.go` | デバイス登録・編集・削除・共有フォーム | device-create / device-edit |
| `handler/device_show.go` | デバイス詳細・期間別 SVG グラフ・最新計測 | device-show |
| `handler/readings.go` | センサー履歴（期間フィルタ + 集計 + 20件/ページ）| readings |
| `handler/alert_rule.go`・`alert_rule_form.go` | アラートルール インライン CRUD・有効切替 | alert-rules |
| `handler/alert_history.go` | アラート履歴（フィルタ + ページング）| alert-history |

- 所有者認可は `internal/authz`（`RequireDeviceOwner` / `RequireAlertRuleOwner`）に集約。HTMX 部分更新と no-JS フォールバック（`_method`）の両経路に対応。

### 5.10 アラート判定サービス（S2）✅

## 7. DB層リファレンス（6テーブル + sqlc 全37クエリ）

### 設計方針
- **外部キー制約は張らない**（参照整合性はアプリ層 JOIN で担保）
- **論理削除**（`deleted_at`）を主要テーブルに採用、sqlc クエリは常に `WHERE deleted_at IS NULL`
- **UNIQUE は部分インデックス**（`WHERE deleted_at IS NULL`）で soft-delete 衝突を回避
- マスターデータは Go 定数 + VARCHAR + CHECK 制約
- **集計テーブルは持たない**（24h は生データ、7日/30日は `GROUP BY DATE(recorded_at)`）

### テーブル一覧

| テーブル | 論理削除 | 主な制約・インデックス |
|---------|:-------:|----------------------|
| **users** | - | `email` UNIQUE。物理削除運用 |
| **devices** | ✅ | `mac_address` CHECK（`^([0-9A-Fa-f]{2}:){5}[0-9A-Fa-f]{2}$`）+ 部分UNIQUE。`user_id`/`is_active` 部分INDEX |
| **device_tokens** | - | `token_hash VARCHAR(64)` UNIQUE。`abilities JSONB`。物理削除（失効は `expires_at`）|
| **sensor_readings** | ✅ | `temperature/humidity NUMERIC(5,2)` CHECK（-40..125 / 0..100）。`(device_id, recorded_at DESC)` 複合INDEX |
| **alert_rules** | ✅ | `metric`/`operator` CHECK。`(device_id, is_enabled)` 部分INDEX |
| **alert_histories** | ✅ | metric/operator/threshold/actual_value を**非正規化保持**。未通知バナー用部分INDEX |

> **`recorded_at`（デバイス計測時刻）と `created_at`（サーバ受信時刻）の差分から通信遅延を分析可能。**

### sqlc 全37クエリ関数（`internal/repository/querier.go`）

| テーブル | 関数（数） |
|---------|-----------|
| users (6) | `GetUser` / `GetUserByEmail` / `CreateUser` / `UpdateUser` / `MarkUserEmailVerified` / `DeleteUser` |
| devices (7) | `GetDevice` / `GetDeviceByMacAddress` / `ListDevicesByUser` / `CreateDevice` / `UpdateDevice` / `UpdateDeviceLastCommunicated` / `SoftDeleteDevice` |
| device_tokens (5) | `CreateDeviceToken` / `GetDeviceTokenByHash` / `UpdateDeviceTokenLastUsed` / `ListDeviceTokensByUser` / `DeleteDeviceToken` |
| sensor_readings (7) | `CreateSensorReading` / `GetLatestSensorReading` / `ListRecentSensorReadings` / `ListDailySensorAggregates` / `GetSensorReadingsSummary` / `ListSensorReadingsPaginated` / `CountSensorReadingsInRange` |
| alert_rules (7) | `GetAlertRule` / `ListAlertRulesByDevice` / `ListEnabledAlertRulesByDevice` / `CreateAlertRule` / `UpdateAlertRule` / `ToggleAlertRule` / `SoftDeleteAlertRule` |
| alert_histories (5) | `CreateAlertHistory` / `ListUnnotifiedAlertHistoriesWithDevice` / `ListAlertHistoriesPaginated` / `CountAlertHistoriesInRange` / `MarkAlertHistoryNotified` |

**JOIN クエリの注意点:**
- `ListAlertHistoriesPaginated` / `CountAlertHistoriesInRange` / `ListUnnotifiedAlertHistoriesWithDevice` は alert_histories→alert_rules→devices の3テーブル JOIN で `devices.name` を取得、3テーブルすべて `deleted_at IS NULL` 確認、`d.user_id` でテナント分離
- `ListAlertHistoriesPaginated` は `sqlc.narg('device_id')` で「未指定なら全デバイス対象」を実現（生成コードは `*int64`）
- 集計クエリの `AVG` は `::NUMERIC(5,2)`、`COUNT(*)` は `::BIGINT` でキャスト。**`MAX`/`MIN` は未キャストで `pgtype.Numeric`(nullable) になる点に注意**

### Go 型マッピング（`models.go`）
- `NUMERIC` → `pgtype.Numeric`、`TIMESTAMPTZ` → `pgtype.Timestamptz`、`JSONB`(abilities) → `[]byte`
- NULL許容 `location` → `*string`（`emit_pointers_for_null_types`）、NOT NULL の id 等 → 素の `int64`/`string`

## 10. リスク・不整合

- ✅ **デバイス側 実機送信を確認済**: 実 ESP8266＋SHT31 から本番へ実測送信→受信(201)→画面反映を確認（id=2, 28.96℃/60.38%）。残るは眞境名さん圃場での**現地連続実証**。
- ⚠️ **言語統一動機の縮小**: 新規構築の当初動機「ESP32 + TinyGo でデバイスもGo」は、**ESP8266（Xtensa lx106）が TinyGo 実質非対応**のため成立しない。ファームは Arduino C++（`ESP8266WiFi`）想定（→ [実装計画.md](実装計画.md) §8-4 / [steering/product.md](../.kiro/steering/product.md)）。
- ✅ **センサー型番 確定**: 実機は **SHT31**（I2C・アドレス `0x45`＝ADDR High／既定 `0x44`、Adafruit SHT31 Library）。実測送信〜画面反映まで確認済み。
- ⚠️ **モジュールパスとアカウント不一致**: go.mod・OpenAPI contact が `github.com/HiroshiKawano/go_iot`、コミットユーザーは `stamppK`（移管/フォークの可能性。CI/配布時の齟齬リスク）。
- ⚠️ **pgconv の握り潰し**: `NumericToFloat`/`TimestamptzToTime` が変換失敗・無効値を error でなく 0/ゼロ値で黙って返すため、「正常な0」と「変換失敗」を区別不能。集計クエリの `MAX`/`MIN` が未キャストで nullable な点と複合し、表示層で誤った0を出すリスク。
- ⚠️ **外部キー制約なし**: 参照整合性をアプリ層 JOIN に委ねる設計のため、device 削除時の孤児レコードや不正 device_id 混入を DB レベルで防げない。
- ⚠️ **ability ベース認可が未使用**: `device_tokens.abilities` を取得するが認可判定に未反映。トークン権限の細分化は運用上ザル。
- ⚠️ **Scalar UI の CDN 依存**: go:embed されておらず、オフライン/閉域環境ではドキュメント UI が表示されない（OpenAPI 仕様と index.html のみ同梱）。
