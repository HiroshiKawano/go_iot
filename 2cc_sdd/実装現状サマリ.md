## 1. エグゼクティブサマリ

**ESP32 温湿度センサー（SHT31）のデータを Go バックエンドで受信・蓄積し、ブラウザで可視化・アラート管理する農業IoTシステム。** cc-sdd（仕様駆動開発）で進行中。

**全体到達度:** **「バックエンド完成・フロントエンド未着手」の中間段階。**
- バックエンド基盤（DB 6テーブル・sqlc 全37クエリ・設定/起動/Graceful shutdown・デバイス Bearer 認証・センサーデータ受信 API・CLI ツール・OpenAPI ドキュメント）は**ほぼ実装完了**。
- 設計書が想定するブラウザ向け Web UI 層（scs Session 認証・templ 画面・HTMX 動的化・アラート判定）は**全面未着手**。
- つまり **デバイス → API → DB の片側パイプラインのみが通っている**状態。

## 2. アーキテクチャ概要

3層モノリス構成。フロント用 JSON API を持たず、templ が HTML を直接返す（グラフも含め全表示ロジックをサーバサイドに集約）。

```
[デバイス層]   SHT31 →(I2C)→ ESP32 →(HTTP POST + Bearer SHA-256)→
[バックエンド]  Go 1.26 + Gin + templ + PostgreSQL 16 (pgx/v5 + sqlc)
                ├─ デバイスAPI (/api/*)        : 自作 Bearer ミドルウェア
                ├─ Web UI (/ non-api)         : scs Session（★未実装）
                ├─ サーバサイド SVG グラフ生成   : self-built（★未実装）
                └─ Scalar + OpenAPI (/docs)    : go:embed
[フロント層]    templ（★未実装）+ HTMX（★未実装）+ Alpine.js（一部）
```

### アプリ内部アーキテクチャ（パッケージ層）

実務的 **Layered-lite**（layer-first + 画面=feature 分割）を採用、**厳格 Clean は不採用**（2026-06-03 決定）。依存は下向き一方向（`cmd → handler/middleware/auth → service → repository → infra`、`domain` は無依存の純粋層。下向きなら隣接層スキップ可）。DIP は2点限定で、① DB ポート=sqlc 生成の `repository.Querier` interface（handler/auth は具象 `*Queries` ではなく Querier 依存＝テストでモック可。`SensorAPI.Repo` / `DeviceAuthConfig.Repo`）、② service の consumer interface は必要時のみ（現状は①のみ実装）。詳細ルールの正は [steering/structure.md「依存方向ルール」](../.kiro/steering/structure.md) / [steering/tech.md「データアクセス方針」](../.kiro/steering/tech.md)。

### 認証方式（接続元別）

| 接続元 | 認証方式 | 状態 |
|--------|----------|------|
| ESP32 | 自作 Bearer トークン（SHA-256 ハッシュ照合）| ✅ 実装済み |
| ブラウザ | Session（alexedwards/scs + PostgreSQL ストア）| ⛔ 未実装 |

### ルーティング（実装の現状 — 全5ルート）

| メソッド | パス | ハンドラ | ミドルウェア | 備考 |
|---------|------|---------|-------------|------|
| GET | `/` | プレースホルダ | Logger, Recovery | 固定文字列を返すだけ（Web UI 未配線）|
| GET | `/health` | inline | Logger, Recovery | DB Ping 込み（200/503）|

## 5. 実装済みサブシステム詳細

### 5.1 ドメイン層 Enum ✅

文字列ベースの型付き Enum。DB には VARCHAR 格納、CHECK 制約で値域保証。マスターデータは DB テーブルではなく Go 定数で管理。

**`internal/domain/metric.go`（59行）** — `type Metric string`
- 定数: `MetricTemperature="temperature"` / `MetricHumidity="humidity"`
- メソッド: `Label()`（温度/湿度）、`Unit()`（℃/%）、`Valid()`、`ParseMetric()`、`AllMetrics()`

**`internal/domain/comparison_operator.go`（75行）** — `type ComparisonOperator string`
- 定数: `OpGreaterThan=">"` / `OpLessThan="<"` / `OpGreaterThanOrEqual=">="` / `OpLessThanOrEqual="<="`
- メソッド: `Label()`（より大きい/より小さい/以上/以下）、**`Evaluate(actual, threshold float64) bool`（アラート判定の中核）**、`Valid()`、`ParseComparisonOperator()`、`AllComparisonOperators()`
- テスト: 境界値（threshold 35.00/30.00 で ±0.01 と等値）を重点検証

### 5.2 インフラユーティリティ ✅

**`internal/infra/pgconv/pgconv.go`（49行）** — pgtype ↔ Go 型変換
- `Numeric2(f float64) pgtype.Numeric`: NUMERIC(5,2) 相当。`math.Round(f*100)` + Exp=-2 で小数第3位四捨五入
- `NumericToFloat(n) float64`: 無効値は **0 を黙って返す（防御的）**
- `Timestamptz(t) pgtype.Timestamptz` / `TimestamptzToTime(t) time.Time`: 無効時はゼロ値

**`internal/infra/token/token.go`（37行）** — Bearer トークン
- `Generate() (plaintext, hash, err)`: crypto/rand 32バイト（256bit）→ hex 64文字平文 + SHA-256 ハッシュ
- `Hash(plaintext) string`: sha256 → hex 64文字（`device_tokens.token_hash VARCHAR(64)` に収まる）
- **方針: 平文は発行時のみ表示、DB には SHA-256 ハッシュのみ保存**

### 5.3 デバイスAPI認証ミドルウェア ✅

**`internal/auth/device_auth.go`（84行）** — `DeviceAuth(DeviceAuthConfig{Repo})`
- `Authorization: Bearer <token>` を `token.Hash` で SHA-256 化 → `GetDeviceTokenByHash` で照合
- **期限切れは SQL 側**で除外（`WHERE expires_at IS NULL OR expires_at > NOW()`）
- エラー: ヘッダ欠如/スキーム不正/空/不一致 → 401、DB失敗 → 500
- 成功時 `c.Set` で `user_id` / `token_id` を格納。`last_used_at` は best-effort 更新（失敗無視）
- ゲッター: `auth.UserID(c)` / `auth.DeviceTokenID(c)`（未設定時 `int64(0)`）

### 5.4 センサーデータ受信ハンドラ ✅

**`internal/handler/sensor_api.go`（109行）** — `SensorAPI.Create`（`POST /api/sensor-data`）
- 処理フロー: `ShouldBindJSON`（binding） → `GetDevice` で存在/所有者確認（403） → `CreateSensorReading` INSERT → `UpdateDeviceLastCommunicated`（失敗無視） → **201**
- エラー振り分け: JSON構文/型エラー → 400、binding違反 → 422、存在しない device_id → 422、DB失敗 → 500
- **⚠️ アラート判定は未実装**（L57 に「Step18/フェーズ7 で同期追加予定」コメントのみ）

## 7. DB層リファレンス（6テーブル + sqlc 全37クエリ）

### 設計方針
- **外部キー制約は張らない**（参照整合性はアプリ層 JOIN で担保）
- **論理削除**（`deleted_at`）を主要テーブルに採用、sqlc クエリは常に `WHERE deleted_at IS NULL`
- **UNIQUE は部分インデックス**（`WHERE deleted_at IS NULL`）で soft-delete 衝突を回避
- マスターデータは Go 定数 + VARCHAR + CHECK 制約
- **集計テーブルは持たない**（24h は生データ、7日/30日は `GROUP BY DATE(recorded_at)`）

### テーブル一覧

| テーブル | 論理削除 | 主な制約・インデックス |
|---------|:-------:|----------------------|
| **users** | - | `email` UNIQUE。物理削除運用 |
| **devices** | ✅ | `mac_address` CHECK（`^([0-9A-Fa-f]{2}:){5}[0-9A-Fa-f]{2}$`）+ 部分UNIQUE。`user_id`/`is_active` 部分INDEX |
| **device_tokens** | - | `token_hash VARCHAR(64)` UNIQUE。`abilities JSONB`。物理削除（失効は `expires_at`）|
| **sensor_readings** | ✅ | `temperature/humidity NUMERIC(5,2)` CHECK（-40..125 / 0..100）。`(device_id, recorded_at DESC)` 複合INDEX |
| **alert_rules** | ✅ | `metric`/`operator` CHECK。`(device_id, is_enabled)` 部分INDEX |
| **alert_histories** | ✅ | metric/operator/threshold/actual_value を**非正規化保持**。未通知バナー用部分INDEX |

> **`recorded_at`（デバイス計測時刻）と `created_at`（サーバ受信時刻）の差分から通信遅延を分析可能。**

### sqlc 全37クエリ関数（`internal/repository/querier.go`）

設計書「sqlc リレーション方針」の9関数はすべて存在（**欠落ゼロ**）。

| テーブル | 関数（数） |
|---------|-----------|
| users (6) | `GetUser` / `GetUserByEmail` / `CreateUser` / `UpdateUser` / `MarkUserEmailVerified` / `DeleteUser` |
| devices (7) | `GetDevice` / `GetDeviceByMacAddress` / `ListDevicesByUser` / `CreateDevice` / `UpdateDevice` / `UpdateDeviceLastCommunicated` / `SoftDeleteDevice` |
| device_tokens (5) | `CreateDeviceToken` / `GetDeviceTokenByHash` / `UpdateDeviceTokenLastUsed` / `ListDeviceTokensByUser` / `DeleteDeviceToken` |
| sensor_readings (7) | `CreateSensorReading` / `GetLatestSensorReading` / `ListRecentSensorReadings` / `ListDailySensorAggregates` / `GetSensorReadingsSummary` / `ListSensorReadingsPaginated` / `CountSensorReadingsInRange` |
| alert_rules (7) | `GetAlertRule` / `ListAlertRulesByDevice` / `ListEnabledAlertRulesByDevice` / `CreateAlertRule` / `UpdateAlertRule` / `ToggleAlertRule` / `SoftDeleteAlertRule` |
| alert_histories (5) | `CreateAlertHistory` / `ListUnnotifiedAlertHistoriesWithDevice` / `ListAlertHistoriesPaginated` / `CountAlertHistoriesInRange` / `MarkAlertHistoryNotified` |

**JOIN クエリの注意点:**
- `ListAlertHistoriesPaginated` / `CountAlertHistoriesInRange` / `ListUnnotifiedAlertHistoriesWithDevice` は alert_histories→alert_rules→devices の3テーブル JOIN で `devices.name` を取得、3テーブルすべて `deleted_at IS NULL` 確認、`d.user_id` でテナント分離
- `ListAlertHistoriesPaginated` は `sqlc.narg('device_id')` で「未指定なら全デバイス対象」を実現（生成コードは `*int64`）
- 集計クエリの `AVG` は `::NUMERIC(5,2)`、`COUNT(*)` は `::BIGINT` でキャスト。**`MAX`/`MIN` は未キャストで `pgtype.Numeric`(nullable) になる点に注意**

### Go 型マッピング（`models.go`）
- `NUMERIC` → `pgtype.Numeric`、`TIMESTAMPTZ` → `pgtype.Timestamptz`、`JSONB`(abilities) → `[]byte`
- NULL許容 `location` → `*string`（`emit_pointers_for_null_types`）、NOT NULL の id 等 → 素の `int64`/`string`

## 9. 次の一手（クリティカルパス）

1. **scs Session 認証基盤の導入**
   `go.mod` に `alexedwards/scs/v2` + PostgreSQL ストア追加 → `internal/auth/session_auth.go` 作成。`SESSION_SECRET` は config で検証済みだが未使用のまま放置されており、ここが起点。**Web UI 全工程がこの1点に依存する。**

2. **Web UI ルーターグループ新設**
   `main.go` に SessionLoad + MethodOverride を適用した non-api グループを追加し、プレースホルダの `/` を置換。MethodOverride が無いとモックの `_method` オーバーライドフォームが機能しない。

3. **templ レイアウト/コンポーネント抽出**
   モック9画面はヘッダ/サイドバーをベタ書き重複保持。まず site-header / sidebar を共通 templ 部品化 → guest-layout（login/register）と認証後 layout の2系統を確立するのが手戻り最小。空ディレクトリは用意済み。

4. **認証フロー（login/register/logout）を最初の templ+HTMX 実装対象に**
   `GetUserByEmail` / `CreateUser` と bcrypt（seed で実績）が揃い、Session 確立の検証も兼ねられる。依存順としてダッシュボードより先に着手すべき。

5. **アラート判定は別レーンで並行**
   Web UI と独立して sensor_api.go の Create 内に `ListEnabledAlertRulesByDevice → Evaluate → CreateAlertHistory` を追加（クエリ・Evaluate とも実装済みのため接続のみ）。設計書どおり同期実行で着手可能。

## 10. リスク・不整合

- ⚠️ **scs が単一障害点**: Web UI = scs Session が中核前提だが、scs は依存にも実装にも一切存在しない。設計書は「将来実装」と注記済みなので方針違反ではないが、Web UI 実装の全工程がこの1点に依存。
- ⚠️ **モジュールパスとアカウント不一致**: go.mod・OpenAPI contact が `github.com/HiroshiKawano/go_iot` を指すが、コミットユーザーは `stamppK`（移管/フォークの可能性。CI/配布時の齟齬リスク）。
- ⚠️ **Blade 表記の残存**: `openapi.yaml` の description に「Blade/templ 返却系」とあり、本プロジェクトの templ 採用方針と不整合（Laravel 用語の混入）。
- ⚠️ **`/health` のミドルウェア適用が設計書とズレ**（設計書「Logger のみ」 vs 実装 Logger+Recovery）。実害は小さいが表が実態とずれている。
- ⚠️ **pgconv の握り潰し**: `NumericToFloat`/`TimestamptzToTime` が変換失敗・無効値を error でなく 0/ゼロ値で黙って返すため、「正常な0」と「変換失敗」を区別不能。集計クエリの `MAX`/`MIN` が未キャストで nullable な点と複合し、表示層で誤った0を出すリスク。
- ⚠️ **外部キー制約なし**: 参照整合性をアプリ層 JOIN に委ねる設計のため、device 削除時の孤児レコードや不正 device_id 混入を DB レベルで防げない。
- ⚠️ **テストカバレッジ未達**: バインド/バリデーション層とユーティリティに限定。DB 接続を伴う主要経路が未検証で80%要件に未達。sqlc `emit_interface=true` で Repo モック生成可能なため、テスト基盤整備が先行課題。
- ⚠️ **seed コメントの件数矛盾**: パッケージコメント（L12「各デバイス…計576件」）と内部コメント（L159「288件」）で表現が矛盾。実際は各288件・2台合計576件。
- ⚠️ **gen-token の Makefile 制約**: `-abilities`/`-expire-days` を渡せず固定。ability ベース認可も未実装でトークン権限管理が運用上ザル。
- ⚠️ **Scalar UI の CDN 依存**: go:embed されておらず、オフライン/閉域環境ではドキュメント UI が表示されない（OpenAPI 仕様と index.html のみ同梱）。
