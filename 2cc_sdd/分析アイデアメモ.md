### フェーズ1. 圃場所在地の地域セレクト 〔スタンプ決定／2026-06-26 平坦化に改訂〕

**spec**: `device-location-select` ／ **依存**: なし（最初の spec）

**狙い**: device 登録の「場所」自由入力は詳細住所・GPS を誰も入れない。**沖縄の地域を選ぶ単一の検索可能セレクト**にして粗くても入力させ、**地点を集計・比較可能なキー**にする（横断①）。

> **改訂（2026-06-26）**: 当初は「市町村→地区の2段カスケード＋市町村/地区マスタ**テーブル**」案だったが、**沖縄の農家≒60歳以上で市町村合併前(旧町村)の呼び名で土地を認識する**実地知見（ユーザー言明・権威）を受け、**旧町村名で直接選べる単一セレクト＝平坦化**へ転換。新合併名を先に選ばせるカスケードは当人の認識と逆順で、実装も単純（カスケード/HTMX/Tom Select swap 不要）。確定データ・設計は `.kiro/specs/device-location-select/`。

**作るもの**

- devices の `場所`(自由入力) に **`locality` 1列（任意）**を追加（既存値は非破壊移行）。**親市町村は** `Locality.Municipality()` **で導出**（DB列に持たない＝YAGNI・将来 SQL 集計で非破壊追加可）
- **地域マスタ53件（未合併36市町村＋平成合併5市町村の旧町村17）を Go 定数で**（structure.md §100＝マスタはテーブルでなく Go 定数+VARCHAR+CHECK）。例: 佐敷→南城市。**同名注意（旧具志川市[→うるま市]≠旧具志川村[→久米島町]）は表示「具志川（◯◯市）」で区別**
- device 登録・編集フォームを**単一の検索可能 select** 化（Tom Select・カスケード無し）。表示は認識名「旧町村（現市町村）」／未合併は市町村名
- device-show・dashboard での所在地表示
- **システムは沖縄専用ゆえ県は固定。字/番地・緯度経度・字レベル細粒度は取らない**。※ ハウス内センサ位置（内/外・東/西）は別軸（フェーズ10）

**▶**: チャートなし。フォーム＋スキーマ(locality 1列)＋Go定数マスタの改修。カスケード/HTMX 不要で単純。地域別集計・圃場ベンチマーク（フェーズ13）の前提キー。

### フェーズ8. 長期トレンド・季節サマリ 〔明示／強く示唆〕

**spec**: `seasonal-trend` ／ **依存**: 長期蓄積・**研究用分析ページの器（左サイドメニュー新設＝横断⑬）**

> **配置（2026-06-29 決定・ユーザー提案）**: 本フェーズは **device-show（デバイス詳細）ではなく、左サイドメニューに新設する独立した「統計分析」ページ**に置く（横断⑬）。device-show は直近監視に特化し、長期トレンドは時間軸・集計軸（横断①）が根本的に違うため分離する。**この統計分析ページが P9 相関・P10 多地点・P13 ベンチマークも受ける研究用分析画面系列の初号**となり、ナビ・ルート・所有者認可(BOLA)・集計軸セレクタ（デバイス/地点/作物/期間）の器を本フェーズで用意する。

**狙い**: 多年・多シーズン横断で季節性・平年比を論じる（データ主権の本領）。device-show と分離した統計分析ページで、デバイス/地点/作物/期間（横断①）を選んで長期トレンドを検定・可視化する。

> ⚠️ **統計的健全性が本フェーズの肝（2026-06-29・`nishizawa.md` 反映）**: 温湿度は自己相関が極めて強く（λ≈0.9+）、**素の線形回帰／Mann-Kendall は p値を桁で過大評価し偽トレンドを量産する**（付録G-1: 分散膨張係数 (1+λ)/(1-λ)）。客は研究者・査読品質を要求する（付録B-1）ため、**自己相関補正・Sen's slope併記・多重比較補正・検出力の留保**を必須前提とする。詳細＝**付録G**（トレンド検定の統計的健全性）。

**作るもの**

- 日次→週次→月次→年次へロールアップした統計サマリ ／ 日較差ΔTの日次推移
- **トレンド検定セット（付録G）**: 自己相関補正つき Mann-Kendall（タイ/連続性補正・G-2）＋ **Sen's slope**（傾きの頑健推定・G-3）。**MK=有無／Sen=大きさ**をペアで提示。素のMKは独立前提なので prewhitening / Seasonal MK（月別）を前置き（G-1）
- **go-echarts でアプリ内可視化**（当初の CSV出力のみ想定から転換＝2026-06-29・ユーザー提案）: 月次ロールアップ折れ線＋ **Sen's slope の markLine（トレンド線）**＋ **MK 判定の信号バッジ**（有意↑/有意↓/非有意・**検出力留保つき＝G-8**）。markArea で有意区間・正常帯を帯表示。アプリ内は「一次判定」、多重比較込みの最終確定はCSV外（二段構え＝G-11）
- **月別/季節別トレンド**（年集約は月別の符号反転を潰す＝G-5 の警告）。集計軸（横断①）に月別軸を確保
- 平年値との比較（今季 vs 平年）。**ただし平年値の不確実性・基準期間依存に留意し、年数不足時は気象庁長期平年値を外部基準に借りる**（G-9）
- **センサ交換の不連続（ギャップδ・Ng）を不連続点付き回帰で織り込み、器差を偽トレンドにしない**（G-4）。※器差の構造化はフェーズ10の役務（フェーズ5は単一デバイス品質メタに限定）
- **検出力の留保表示**: N（年数）不足時は「検定」でなく Sen傾き＋符号に留め、「非有意≠トレンド無し」を明示（G-8）／ 多地点×多指標×多月でMKを大量に回すなら FDR/Bonferroni 補正（G-7）

**使う手法 → 付録A/G**: A／ B（ロールアップ・回帰）／ **G（自己相関補正・MK・Sen・ギャップδ・多重比較・検出力留保）**／ F

**▶**: ◎ 平年比・ロールアップ ／ ◎ **MK＋Sen トレンド（アプリ内Go・markLine＋有意なら markArea＝G-10 で外出しから昇格）** ／ ○ 回帰トレンド線（探索用・有意性主張はCSV外＝G-11）／ △ STL/ACF/Edgeworth/ブロックブートストラップ/SARIMA（CSV外出し・G-6/G-10）。

### フェーズ9. 温湿度相関・環境応答 〔強く示唆〕

**spec**: `temp-humidity-correlation` ／ **依存**: 多地点（内外比較で活きる＝フェーズ10）

**狙い**: 相関で施設の挙動（熱容量・換気の効き・気密）を物理的に理解。出発点（本命はVPD）ゆえ優先度は中位。

**作るもの**

## 第2章. go-echarts での可視化方針と優先順位（2026-06-26 検証反映）

device-show のグラフを go-echarts（Apache ECharts）へ移行した（**方式B**＝サーバーで option(JSON) を組み、クライアントで `echarts.init`/`setOption`）。各フェーズの「▶」がこの方針の各論。

### 2-1. 大前提：「描けるか」はほぼ全部 YES。律速は計算層

- **可視化**: ECharts は line／多軸／**markArea（帯塗り）**／**visualMap**／**heatmap**／**calendar**／scatter／gauge／**dataZoom**／**connect** を持ち、全フェーズの「グラフ」は表現可能。方式B なら opts に無いキーもクライアントで足せる。→ **実質 全フェーズ可**。
- **律速は計算**: VPD/GDD/露点/THI/AH は即計算で軽い。**STL・ACF・SARIMA・Edgeworth・ブロックブートストラップは重く CSV 外出し**（横断⑧）。**ただし Mann-Kendall／Sen's slope は依存ゼロ・軽量でアプリ内Go実装に昇格**（2026-06-29 修正＝付録G-10）。



### 2-2. クラッタ対策の設計パターン（最重要）

1. **1チャート＝主役1〜2系列**。デフォルトは ◎ のみ。
2. **legend** `selected:false` **でトグル化**。○ は凡例クリックで出す（方式B でクライアント制御・追加コストほぼゼロ）。
3. **線を増やすより「帯」で見せる**。固定しきい値帯＝**markArea**、曲線追従帯（SMA±kσ）＝**2系列の積み上げ area**。
4. **研究用画面 と 農家共有画面を分離**（横断④）。さらに**直近監視（device-show）と長期統計分析ページ（横断⑬・P8で新設）を分離**＝device-show にトレンド検定を積まない。
5. **派生指標は温湿度と別パネル/タブ**。
6. **計算層と描画層を分離**（重い統計は CSV 外出し）。

## 第3章. 設計ガードレール（全 spec が従う前提）

各フェーズ spec を作る際に**共通して守る設計判断**。spec 自体にはせず、最初に決めておく（後付けは破綻する）。

- **① 集計軸を最初に固定**（圃場×地点×作物×期間×指標）。どの軸でも横並び集計・CSV化できること。後付けで軸を足すとスキーマ破綻。→ フェーズ1・4・13 が依存。
- **② スキーマに派生指標・多項目・品質フラグ列を後付け可能に設計**（VPD/露点/GDD/THI/日較差/AH／CO2等／品質フラグ）。**実列の追加は必要なフェーズの spec で**（YAGNI）。ただし集計軸を壊さない移行にする。
- **③ データ品質メタ層を最初から想定**（品質フラグ列を schema に確保）。実装はフェーズ5 spec。
- **④ 研究用画面と農家共有画面を分離**。高度指標（VPD/GDD/病害）は信号色・適正帯バンド・平易ラベルへ翻訳。→ フェーズ13。
- **⑤「時間割合（滞在率）」の発想**。瞬間値や平均でなく「1日の何%が適正帯か」。→ フェーズ3。
- **⑥ 作物メタデータの保持**（作物マスタ）。GDDのTbase・VPD適正帯・病害モデルは作物依存。→ フェーズ3・6・7・13。
- **⑦ 長期蓄積・保持期間を自前制御**（積算温度・季節性・SARIMA の可否を左右）。→ フェーズ8・15。
- **⑧ 計算層と描画層を分離**。**ただし MK/Sen は依存ゼロ・軽量でアプリ内Go実装に昇格**（2026-06-29・付録G-10）。CSV外出しは Edgeworth／ブロックブートストラップ／STL／ACF／SARIMA／レアイベント厳密p値／変化点 Ng 推定に限る。アプリ内 ECharts は「軽い指標＋帯＋ヒートマップ＋MK/Sen トレンド」に集中。→ フェーズ8・11・12。
- **⑨ トレンド検定は自己相関補正を必須前提に**（付録G-1）。温湿度は λ≈0.9+ で素の回帰/MK は偽トレンドを生む（分散が (1+λ)/(1-λ) 倍に膨張）。ロールアップ＋季節除去＋N_eff／prewhitening を前置きする。→ フェーズ8・11・12。
- **⑩ MK と Sen's slope は必ずペア**（付録G-3）。MK=有無／Sen=大きさ。OLS傾きは台風スパイクに弱いので Sen 併記必須。→ フェーズ8・11・12。
- **⑪ 多重比較を制御**（付録G-7）。多地点×多指標×多月で MK を大量実行＝FDR/Bonferroni 補正を CSV外解析の前提に。→ 全トレンド/相関フェーズ（8-13）。
- **⑫ 検出力・N年数の留保**（付録G-8）。数年データで「トレンドあり」と断定しない。当面は Sen傾き＋符号の記述統計に留め「非有意≠トレンド無し」を明示。平年値も年数不足時は外部基準（気象庁長期平年値）を借りる（G-9）。→ フェーズ8・11・12。
- **⑬ 統計分析は device-show と分離した「研究用分析ページ」に置く**（2026-06-29 決定・横断④の具体化・付録B-1の研究者ニーズ）。**device-show（デバイス詳細）は単一デバイスの直近監視（24h〜30日のリアルタイム可視化）に特化**し、**長期トレンド・季節サマリ（P8）・相関（P9）・多地点比較（P10）・圃場ベンチマーク（P13）は左サイドメニューに新設する独立した「統計分析」ページ系列**へ。理由＝(a)時間軸が違う（直近 vs 多年蓄積）、(b)研究者の本領は地点/作物/期間を横断した集計（横断①）で単一デバイス固定の device-show に乗らない、(c)device-show のパネル過密回避。**P8 がこのページ系列の初号**（ナビ・ルート・所有者認可(BOLA)の器を用意）。→ フェーズ8・9・10・13。
- **⑭ 可視化は go-echarts 一本に統一、計算は gonum/stat＋自前Go**（2026-06-29 gonum調査）。統計プロット（信頼区間帯・ヒストグラム・箱ひげ・ACF・Q-Q・トレンド線・MKバッジ）は ECharts 標準（bar/scatter/markArea/markLine/visualMap）で全て描けるため、**`gonum/plot`（静的画像生成）は画面描画に採用しない**（二系統化＝CSS単一ソース運用・モック運用・templ/HTMX動的性・テストの全層で二重化コスト、UX断裂）。計算は `gonum/stat`・`distuv` を p値化等の土台に使い、検定本体は自前Go（→付録G-10/G-11）。※ `gonum/plot` は将来「論文・報告書用の静的高品質図（SVG/PDF/EPS）エクスポート」が研究上必要になった場合のみ、その用途限定で再検討。
- ※ チャートの描画クラッタ対策（主役1〜2系列＋legendトグル／線より帯で見せる）は **§2-2** に集約。

- **自己相関係数 λ の実測値**（付録G-1 の N_eff・分散膨張を左右）。サンプリング間隔ごとに実データで λ̂ を測り、ロールアップ粒度（日次/月次）と prewhitening の要否を決める。
- **トレンド検定の外部解析スタック**（付録G）。CSV外出し先を R（`trend`/`zyp`）か Python（`pyMannKendall`/`statsmodels`）に決め、Hamed-Rao補正・ブロックブートストラップ・多重比較補正の標準手順を定める。
- これらは実データのスキーマ（`docs/database_snapshot/`）を見て前提を詰める。

# 付録A. 数式・指標リファレンス（A〜F カタログ）

## A. 基本統計量（土台・すぐできる）


| 指標             | 数式                        | 農業的な意味                           |
| -------------- | ------------------------- | -------------------------------- |
| 平均 μ           | `μ = (1/N)Σxᵢ`            | 期間の代表値                           |
| 中央値 / 最頻値      | 順序統計量                     | 外れ値に強い代表値                        |
| 最高 / 最低        | `T_max`, `T_min`          | 日内の極値。熱・冷害の把握                    |
| 標準偏差 σ         | `σ = √((1/N)Σ(xᵢ-μ)²)`    | 環境の安定度。大きい＝乱高下                   |
| 分散 σ²          | `σ² = (1/N)Σ(xᵢ-μ)²`      | ばらつきの基礎量                         |
| 変動係数 CV        | `CV = σ/μ`                | 平均で正規化したばらつき。温度帯が違う日同士を比較できる     |
| 日較差 ΔT         | `ΔT = T_max - T_min`（その日） | **果実の糖度・着果に直結**。農業では平均より重要なことが多い |
| 範囲 / 四分位 / IQR | `IQR = Q3 - Q1`           | 外れ値に強いばらつき。異常検知の土台               |


**ポイント**: 日較差は地味だが、トマトの糖度や花芽分化に効く実用指標。最初から入れる価値がある。

## B. 時系列・移動平均系

```
単純移動平均 SMA  : SMA_t = (1/N) Σ_{i=t-N+1}^{t} xᵢ
加重移動平均 WMA  : WMA_t = (Σ wᵢ·xᵢ) / (Σ wᵢ)        ← 直近ほど重み大
指数移動平均 EMA  : EMA_t = α·x_t + (1-α)·EMA_{t-1},  α = 2/(N+1)
                   ← 直近を重視。反応が速い
乖離率(disparity) : 乖離率 = (x_t - SMA_t) / SMA_t × 100  [%]
変化率(微分)      : Δ = (x_t - x_{t-1}) / Δt   ← 急変検出（ドア開放・機器停止）
```

- **ボリンジャーバンド的アプローチ**: `SMA ± k·σ`（移動窓のσ）で「正常帯」を作り、外れたら警告。
- **STL分解**: `トレンド + 周期(日周変動) + 残差` に分解。残差で「いつもの昼夜変動を除いた異常」を拾う。
- **自己相関 ACF**: ラグ24hで強い相関 → 健全な日周リズム。**この自己相関こそトレンド検定を壊す主犯**（付録G-1）。
- **トレンド検出**: 線形回帰の傾き、または Mann-Kendall 検定＋**Sen's slope（傾きの頑健推定）**。⚠️ **温湿度は自己相関が強く、素の回帰／MK は偽トレンドを生む。自己相関補正・タイ補正・多重比較・検出力の留保が必須＝詳細は付録G**（トレンド検定の統計的健全性）。

## C. 温度 × 湿度の関係

```
ピアソン相関  : r = Σ(xᵢ-x̄)(yᵢ-ȳ) / √(Σ(xᵢ-x̄)²·Σ(yᵢ-ȳ)²)
               ← 温度と相対湿度は通常「負の相関」(昼:温度↑→RH↓)
相互相関(ラグ) : 湿度が温度に何分遅れて追従するかを検出
共分散        : cov(x,y) = (1/N)Σ(xᵢ-x̄)(yᵢ-ȳ)
回帰          : y = a·x + b で関係を定量化
```

**注意**: 物理的に最も意味のある"乖離"は **飽差（VPD）**。「温度40℃・湿度50%」と「温度20℃・湿度50%」は植物にとって別物。これを1指標に統合したのが D①VPD。

## D. 農業特化の派生指標（★本命）

温度と湿度の **2つだけ** から計算できて、生データより遥かに実用的。

### ① 飽差 VPD（最重要）

```
飽和水蒸気圧 es(T) = 0.6108 · exp(17.27·T/(T+237.3))   [kPa]   (Tetens式)
実水蒸気圧   ea     = es(T) · RH/100
飽差 VPD            = es(T) − ea = es(T)·(1 − RH/100)   [kPa]
```

植物の蒸散・気孔の開閉を支配。施設園芸の環境制御は事実上これを目標値に管理（多くの作物で 0.3〜1.5 kPa が適正帯）。

### ② 積算温度 GDD

```
GDD = Σ max( (T_max + T_min)/2 − T_base , 0 )
      T_base は作物固有（例: 多くの作物で 10℃）
```

収穫日予測・生育ステージ判定に直結。

### ③ 露点温度・結露リスク

```
γ  = ln(RH/100) + 17.27·T/(T+237.3)
露点 Td = 237.3·γ / (17.27 − γ)   [℃]
結露リスク: 葉面温度 ≦ Td のとき発生
```

結露 → 灰色かび病・うどんこ病の温床。病害アラートの基礎。

### ④ 絶対湿度（容積絶対湿度）

```
絶対湿度 AH ≈ 217 · ea / (T + 273.15)   [g/m³]
            （ea は ①の実水蒸気圧 [kPa] を Pa 換算して使う）
```

温度に依存しない「実際の水分量」。換気・除湿判断に有用。

### ⑤ 病害リスクモデル

「温度帯 × 葉面湿潤時間」で発病確率が上がる作物が多い。**葉面湿潤時間（高湿度継続時間）** を積算してリスクスコア化。

### ⑥ THI（温湿度指数 / 高温ストレス）

```
THI = 0.8·T + (RH/100)·(T − 14.4) + 46.4
```

高温多湿の複合ストレスを1指標化。

# 付録G. トレンド検定の統計的健全性（自己相関・MK・Sen・レアイベント）

> **出典**: `nishizawa.md`（西澤誠也氏「気象学分野におけるトレンドの検出について」CPSセミナー2008・神戸大学大学院理学研究科）を農業IoT（温湿度時系列）向けに翻案（2026-06-29）。**フェーズ8（seasonal-trend）の中核であり、フェーズ5・6・11・12 にも横断する**。付録A-B「トレンド検出」の正式版。
>
> **一言**: 付録A-B が「トレンド検出＝線形回帰の傾き or Mann-Kendall」と一行で済ませていたのは**危険**。温湿度のように自己相関の強いデータでは、素の回帰／MK は p値を激しく過大評価し**偽トレンドを量産する**。客（眞境名さん）は研究者・査読品質を要求する（付録B-1）ため、以下は「あれば良い」ではなく「無いと成立しない」前提。

## G-1.【最重大】自己相関の罠 — 素の回帰／MK は偽トレンドを生む

西澤資料が**トレンド検出を困難にする筆頭要因**に挙げるのが自己相関（スライド3）。線形回帰モデル（Weatherhead 1998・スライド5-6）は誤差に AR(1) を仮定する:

```
Xn = a·n + b + vn        （a=トレンド, b=定数, vn=自己相関する誤差）
vn = λ·vn-1 + εn         （λ=自己相関係数, εn=独立ランダム変数）
```

このとき傾き推定値 â の分散は（スライド6）:

```
Var(â) ≈ Var(v)·(12/N³)·((1+λ)/(1-λ))·(1/{1 - 3γ(1-γ)})     γ=(Ng-1)/N
                          ↑ 自己相関による分散膨張係数
```

**(1+λ)/(1-λ) が要点**。温湿度は10分〜1時間サンプリングで隣接相関 λ≈0.9 以上、日周で擬似周期を持つため、λ=0.9 なら分散は **約19倍**、λ=0.95 なら **約39倍**に膨らむ。言い換えると**実効サンプル数が見かけの N より桁で少ない**:

```
有効サンプル数 N_eff ≈ N·(1-λ)/(1+λ)
```

→ 標準誤差をこの N_eff で評価し直さない限り、回帰の p値も MK の Z も有意性を**過大評価**し、「有意な温暖化」を偽検出する。

**処方箋（フェーズ8の必須前提）**:
1. **ロールアップで日周相関を弱める** … 生粒度→日次→月次（メモ既存のロールアップを統計的根拠で位置づけ直す）。
2. **季節成分を先に除く** … 月次でも年周が残る → Seasonal Mann-Kendall（月別に別々に S を計算して合算）か STL残差化を**前置き**。
3. **残差の λ̂ で標準誤差を膨らませる** … N_eff で再評価（実装はGo数行）。MK 側は prewhitening（TFPW: Trend-Free Pre-Whitening）または **Hamed-Rao（1998）の分散補正MK**。
   - ※ 分散膨張の「原理」は西澤資料収録。具体的な補正アルゴリズム（Hamed-Rao・prewhitening・Yue-Pilon）は**外部の標準手法**で本資料の射程外。CSV外出しで `pyMannKendall`／`statsmodels` 等に渡すのが現実的。

## G-2. Mann-Kendall 検定（タイ補正・連続性補正つきの正式版）

付録A-B が名前だけ挙げていた MK の中身（スライド11-12・Kendall 1938・ノンパラメトリック）:

```
統計量 S    = Σ_{n<m} sign(Xm - Xn)            （sign: +1 / 0 / -1）
分散 Var(S) = (1/18)·[ N(N-1)(2N+5) - Σi ti(ti-1)(2ti+5) ]
              ↑ タイ補正項（ti = 同値グループ i のデータ数）
標準化 Z    = (S-1)/√Var(S)  if S>0
            = 0              if S=0
            = (S+1)/√Var(S)  if S<0       （±1 = 連続性補正）
→ Z を標準正規分布に通して p値
```

- **タイ補正は必須**: 温湿度はセンサ分解能（0.1℃刻み等）や日数系列（熱帯夜0日の年が複数）で**同値（タイ）が頻発**。タイ補正項を省くと Var(S) が歪み、Z が過大／検出力低下。
- **ノンパラの利点**: 分布の形によらない（スライド11）＝亜熱帯の歪んだ温湿度・台風スパイクに強い。
- **欠点**: 検出力はパラメトリック法より低いことが多い（スライド11）＝N不足では「非有意」が出やすい（→G-8）。素のMKは**独立前提**なので自己相関補正（G-1）と必ずセット。

## G-3. Sen's slope（トレンドの「大きさ」の頑健推定）— MK と必ずペア

付録A-B・フェーズ8に**完全に欠落していた**手法。西澤資料は MK と Sen's slope を**必ずペア**で扱う（スライド4・13・25）:

```
Sen's slope = median{ (Xn - Xm)/(n - m) }   全ペア (n>m) の傾きの中央値
```

- MK＝「増えているか否か（有無の検定）」／Sen＝「**何℃/年・%/年・日数/年か**（大きさの点推定）」の**役割分担**。
- **中央値ベース＝外れ値に頑健**。沖縄は台風スパイクで最小二乗（OLS）傾きが歪むため、Sen's slope が必須。
- ECharts では Sen 傾きを `markLine`、信頼帯を2系列積み上げ `area` で描ける（§2-3）。
- **注意**: Sen の信頼区間（Kendall順位法）は西澤資料にもメモにも数式が無い。点推定は Go 内で完結、区間が要るなら G-6 のブートストラップ（CSV外）か R/Python（`trend`/`zyp`）。区間も自己相関補正（N_eff）を併せないと狭すぎる。

## G-4. ギャップ δ 項 — センサ交換の不連続をトレンドに織り込む（器差はフェーズ10で構造化 → フェーズ8で織り込み）

西澤資料は「データの質のギャップ＝観測機器の変化」をトレンド検出の困難要因に挙げ（スライド3）、回帰モデルに**ステップ項 δ** を組み込む（スライド5-6）:

```
b = b0          (n < Ng)        Ng = 変化点（センサ交換）位置
  = b0 + δ      (n ≥ Ng)        δ  = 器差オフセット（ジャンプ量）
E[â] = a + {(h1h5 - h2h4)/(h1h3 - h2²)}·δ
       ↑ δ を無視するとトレンド推定がこの分だけバイアスする
```

- **センサ交換時刻 Ng・器差オフセット δ を「不連続点付き回帰」としてトレンドモデルに織り込む**。これをやらないと、交換前後を素直に1本回帰したとき**器差が偽トレンドになる**。塩害で多台数・交換頻発の沖縄では実害が大きい。
- ⚠️ **担当フェーズの実態（2026-06-29 実装調査で補正）**: 器差の検出には複数センサ＝**位置軸**が要るため、**フェーズ5（data-quality-meta）は器差・交換履歴の構造化を意図的にフェーズ10（multipoint-compare）へ defer 済み**（requirements Req7「器差スコープ境界」）。フェーズ5は単一デバイスの品質メタ（欠測/stuck/物理異常/外れ値）に限定。よって **Ng・δ の構造化保持は P10、それをトレンドへ不連続点として織り込むのは P8** の役務＝**フェーズ5・8 に「器差受け渡し」を今足す必要はない**（受け渡し先の P8 も未実装＝YAGNI）。
- **注意**: δ 推定には交換前後の重複観測期間が要る（事後では分離困難）。Ng が複数（多段 δ）なら γ=(Ng-1)/N が中央付近で分散膨張も最大化（スライド6の 1/{1-3γ(1-γ)}）＝交換タイミング自体が精度を左右。Ng 推定は Go 中難度＝当面は CSV外出しか、P10 メタから Ng を与える運用。

## G-5. レアイベントのトレンド検定 — 熱帯夜・台風・結露を統計にする（フェーズ6/11/12）

西澤資料の独自貢献（スライド15-21・26）が**農業の極端現象とほぼ一対一**で対応。鍵は**二分法**（スライド16 vs 18-21）:

**ルートA: 年間日数化 → 従来トレンド手法（MK/Sen）**
- 真夏日・熱帯夜・無降水日のように**閾値超えを年間日数へ集計すれば各年に1値が立つ**ので、G-2/G-3 をそのまま適用できる（スライド16）。
- 対応: **フェーズ12**（熱帯夜＝夜温≥25℃の年間日数）・**フェーズ6**（結露日数・葉面湿潤時間≥閾値の日数）・**フェーズ3**（VPD適正帯逸脱日の年間日数）。閾値判定は付録D③（露点Td）・付録E を流用。年集計→MK/Sen は軽く Go 実装可。※実装現状（2026-06-29）: フェーズ6 は日次の結露帯有無を集約する素地（`DewpointDailyRow`）を持つ＝最小追加（`HasCondensation` フラグ）で年集計でき、**年集計→MK/Sen は P8 の役務**。今フェーズ6 に MK/Sen を個別実装はしない（P8 の共通 `trend.go` に集約＝重複回避）。

**ルートB: 希少バイナリ → イベント位置の総和 S 統計量**
- 台風直撃のような**年に数回の希少事象は日数化すると 0/1 だらけ**で MK が無力。イベント発生**位置の前半/後半の偏り**で検定（スライド18-21）:

```
φn ∈ {0,1}（時刻 n でイベント発生=1）
S = Σ_{n=1}^{N} (n - (N+1)/2)·φn        前半偏り→負 / 後半偏り→正
p = M/N（総発生回数 M / N）を最尤推定（スライド21）
S の厳密分布は漸化式 Q_{l/2,N}（スライド20）で構築 → 両側 p値
```

- 対応: **フェーズ11**（台風）。前後半比較（検出力が低い素朴法・スライド17）の上位互換＝全イベント位置を使う。
- **重大な注意**: S の解析分布は「発生確率 p が時間一様」を仮定（スライド19の特性関数の積分解）。**台風（夏〜秋集中）・結露（梅雨集中）は月別 p が激しく非一様**で、そのまま当てると季節構造を偽トレンドと誤検出する。非斉次 p 版への拡張が要り、当面はデモ的な台帳の偏り提示に留める。S の符号だけなら Go で O(N)、厳密 p値（漸化式）は CSV外。

**成層圏二層構造の類比（スライド22-26）**: 西澤資料は「平均は寒冷化トレンド＋極端高温イベントの**月別**増減トレンド（Dec/Jan 増・Feb/Mar 減）」という二層を示す。沖縄版＝「平均気温・平均VPDの緩やかな温暖化トレンド（層1＝フェーズ8）＋台風影響日・熱帯夜・激甚結露の発生月の前倒し/件数増減トレンド（層2＝フェーズ11/12）」。スライド24の「閾値線＋○×マーカー」可視化はそのまま移植可（フェーズ11/12 の▶と整合）。スライド26「CO2増加で突然昇温が早まる」→「台風シーズン・出穂期高温の前倒し（作型・二期作の播種設計に直結）」＝極端イベントの**年内初発日**（初熱帯夜日・台風初日）の年系列に Sen's slope を当て「1年あたり何日早まったか」を出す筋へ翻訳できる。
- **月別符号反転の警告**: スライド24 は月で増減が逆＝**年集約すると相殺して見えなくなる**。フェーズ8の「年次ロールアップ」が月別情報を潰すと無力化する → 月別/季節別トレンド軸を集計軸（横断①）に最初から確保。

## G-6. 非正規・非独立への対処（ブートストラップ／Edgeworth）

- **ブロック・ブートストラップ**（スライド14）: 滞在率%（フェーズ3）・連続日数（フェーズ12）・回復時間（フェーズ11）など**解析式の無い統計量の信頼区間**を分布仮定なしで出す。**素朴な復元抽出（スライド14の原型）は自己相関を壊し信頼区間を過小評価**＝温湿度では連続区間単位の**ブロック**ブートストラップ必須（ブロック長 ≈ 相関長）。B×O(N²) で重く、乱数 seed 記録（再現性）必須＝**CSV外出し**。
- **Edgeworth展開**（スライド7-9, Nishizawa and Yoden 2005）: 高次キュムラント（歪度 κ4・尖度 κ6）で正規近似を補正。湿度（0-100%天井）・VPD（下限0で右歪）・降水（ガンマ）など非正規（スライド8）の理論的裏付け。ただし**高次キュムラント推定が小標本で極端に不安定**＋λ=0 前提（スライド7・10）で自己相関の強い温湿度に直接使えない＝**温湿度2変数・数年データには明確にオーバースペック**。同じ目的は MK／ブートストラップで代替。**実装せず「正規前提がどれだけ崩れるか」の前提チェックの根拠として引用**するに留める。

## G-7. 多重比較の制御（査読品質に必須）

15フェーズ×多地点×多作物×多指標×多月で MK／回帰／相関を**大量に回す**設計（フェーズ8-13）。p<0.05 で多数回検定すれば偽陽性が必然的に出る。**FDR（Benjamini-Hochberg）／Bonferroni 補正**への言及がメモに皆無＝研究・査読品質を要求する客に致命的。全トレンド／相関フェーズで多重比較補正を**CSV外解析の前提**として明記する。

## G-8. 検出力・必要データ年数の留保（「非有意 ≠ トレンド無し」）

西澤資料は MK の低検出力（スライド11）とレアイベントの検出困難（スライド17＝「数年に一度のイベントのトレンド検出は難しい・検出力がとても低い」）を繰り返し警告。**本案件は蓄積数年で N（年数）が小さい**＝フェーズ8/11/12 で「トレンドあり」と断定するのは時期尚早。
- **当面の運用**: 「検定（有意性）」を主張せず、**Sen's slope の傾き点推定＋符号**と記述統計（平年比・年次推移の可視化）に留める。有意性結論は長期蓄積後に限定。
- **誤読の禁止**: 「有意でない＝トレンドが無い」ではない（検出力不足）。この注記を各トレンドフェーズに添える。

## G-9. 平年値の不確実性（基準期間依存）

フェーズ8の「今季 vs 平年」（187行相当）は、平年値自体が標本平均で**不確実性を持ち、基準期間の取り方に依存**する（西澤資料の基準期間・観測機器変化の議論＝スライド3・5）。**新規案件では平年値を作るほどの年数が無い**＝平年比は当面成立しない。
- **折衷**: 気象庁アメダス長期平年値を**外部基準**として借り、自社データは超過判定のみに使う（フェーズ5データ品質メタ・横断⑦保持期間と連動）。自前平年を出すなら信頼区間と基準期間定義を明示し、年数不足時は平年比を出さない。

## G-10. Go実装 vs CSV外出しの線引き（横断⑧の精緻化）

メモは「STL・ACF・**Mann-Kendall** は重いので CSV外出し」と三者を一括退避（188・192・385行相当）。**これは過剰退避**。MK/Sen は依存ゼロ・閉形式・月次化すれば N≦数百で軽い:

| 手法 | 計算量 | 実装場所 |
| --- | --- | --- |
| MK統計量 S・Var(S)・Z（タイ/連続性補正） | O(N²)・依存ゼロ | **Goアプリ内**（ロールアップ後） |
| Sen's slope 点推定 | O(N²logN)・依存ゼロ | **Goアプリ内** |
| 自己相関 N_eff 補正（λ̂ で標準誤差膨張） | O(N) | **Goアプリ内**（数行） |
| MK の p値化（正規近似） | O(1) | **Goアプリ内**（`gonum/stat/distuv.Normal`） |
| Hamed-Rao 補正MK／prewhitening | 中 | **Goアプリ内**（自前実装・pyMannKendall/R modifiedmk を移植＋golden test） |
| 多重比較補正（FDR/Bonferroni） | O(N logN) | **Goアプリ内**（数十行） |
| ブロックブートストラップCI | B×O(N²) | **Goアプリ内**（非同期＋キャッシュ・乱数seed固定で再現性） |
| レアイベント S 統計量の符号 | O(N) | **Goアプリ内** |
| レアイベント S の厳密 p値（漸化式 Q） | O(N³)級＋検証重 | CSV外 or 非同期Go |
| Edgeworth／STL／ACF／SARIMA／変化点 Ng 推定 | 重・オーバースペック | CSV外（R/Python。当面アプリ内不要） |

**線引きの肝**: 生の分/秒粒度（数万点）に O(N²) を直接かけると Go でも破綻 → **必ず日次/月次ロールアップ後に適用**（横断⑦の保持期間設計が前提）。

**実装現状（2026-06-29 調査）**: 既存の `internal/chart/stats.go` に **`LinearFit`（汎用最小二乗・コメントに「将来の trend 検出が再利用可」と明記）** があり、`internal/chart/quality.go` に `IQRBounds`／分位関数（Sen の中央値計算に再利用可）。**統計ライブラリ依存ゼロ**（go.mod に gonum 等なし）＝付録Gの「依存ゼロでアプリ内実装可」は実証済み。派生指標は計算層（vpd.go/gdd.go/dewpoint.go）と描画層（*_echarts.go）が分離。日次ロールアップは SQL `ListDailySensorAggregates` で確立、**月次ロールアップは未実装**。P8 着手時は **`internal/chart/trend.go` 新設（MK／Sen／自己相関 λ̂・N_eff／prewhitening／Hamed-Rao／ブロックブートストラップ／多重比較）＋月次集計 SQL＋`seasonal_trend_handler.go`** が推奨経路（既存の単一責務パターンに従う）。**ライブラリ方針（2026-06-29 gonum調査）**: p値化に `gonum.org/v1/gonum/stat/distuv`（Normal の CDF/Quantile）を使う＝gonum コア v0.16.0 は既にビルドグラフ内（grpc 経由の間接依存）なので `go.mod` の direct 昇格のみで追加コストほぼゼロ・純Go・BSD-3。検定本体（MK・Sen・Hamed-Rao・ブロックブートストラップ・FDR）は自前の純Go関数。**数値正確性は pyMannKendall/R modifiedmk の既知データを golden test の期待値にして TDD で担保**（VPD符号反転の教訓＝数式の取り違えはテストが仕様前提を符号化すると捕捉できないため、外部参照実装との数値照合が安全網）。ブロックブートストラップは重い（B×O(N²)）ので非同期計算＋結果キャッシュ前提。

## G-11. システム内完結（全統計を Go で計算し go-echarts で表示）

**方針（2026-06-29 再整理・gonum 調査反映）**: フェーズ8 の統計分析ページ（横断⑬）では、**Mann-Kendall・Sen's slope・自己相関補正（N_eff/prewhitening）・Hamed-Rao 補正MK・ブロックブートストラップCI・多重比較補正（FDR/Bonferroni）を、すべて Go で計算し go-echarts でアプリ内表示する＝外部アプリ（R/Python）に依存しないシステム内完結**。これらは純粋な数値アルゴリズムで、p値化に `gonum/stat/distuv.Normal`（CDF/Quantile）を使い、検定本体は自前の純Go関数（`internal/chart/trend.go` 等）で実装できる（gonum は既にビルドグラフ内＝依存追加コストほぼゼロ・純Go・BSD-3）。**「CSV外でないと最終確定できない」わけではない**（旧版の二段構え表現を撤回）。

表示は「軽い一次判定」と「重い厳密判定」を段階表示する（**どちらもアプリ内**）:
- **一次判定（即時・軽量）**: 月次ロールアップ折れ線＋ Sen's slope の markLine ＋ MK の Z/p値バッジ（信号色）＋ 自己相関補正（λ̂・N_eff）。MK/Sen は依存ゼロ・O(N²)・月次なら N 小で軽い（G-10）。
- **厳密判定（非同期・重め・アプリ内表示）**: Hamed-Rao 補正MK・ブロックブートストラップCI（area帯）・多重比較補正（G-7）。計算は重い（B×O(N²)）ので**非同期計算＋結果キャッシュ**で go-echarts に表示。査読用の確定エビデンスもこの画面から得られる。
- **検出力の留保（G-8）併記必須**: 数年では非有意が出やすい・「非有意≠トレンド無し」。自己相関補正なしの素の回帰トレンド線は探索用に留める（有意性主張に使わない）。

**CSV出力の位置づけ**: 表示のためではなく、**研究者がデータ・検定結果を外部へ持ち出して独自に再解析する選択肢（データ主権・付録B-1）**として併設する。本システムの表示・計算が外部アプリに依存するわけではない。
