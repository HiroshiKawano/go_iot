## PL（プロジェクトリーダー）が抱く「幻想」を打ち砕き、なぜ `speckit` では動くものができないのか、その「記憶喪失」のメカニズムを解剖します。

### 1. 【深掘り】基本コマンドの真の意味と「失敗の火種」


| フェーズ      | **speckit（官僚型・形式主義）**               | **cc-sdd（現場型・実利主義）**                     | **極端な違いの解説**                                                     |
| --------- | ----------------------------------- | ---------------------------------------- | ---------------------------------------------------------------- |
| **要件定義**  | `/speckit.specify` → テンプレートを埋める作業。  | `npx cc-sdd specify` → AIとの「対話」で矛盾を潰す作業。 | `speckit` は「空欄を埋める」ことが目的になりがち。`cc-sdd` は「不明点を人間に吐き出させる」ことが目的。    |
| **設計・計画** | `/speckit.plan` → 手順書（マニュアル）を作る。    | `npx cc-sdd design` → **構造（骨格）**を定義する。   | **ここが最大の差。** `speckit` は「やり方」を書き、`cc-sdd` は「何を作るか（DB設計等）」を確定させる。 |
| **タスク化**  | `/speckit.tasks` → AIが勝手にリスト化。      | `npx cc-sdd tasks` → 設計から作業を「削り出す」。      | `speckit` のタスクは設計から浮きがち。`cc-sdd` は設計図をバラして作業にするため、一貫性が強い。        |
| **実装**    | `/speckit.implement` → **自動実行ループ。** | (エージェントに直接依頼) → **人間が伴走する実装。**           | `speckit` は「あとはよろしく」と丸投げ。`cc-sdd` は「設計通りにやれ」とステアリングし続ける。         |

### 2. なぜ speckit は「記憶喪失（アムネシア）」になるのか？

IT業界7年のあなたが直感した通り、`speckit` の「やったつもり」は以下のメカニズムで発生します。

#### **speckit のアムネシア構造：**

1. **コンテキストの外部化**: 規約を `.specify/memory/constitution.md` という「別室」に置いてしまいます。
2. **トークンの節約という罠**: 実装コマンドを打った際、AIは「目の前のタスク」と「関連コード」を優先的に読み、**分厚い憲法のファイルは「背景」として圧縮、あるいは無視**されます。
3. **伝言ゲームの崩壊**: `Specify` → `Plan` → `Tasks` と進むたびに、AIは前のフェーズの「テキスト」だけを読みます。**「なぜそう決めたか」というニュアンス（行間）は、セッションを跨ぐたびに削ぎ落とされ、最後には「命令だけが残った空疎なタスク」になります。**

#### **cc-sdd の「忘却防止」策：**

- **ステアリング（舵取り）**: 常に `requirements.md` や `design.md` を「現在の指示」の中に強制的に含めます。「図書館の憲法」ではなく、**「常に手元にあるカンニングペーパー」**として機能させるため、記憶が薄れません。
- **承認ゲート**: 人間が「OK」を出す瞬間に、文脈がリセットされるのを防ぎ、「よし、この設計を前提に進めるぞ」とAIと意識合わせを強制します。

### 3. 極端比較：PLが安心し、エンジニアが絶望する理由


| 項目         | **speckit (PLの精神安定剤)**               | **cc-sdd (エンジニアの武器)**                |
| ---------- | ------------------------------------ | ------------------------------------ |
| **PLの視点**  | 「憲法があるから、誰がやっても同じ品質になるはずだ（幻想）」       | 「各工程で私がハンコ（承認）を押すから、逃げ道がない（現実）」      |
| **実装時の挙動** | 憲法を無視して、AIが「いつもの慣れた書き方（API等）」で暴走する。  | 設計図を突きつけられ、HTMXやスナップショットの規約通りに書かされる。 |
| **エラー発生時** | AIが「プランに従って直します」と言いながら、プラン自体を誤読し始める。 | 「設計のここが矛盾している」とAIが人間に泣きつき、設計の修正に戻れる。 |
| **完成度**    | **「動くが、仕様（規約）とは別物」**ができる。            | **「仕様通りに動くもの」**ができる。                 |
