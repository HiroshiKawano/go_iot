## 農業IoTシステム 構成概要（Go + HTMX 構成）

### **[デバイス層]：ESP32 + SHT31 (高精度温湿度センサー)**

* **実装**: Arduino (C++) または Go (TinyGo) で温湿度計測ファームウェア。SHT31 から I2C で値を取得。
* **通信 (Device → Server)**: **REST API (HTTP POST/GET)**
* ※デバイスはブラウザではないため、ここだけは引き続き REST API を使用します。


* **認証**: ヘッダーに **Bearer トークン**を付与。
  サーバ側は `device_tokens` テーブルで SHA-256 ハッシュを保存し、受信したトークンを同方式でハッシュ化して突合する（Laravel Sanctum の役割を自作ミドルウェアで代替）。

---

### **[バックエンド]：Go 1.26 + Gin + templ モノリス構成**

* **役割**: データの受信・保存、SVGグラフのサーバーサイド生成、templ テンプレートへのデータ供給。
* **Web フレームワーク**: **Gin v1** (`github.com/gin-gonic/gin`、`net/http` 標準互換、軽量・高速、ルーターグループと binding タグによる入力検証内蔵)
* **テンプレート**: **templ** — `.templ` ファイルをコンパイルして型安全な Go コードを生成。HTML モックと同じ感覚で書け、IDE 補完も効く。
* **DB アクセス**: **sqlc** + **pgx/v5** — `db/queries/*.sql` から型安全なリポジトリ Go コードを生成。ORM は使わない。
* **内部アーキ**: 実務的 **Layered-lite**（厳格 Clean は不採用）。依存は下向き一方向で、DB ポートは sqlc 生成の `repository.Querier` interface（具象 `*Queries` ではなく interface 依存＝モック可）。詳細ルールの正は `.kiro/steering/structure.md`「依存方向ルール」/ `.kiro/steering/tech.md`「データアクセス方針」。
* **マイグレーション**: **goose** — `db/migrations/*.sql` を順次適用。
* **グラフ生成**: サーバーサイド SVG 生成（`html/template` で直接 `<svg>` を組み立てる方式を第一候補、将来 `go-echarts` 等の検討余地あり）。
* **ドキュメント**: **Scalar** + **OpenAPI 3.0 (yaml)** — デバイスAPI専用。`internal/docs/openapi.yaml` を `go:embed` で同梱。
* ※フロントエンド用の API 定義は不要（templ が HTML を直接返すため）、デバイスが叩くエンドポイントのみをドキュメント化します。


* **通信 (Server → Frontend)**: **templ レンダリング済み HTML（SVG 埋め込み）**
* ※API エンドポイントを作らず、Gin Handler が templ コンポーネントを直接レンダリングします。


* **認証 (Web UI)**: **scs (alexedwards/scs)** によるサーバサイドセッション
* ※Laravel Breeze 相当の機能は自作。ログイン・登録・パスワードハッシュ (bcrypt) は標準ライブラリで構築。

---

### **[フロントエンド]：templ テンプレート + HTMX + Alpine.js**

* **構成**: JavaScript ビルドツール不要。フロント担当者は HTML モック作成のみ。Alpine.js で UI の開閉・タブ切替等の軽量インタラクションを実現。
* **可視化**: **サーバーサイド SVG 生成 → HTML 埋め込み**
* ※Go サーバがレンダリングした SVG をそのまま templ 経由で埋め込みます。


* **制御**: HTMX の `hx-get` / `hx-post` / `hx-put` / `hx-delete` / `hx-patch` を使用し、ボタン操作でデバイス設定などを更新。
* **部分更新**: templ コンポーネント関数を Handler から直接 `.Render(ctx, w)` で返却する（Laravel Blade の `@fragment` + `->fragment()` と同等の挙動を、関数呼び出しだけで実現）。部分テンプレートファイルは作成せず、1つの templ ファイルで画面全体 + 部分更新の両方を定義する。

---

### **[永続化層]：PostgreSQL 16**

* **役割**: ユーザー・デバイス・センサーデータ・アラートルール/履歴を保存。
* **接続プール**: `pgxpool` (pgx/v5 付属)
* **接続設定 (`.env`)**:
  ```
  DATABASE_URL=postgres://go_iot:go_iot_dev@localhost:5432/go_iot?sslmode=disable
  ```
* **開発環境**: `docker-compose.yml` で PostgreSQL 16-alpine のみコンテナ起動、Go 本体は `air` でホスト実行。

---

### **[開発ツール]：Go 1.24+ tool ディレクティブで一元管理**

以下のツールは `go.mod` の `tool` ブロックに登録し、`go tool <name>` で呼び出す（グローバル汚染なし）。

| ツール | 役割 |
|---|---|
| `air-verse/air` | ホットリロード開発サーバ |
| `a-h/templ/cmd/templ` | `.templ` → Go コンパイル |
| `sqlc-dev/sqlc/cmd/sqlc` | SQL → Go リポジトリ生成 |
| `pressly/goose/v3/cmd/goose` | DB マイグレーション |

Makefile に集約：`make dev` / `make templ` / `make sqlc` / `make migrate-up` 等。

---

### **[本番デプロイ]：AWS Lightsail + Docker Compose**

* **Lightsail インスタンス** に Docker をインストールし、Go バイナリと PostgreSQL コンテナを `docker compose up -d` で起動。
* **リバースプロキシ**: Caddy などで HTTPS 終端（将来構築）。
* **k8s は採用しない**（IoT規模で過剰、運用負荷が Lightsail を超える）。
