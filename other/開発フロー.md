### 農業IoTシステム 段階的開発フロー（Go + Gin + templ + HTMX 構成）

> **方針:**
> - フロントエンド・バックエンド開発をメインとし、TDD で段階的に構築する
> - 物理デバイス（ESP32）との接続は後回しとし、最終フェーズで統合する
> - 開発順序は「データフローの上流から下流へ」を基本とする
> - Go + templ モノリス構成のため、各機能フェーズで Gin Handler・templ テンプレート・HTMX をまとめて実装する
> - 開発ツール（air / templ / sqlc / goose）は `go.mod` の `tool` ディレクティブでプロジェクトローカル管理

> **各 Step の記載様式:**
> - **前提**: このステップを開始する前に完了している必要があるもの
> - **成果物**: このステップが完了したときに得られるファイル・動作
> - **検証**: 完了を確認する具体的な手段 (テスト / 手動確認)

**Step 9: センサーデータ受信API** ✅ 完了

- **前提:** Step 7, 8 完了 (DB にデバイスとトークンが入っている)
- **成果物:** `POST /api/sensor-data` エンドポイント、`internal/auth/device_auth.go`, `internal/handler/sensor_api.go`
- **作業内容:**
  - **認証**: 自作 Bearer トークンミドルウェア。受信した Bearer を SHA-256 し `device_tokens.token_hash` と突合。成功時に user_id と token_id を Gin コンテキストへ格納。
  - **リクエストボディ**:
    ```json
    {
      "device_id": 1,
      "temperature": 28.50,
      "humidity": 65.30,
      "recorded_at": "2026-04-20T14:30:00+09:00"
    }
    ```
  - **バリデーション**: Gin 内蔵の `go-playground/validator` を `binding` タグで起動し範囲指定
  - **処理**: 1) 認証 2) バリデーション 3) デバイスの存在確認 + 所有者照合 4) `sensor_readings` INSERT 5) `devices.last_communicated_at` UPDATE
  - **レスポンス**: 201 Created + 保存されたデータの JSON
- **検証 (curl E2E):**
  - 認証なし → 401
  - 不正トークン → 401
  - 正常データ → 201 + sensor_readings に1件追加 + last_communicated_at 更新
  - バリデーションエラー（humidity 範囲外）→ 422
  - 存在しない device_id → 422
  - 他ユーザーのデバイスの device_id → 403

> ※ アラート判定ロジックはフェーズ7 (Step 21) で後から組み込む。この時点ではデータ保存のみ。
