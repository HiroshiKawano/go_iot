### 農業IoTシステム 段階的開発フロー（Go + Echo + templ + HTMX 構成）

> **方針:**
> - フロントエンド・バックエンド開発をメインとし、TDD で段階的に構築する
> - 物理デバイス（ESP32）との接続は後回しとし、最終フェーズで統合する
> - 開発順序は「データフローの上流から下流へ」を基本とする
> - Go + templ モノリス構成のため、各機能フェーズで Echo Handler・templ テンプレート・HTMX をまとめて実装する
> - 開発ツール（air / templ / sqlc / goose）は `go.mod` の `tool` ディレクティブでプロジェクトローカル管理

**Step 9: センサーデータ受信API** ✅ 完了

「以下の仕様で `POST /api/sensor-data` エンドポイントを TDD で実装する:

- **認証**: 自作 Bearer トークンミドルウェア (`internal/auth/device_auth.go`)。受信した Bearer を SHA-256 し `device_tokens.token_hash` と突合。成功時に user_id と token_id を Echo コンテキストへ格納。
- **リクエストボディ**:
  ```json
  {
    "device_id": 1,
    "temperature": 28.50,
    "humidity": 65.30,
    "recorded_at": "2026-04-20T14:30:00+09:00"
  }
  ```
- **バリデーション**: `go-playground/validator` の struct タグで範囲指定（temperature: -40〜125, humidity: 0〜100, device_id: min=1, recorded_at: required）
- **pgtype 変換**: `internal/infra/pgconv` で `float64 ⇔ pgtype.Numeric`, `time.Time ⇔ pgtype.Timestamptz` を隠蔽
- **処理**: 1) 認証 2) バリデーション 3) デバイスの存在確認 + 所有者照合 4) `sensor_readings` INSERT 5) `devices.last_communicated_at` UPDATE
- **レスポンス**: 201 Created + 保存されたデータの JSON

`curl` でのE2E検証:
- 認証なしのリクエスト → 401
- 不正トークン → 401
- 認証ありで正常なデータ → 201 + sensor_readings に1件追加 + last_communicated_at 更新
- バリデーションエラー（humidity 範囲外）→ 422
- 存在しない device_id → 422
- 他ユーザーのデバイスの device_id → 403

※ アラート判定ロジックはフェーズ7(Step 18)で後から組み込む。この時点ではデータ保存のみ。」

システムの最重要パス。ESP32 からのデータがここを通って保存される。
