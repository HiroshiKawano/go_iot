# Implementation Plan

- [x] 1. ナビ文脈対応サイドバーの導入（型・active・文脈リンク）
- [x] 1.1 ナビ文脈型を定義し、active を現在ページ連動にして App へ配線する
  - `component` パッケージにナビ文脈の値型を定義する: 現在ページ識別子（ダッシュボード／デバイス詳細／センサーデータ履歴／アラートルール／アラート履歴の typed 定数）と、現在ページ＋選択中デバイス ID を束ねる文脈値（デバイス ID 0 は文脈なし）。layout は component を import するため型は component 側に置く（循環回避）。
  - サイドバーを「引数なし固定 HTML」から「ナビ文脈を受け取る描画」へ変える。常時表示3項目（🏠 ダッシュボード／🔔 アラートルール／🕐 アラート履歴）の active を現在ページ連動にし、ゼロ値（対応項目なし＝登録/編集相当）ではどれも active にしない。モバイルドロワー開閉用の Alpine 属性は保持する。
  - **このサブタスクは原子的に行う（分割不可）**: サイドバーは App レイアウトが唯一の呼び出し元のため、描画契約の変更と App 側の受け渡し（共通レイアウトデータへの文脈フィールド追加＋サイドバーへ渡す配線）を同一タスクで行わないとビルドが通らない。templ 生成物を再生成する。
  - シグネチャ変更で破綻する既存のサイドバー描画テストを新契約へ更新する。
  - サイドバー描画テスト（Render→文字列検証）: ダッシュボード／アラートルール／アラート履歴の各文脈で該当項目のみ active、ゼロ値で active が0件、常時3項目のリンクが存在、ドロワー属性が存在、active 件数が1以下。
  - Observable: `go test ./internal/view/...` が green。各現在ページ値でちょうど1項目だけ active になり、ゼロ値では active マークを一切含まない HTML が描画される。
  - _Requirements: 2.1,2.4,2.5,2.6,2.7,4.1,4.2,4.4_
  - _Boundary: component.Sidebar, component.SidebarNav, layout.App_

- [x] 1.2 デバイス選択中のみ「デバイス詳細／センサーデータ履歴」の文脈リンクを描画する
  - サイドバーに、選択中デバイス ID がある場合のみ2つの文脈リンク（📟 デバイス詳細・📈 センサーデータ履歴）をダッシュボードとアラートルールの間に描画する。リンク先はその選択中デバイス（詳細＝デバイス詳細画面、履歴＝センサーデータ履歴画面）に向け、同一デバイスで揃える。
  - 文脈リンクの active は、デバイス詳細は現在ページ＝デバイス詳細のとき、センサーデータ履歴は現在ページ＝履歴のときに付ける。templ 生成物を再生成する。
  - サイドバー描画テスト: デバイス ID 指定時に両リンクがその ID 付きで存在し、現在ページ＝デバイス詳細で 📟 が active・現在ページ＝履歴で 📈 が active になる。デバイス ID なし（0）では両リンクが描画されない。
  - Observable: `go test ./internal/view/...` が green。デバイス ID ありの HTML に同一デバイスを指す文脈2リンクが含まれ、デバイス ID なしでは文脈リンクの文言を含まない。
  - _Requirements: 1.1,1.2,1.3,1.4,2.2,2.3,3.1,3.2,3.3_
  - _Boundary: component.Sidebar_
  - _Depends: 1.1_

- [x] 2. 各画面ハンドラのナビ文脈設定
- [x] 2.1 デバイス詳細・センサーデータ履歴がデバイス文脈を設定する
  - デバイス詳細画面・センサーデータ履歴画面のハンドラが、共通レイアウトデータに「現在ページ＝自画面」と「所有者認可後に確定した選択中デバイス ID」を設定する。新規のデータ取得は行わず、既に取得済みのデバイス ID を渡すだけにする（認可済みデバイス以外を指さない）。
  - ハンドラテスト（httptest＋DB ポートのモック）で、フルページ応答にその画面のナビ状態が反映されることを検証する。
  - Observable（デバイス詳細）: 詳細画面のフルページ HTML に、その要求デバイス ID を指す文脈2リンクが含まれ、デバイス詳細が active。
  - Observable（センサーデータ履歴）: 履歴画面のフルページ HTML に、その要求デバイス ID を指す文脈2リンクが含まれ、センサーデータ履歴が active。
  - _Requirements: 1.1,1.2,1.4,1.5,2.2,2.3,3.1,3.2,3.3_
  - _Boundary: handler(device_show, readings)_
  - _Depends: 1.2_

- [x] 2.2 文脈外3画面（ダッシュボード／アラートルール／アラート履歴）が active を設定する
  - ダッシュボード・アラートルール・アラート履歴のハンドラが、共通レイアウトデータに「現在ページ＝自画面」を設定する（選択中デバイス ID は設定しない）。
  - ハンドラテストで各画面のフルページ応答を検証する。
  - Observable: 3画面それぞれの応答 HTML で該当メニュー項目のみが active になり、デバイス文脈リンク（デバイス詳細／センサーデータ履歴）の文言を含まない。
  - _Requirements: 1.3,2.1,2.4,2.5_
  - _Boundary: handler(dashboard, alert_rule, alert_history)_
  - _Depends: 1.2_

- [x] 2.3 デバイス登録・編集を文脈に入れない（編集はデバイス ID を持つが設定しない）
  - デバイス登録・編集のハンドラが、共通レイアウトデータにゼロ値のナビ文脈を明示的に設定する。編集画面は URL にデバイス ID を持つが、確定済みのユーザー判断として文脈に入れない（デバイス ID を設定しない）ことをコメントで明示する。
  - ハンドラテストで、デバイス ID を持つ編集画面でも文脈リンクが出ないことを回帰防止として固定する。
  - Observable: 登録・編集のフルページ HTML がいずれの active マークも含まず、デバイス文脈リンクの文言も含まない（編集は URL にデバイス ID があっても文脈リンクなし）。
  - _Requirements: 1.3,2.6_
  - _Boundary: handler(device)_
  - _Depends: 1.2_

- [x] 3. モック正本の同期
- [x] 3.1 モック正本のサイドバーを実装状態に合わせる（更新4枚＋確認3枚）
  - 更新4枚: デバイス詳細・センサーデータ履歴のモックに文脈2リンクを追加し active を該当項目へ移す。デバイス登録・編集のモックからダッシュボード固定の active を除去する（どれも非 active）。
  - 確認3枚: ダッシュボード・アラートルール・アラート履歴のモックは既に正しい active 位置のため、対応する templ 描画と一致するか確認のみ行う（ログイン／登録の認証前モックは対象外）。
  - サイドバー用の独自 CSS クラスは新設せず既存スタイルを流用する（CSS は変更しないため CSS 同期は不要）。
  - Observable: 各モックをブラウザで開くと、active 位置と文脈リンクの有無が対応する画面の templ 描画と一致する（デバイス詳細＝📟 active＋文脈2リンク、履歴＝📈 active＋文脈2リンク、登録/編集＝active なし）。
  - _Requirements: 1.1,1.2,1.3,2.1,2.2,2.3,2.4,2.5,2.6_
  - _Boundary: mocks/html_
  - _Depends: 1.2, 2.1, 2.2, 2.3_
