# Requirements Document

## Project Description (Input)

デバイス詳細画面（GET /devices/{device}）を実装する。業務名は device-detail、REST 観点のルート/templ では device-show を用い、両者は同一画面を指す。

**誰の課題か**: 農場運営者（認証済みユーザー）。デバイスから送られる温湿度計測データを時系列で可視化し、デバイスの稼働状態を監視したい。

**現状**: バックエンド（DB・sqlc 全クエリ・デバイス Bearer 認証・POST /api/sensor-data・CLI）はほぼ完成しているが、ブラウザ向けのデバイス詳細表示画面が全面未着手。internal/service・internal/view 等は空で、HTMX 属性は 0 件。S1（基盤: セッション認証・ルーティング・エラーハンドリング）と S4（デバイス登録・編集）の後に本セッションを実行し、templ + HTMX + Alpine.js の統合を初めて具体化する。

**何を変えるか**: デバイス情報パネル、期間切替によるグラフ部分更新、サーバーサイド SVG グラフ生成、最新計測データテーブル、削除機能（確認ダイアログ → 論理削除）を実装する。

**スコープ外**: デバイス登録/編集フォーム（S4）、センサーデータ履歴全件・ページネーション（S6）、アラート判定エンジン、セッション認証ミドルウェア（S1 済み想定）、自動更新ポーリング、Tom Select 等のセレクタ。

**参照（design フェーズ）**: HTMX実装ガイド(動的).md §3.3・§4 device-show・§10-D・§11・§24、システム構成図.md「サーバサイド SVG グラフ自作」、DB設計書.md devices/sensor_readings、mocks/html/device-show.html。

## Introduction

本機能は、農場運営者が自分の所有するデバイスの状態と温湿度の推移を 1 画面で確認し、不要なデバイスを削除できるようにする「デバイス詳細画面」を提供する。画面はデバイス基本情報、期間切替で差し替わる温度・湿度グラフ（サーバー側生成）、固定 10 件の最新計測テーブル、削除導線で構成される。本要件は WHAT（利用者・運用者から観測可能な振る舞いと境界）のみを定義し、ルーティング・テンプレート構造・クエリ・SVG 生成手法などの HOW は design フェーズで確定する。

## Boundary Context

- **In scope（本機能が担う振る舞い）**:
  - デバイス詳細ページのフルページ初期表示（既定期間 24時間）
  - デバイス情報パネル（名前・MAC・場所・稼働状態・最終通信日時）の表示
  - 期間切替（24時間 / 7日間 / 30日間）によるグラフ領域のみの部分更新と選択状態の往復
  - サーバー側生成の温度グラフ・湿度グラフ（24時間=生データ線グラフ、7日/30日=日次集計）
  - 最新計測データテーブル（新しい順・最大10件・期間切替に非連動）
  - 削除（確認ダイアログ → 論理削除 → ダッシュボードへ遷移）
  - 所有者認可（自分のデバイスのみ閲覧・操作）と不正入力の拒否
- **Out of scope（本機能が担わない）**:
  - デバイス登録/編集フォーム本体（S4 が提供。本画面は「編集」への導線のみ持つ）
  - 計測履歴の全件取得・絞り込み・ページネーション（S6 が提供。本画面は「もっと見る」導線のみ持つ）
  - アラート判定・履歴記録
  - セッション認証・CSRF 保護・共通レイアウト等の基盤機構（S1 が提供。本画面はそれに依存する）
  - グラフの自動更新（ポーリング）、リッチなセレクタ UI
- **Adjacent expectations（隣接機能への前提）**:
  - 認証・ルーティング・共通エラー応答・CSRF 保護・共通レイアウトは S1 が提供済みであることを前提とする。
  - 「編集」導線の遷移先ページ（/devices/{device}/edit）は S4 が提供する。
  - 「もっと見る」導線の遷移先ページ（計測履歴一覧）は S6 が提供する（本セッション時点では未提供でもよい）。
  - 所有者認可は本画面でも貫徹するが、その判定ロジックは全画面共通の集約された認可機構を利用する想定。

## Requirements

### Requirement 1: デバイス詳細ページの初期表示
**Objective:** As a 農場運営者, I want デバイス詳細ページを開くと基本情報・グラフ・最新計測が 1 画面で確認できるようにしたい, so that デバイスの状態と環境傾向を素早く把握できる

#### Acceptance Criteria
1. When 農場運営者が自分の所有するデバイスの詳細ページを開く, the デバイス詳細画面 shall デバイス情報パネル・温度グラフ・湿度グラフ・最新計測データテーブルを 1 ページにまとめて表示する
2. When デバイス詳細ページを期間指定なしで開く, the デバイス詳細画面 shall 期間「24時間」を選択状態の既定とし、24時間分のグラフを描画する
3. The デバイス詳細画面 shall 「編集」への導線、「削除」操作、および計測履歴一覧（もっと見る）への導線を画面内に表示する
4. The デバイス詳細画面 shall ページ見出しに対象デバイス名を表示する

### Requirement 2: デバイス情報パネルの表示
**Objective:** As a 農場運営者, I want デバイスの識別情報と稼働状態を確認したい, so that どのデバイスか、正常に通信しているかを判断できる

#### Acceptance Criteria
1. When デバイス情報パネルが表示される, the デバイス詳細画面 shall デバイス名・MACアドレス・場所・稼働状態・最終通信日時を表示する
2. While デバイスが稼働中, the デバイス詳細画面 shall 稼働状態を稼働中を示す記号（●）と「稼働中」の文言で表示する
3. While デバイスが停止中, the デバイス詳細画面 shall 稼働状態を停止を示す記号（○）で稼働中と区別して表示する
4. The デバイス詳細画面 shall 最終通信日時を「YYYY-MM-DD HH:MM:SS」形式で表示する
5. If デバイスの最終通信日時が一度も記録されていない, the デバイス詳細画面 shall 未通信であることを判別可能に表示する
6. If デバイスの場所が登録されていない, the デバイス詳細画面 shall 場所欄を未設定と判別可能に表示する

### Requirement 3: 期間切替によるグラフの部分更新
**Objective:** As a 農場運営者, I want 24時間 / 7日間 / 30日間を切り替えて推移を見たい, so that 短期変動と長期傾向の両方を把握できる

#### Acceptance Criteria
1. The デバイス詳細画面 shall 期間切替として「24時間」「7日間」「30日間」の 3 つの選択肢のみを提供する
2. When 農場運営者が期間を選択する, the デバイス詳細画面 shall グラフ領域（温度グラフ・湿度グラフ・期間セレクタ）のみを選択期間の内容へ差し替え、ページ全体を再読み込みしない
3. When 期間が選択される, the デバイス詳細画面 shall 選択中の期間を選択状態（アクティブ）として、他の期間を非選択状態として表示する
4. When 期間が更新される, the デバイス詳細画面 shall デバイス情報パネルと最新計測データテーブルを変更しない
5. Where 期間が選択された, the デバイス詳細画面 should 選択中の期間をページのアドレスに反映し、共有・ブックマークおよびブラウザの戻る操作で同じ期間表示へ復帰できるようにする

### Requirement 4: 温度・湿度グラフの内容（サーバー側生成）
**Objective:** As a 農場運営者, I want 温度と湿度の推移をグラフで見たい, so that 環境の変化を視覚的に把握できる

#### Acceptance Criteria
1. The デバイス詳細画面 shall 温度グラフと湿度グラフをサーバー側で生成したベクター画像（SVG）として表示する
2. While 期間が「24時間」, the デバイス詳細画面 shall 24時間分の計測値（生データ）を時系列の線グラフで描画する
3. While 期間が「7日間」または「30日間」, the デバイス詳細画面 shall 日ごとに集計した値（日次の最大値・最小値）を描画する
4. The デバイス詳細画面 shall 各グラフに軸ラベル（横軸=時間または日付、縦軸=温度℃ または 湿度%）を表示する
5. If 選択した期間に計測データが 1 件も存在しない, the デバイス詳細画面 shall 「データはまだありません」を示す空グラフを表示する

### Requirement 5: 最新計測データテーブル
**Objective:** As a 農場運営者, I want 直近の計測値を数値で確認したい, so that 現在の温湿度を正確に把握できる

#### Acceptance Criteria
1. The デバイス詳細画面 shall 当該デバイスの計測データを新しい順に最大 10 件、テーブルで表示する
2. When 最新計測データテーブルが表示される, the デバイス詳細画面 shall 各行に計測日時・温度・湿度を表示する
3. The デバイス詳細画面 shall 計測日時を「YYYY-MM-DD HH:MM」形式、温度を小数 2 桁の℃、湿度を小数 2 桁の%で表示する
4. While 農場運営者が期間を切り替えている間, the 最新計測データテーブル shall その操作に連動せず常に最新 10 件を表示し続ける
5. If デバイスに計測データが 1 件も存在しない, the デバイス詳細画面 shall 「計測データはまだありません。」というメッセージを表示する

### Requirement 6: デバイスの削除
**Objective:** As a 農場運営者, I want 不要になったデバイスを削除したい, so that デバイス一覧を整理できる

#### Acceptance Criteria
1. When 農場運営者が「削除」を押す, the デバイス詳細画面 shall 削除確認ダイアログを表示する
2. When 削除確認ダイアログが表示される, the デバイス詳細画面 shall 対象デバイス名と「※この操作は取り消せません。計測データも削除されます。」という注意書きを表示する
3. When 農場運営者が確認ダイアログで削除を確定する, the システム shall デバイスを論理削除し、ダッシュボード（デバイス一覧）へ遷移させる
4. When 農場運営者が確認ダイアログでキャンセルする, the デバイス詳細画面 shall ダイアログを閉じ、デバイスを削除しない
5. When デバイスが削除される, the システム shall 当該デバイスを以後の一覧・詳細表示から除外する（計測データを含む物理データは論理削除として保持する）
6. If 削除操作が正当なページ経由でない（クロスサイトリクエストフォージェリ対策に反する）, the システム shall その削除を拒否する

### Requirement 7: 所有者認可とアクセス制御（BOLA 防止）
**Objective:** As a システム運用者, I want 各ユーザーが自分のデバイスのみ閲覧・操作できるようにしたい, so that 他ユーザーのデータ漏洩・改ざんを防げる

#### Acceptance Criteria
1. If 認証されていない利用者がデバイス詳細ページ・グラフ更新・削除のいずれかを要求する, the システム shall 処理を行わずログインページへ誘導する
2. If 認証済みユーザーが自分の所有しないデバイス、または存在しないデバイスを要求する, the システム shall 「見つかりません」（404）として応答し、デバイスの存在有無を区別できる情報を返さない
3. If 認証済みユーザーが自分の所有しないデバイスの削除または期間切替を要求する, the システム shall 操作を拒否し、当該デバイスの情報を一切返さない

### Requirement 8: 入力バリデーションとエラーハンドリング
**Objective:** As a システム運用者, I want 不正な入力を安全に弾きたい, so that 予期しない動作や内部情報の露出を防げる

#### Acceptance Criteria
1. If デバイス識別子が数値として解釈できない形式で要求される, the システム shall 不正な要求として 400 で拒否する
2. If 期間として「24時間 / 7日間 / 30日間」以外の値が要求される, the システム shall 不正な要求として拒否する（400 または 422）
3. If デバイス情報または計測データの取得中に予期しない障害が発生する, the システム shall 利用者に内部実装の詳細を露出しないエラー応答を返す
4. The システム shall 本画面のすべての表示・メッセージ・エラー文言を日本語で提供する
