# Requirements Document

## Introduction

本機能は、農業IoTシステムの**永続化層を、独立したサーバを必要とするデータベースから、ローカルのファイルベース・データベース（SQLite）へ全面移行する基盤差し替え**である。これは、サーバ不要で圃場のノートPC + ESP32 だけで完結するデスクトップアプリへ進化させる第一歩（後続 S10 desktop-exe-packaging の前提）に位置づけられる。

本セッションのゴールは UI 機能の追加ではなく、**既存の全機能（認証・ダッシュボード・デバイス管理・センサーデータ履歴・アラートルール・アラート履歴・デバイス API）を新しい永続化基盤の上で移行前と等価に動作させ、全テストが green になること**である。データベースファイルの場所指定とマイグレーション適用は開発者の手動操作で足り、起動時自動マイグレーション・単一実行ファイル化・配布は本スコープに含めない（S10 へ委譲）。

技術選定として永続化先は SQLite（ローカルファイル・サーバプロセス不要）を採用する（オーナー決定済み）。数値は小数第2位精度を維持し、日次集計は JST（日本標準時）の日境界でバケットする（いずれもオーナー決定済み 2026-06-08）。具体的なドライバ・型マッピング・接続設定などの実現方法は design フェーズで確定する。

## Boundary Context

- **In scope（本機能が担う）**:
  - 永続化先のローカルファイル・データベースへの差し替え。
  - 既存の全 Web UI 画面・デバイス API の振る舞い等価性（操作結果・表示内容・エラー応答が移行前と同一）。
  - 数値精度・日時表示・集計値の忠実性（silent な平坦化の防止を含む）。
  - セッション認証の永続性、テナント分離・所有者認可・CSRF・Bearer 認証の維持。
  - 現場運用 CLI（ESP32 用トークン発行）・開発用シード投入 CLI の新ストア上での動作。
  - ビルド成功・全テスト green・カバレッジ維持・スキーマ現状を参照する手段の確保（品質ゲート）。
- **Out of scope（本機能では担わない・S10 へ委譲）**:
  - 単一 Windows 実行ファイル化に関する全て（起動時自動マイグレーション・マイグレーション同梱・必須環境変数の緩和とローカルDBパス既定化・ブラウザ自動オープン・Windows ビルド・docker-compose 定義の削除）。
  - 新しい画面・機能の追加。
  - ESP32 ファームウェア側の変更（送信先 IP 設定等の運用設計）。
- **Adjacent expectations（隣接領域への期待）**:
  - 後続の S10 desktop-exe-packaging は本セッションの完了に厳密に依存する（本セッション未完なら着手不可）。
  - マイグレーション・シード・トークン発行などの開発/運用ツールは新しい永続化先を対象に動作すること。
  - CLAUDE.md がスキーマスナップショット参照を強制するため、新ストアに対する権威あるスキーマ現状の再生成手段を残すこと（初期は機能縮退を許容するが、最低限の再生成手段は確保する）。

## Requirements

### Requirement 1: ローカルファイルベース永続化への移行

**Objective:** As a 圃場でシステムを運用する開発者/運用者, I want アプリのデータが独立したDBサーバを必要とせずローカルのDBファイルに永続化されること, so that サーバなしでノートPCだけで完結する運用への足がかりを得られる

#### Acceptance Criteria

1. The 永続化基盤 shall アプリケーションの全データ（ユーザー・デバイス・センサー計測・アラートルール・アラート履歴・デバイストークン・セッション）を、独立したデータベースサーバプロセスを必要としないローカルのデータベースファイルに永続化する
2. While アプリケーションが稼働している間, the 永続化基盤 shall 別途のデータベースサーバ（docker-compose の PostgreSQL）を起動していなくてもデータの読み書きを行える
3. The 移行成果物 shall 開発者がデータベースファイルの場所を指定し手動でマイグレーションを適用することで動作する（起動時自動マイグレーション・配布パッケージングは本スコープ外）
4. When 同一プロセス内から複数のリクエストが並行してデータの読み書きを行った場合, the 永続化基盤 shall ロック起因の処理失敗やデータ破損を起こさずに整合性を保つ

### Requirement 2: 既存 Web UI 機能の振る舞い等価性

**Objective:** As a Web UI を利用するユーザー, I want 移行後も全画面の操作・表示が移行前と同一に振る舞うこと, so that データストアの変更を意識せず従来どおり利用できる

#### Acceptance Criteria

1. When 登録済みユーザーが正しい資格情報でログインした場合, the 認証システム shall 移行前と同様にログインを成立させダッシュボードへ遷移させる
2. When ユーザーがダッシュボードを開いた場合, the ダッシュボード shall 各デバイスの最新計測値・期間別グラフ（24h/7d/30d）・オフライン状態を移行前と同じ内容で表示する
3. When ユーザーがデバイスの登録・編集・論理削除を行った場合, the デバイス管理機能 shall 移行前と同じ結果（成功・バリデーション・MACアドレス重複検知）を返す
4. When ユーザーがセンサーデータ履歴を期間・ページング・集計付きで閲覧した場合, the センサーデータ履歴機能 shall 移行前と同じ並び順・件数・集計値・ページング挙動で結果を表示する
5. When ユーザーがアラートルールを作成・編集・削除した場合, the アラートルール管理機能 shall 移行前と同じ結果（成功・バリデーション）を返す
6. When ユーザーがアラート履歴をフィルタ・ページング付きで閲覧した場合, the アラート履歴機能 shall 移行前と同じフィルタ結果・件数・並び順で表示する
7. If 要求された対象データが存在しない場合, then the システム shall 移行前と同じ「未検出」応答（該当画面の 404 または空状態表示）を返す

### Requirement 3: デバイスデータ受信 API とアラート同期判定の等価性

**Objective:** As a ESP32 デバイスおよび現場運用者, I want デバイスからのデータ送信と受信時のアラート判定が移行後も同一に動作すること, so that 計測の蓄積と異常検知が途切れない

#### Acceptance Criteria

1. The デバイスデータ受信API shall 登録済みデバイスの有効な Bearer トークンを持つリクエストからのみ温湿度計測を受け付ける
2. When 有効なリクエストでセンサーデータが送信された場合, the デバイスデータ受信API shall 計測を保存し移行前と同じ成功応答（201 Created）を返す
3. If 無効または失効した Bearer トークンでデータ受信APIにアクセスされた場合, then the システム shall 移行前と同様に 401 Unauthorized で拒否する
4. When センサーデータが受信された場合, the アラート判定 shall 対象デバイスの有効なルールを同期的に評価し、閾値超過時に移行前と同じ条件でアラート履歴を生成する

### Requirement 4: 数値・日時データの忠実性

**Objective:** As a データを閲覧するユーザー, I want 温湿度・閾値・実測値の精度と表示される日時が移行前と同一であること, so that 数値や時刻の見え方が変わらない

#### Acceptance Criteria

1. The 永続化基盤 shall 温度・湿度・閾値・実測値を小数第2位までの精度で、移行前と同一の値として保存・取得する
2. When 数値が画面に表示される場合, the システム shall 移行前と同じ書式（小数第2位）で表示する
3. The 永続化基盤 shall 計測・記録日時を UTC で保存し、画面表示の直前に JST（日本標準時）へ変換する現行戦略を維持する
4. When 日時が画面に表示される場合, the システム shall 移行前と同一の JST 表記で表示する

### Requirement 5: 集計値の正当性（silent な平坦化の防止）

**Objective:** As a グラフ・集計を閲覧するユーザー, I want 最大/最小/平均や日次集計が正しい値を返し、誤って 0 や平坦な値にならないこと, so that 傾向把握の判断を誤らない

#### Acceptance Criteria

1. When ユーザーが期間別グラフまたは集計を表示した場合, the センサーデータ履歴機能およびダッシュボード shall MAX・MIN・AVG を移行前と同じ正しい値で算出する
2. The センサーデータ履歴機能 shall 日次集計を JST（日本標準時）の日境界でバケットし、移行前と同じ日付グルーピングで返す（二重のタイムゾーン補正をしない）
3. If 集計値が期待する数値型として取得できない場合, then the システム shall 黙って 0 等へフォールバックせず、明示的なエラーとして扱う

### Requirement 6: セッション認証の永続性

**Objective:** As a Web UI を利用するユーザー, I want ログイン状態が移行後も保持され、セッション管理が安定して動作すること, so that 再ログインを強いられたり処理が詰まったりしない

#### Acceptance Criteria

1. While ユーザーがログイン済みの間, the セッション認証 shall リクエストをまたいでログイン状態を保持する
2. When ユーザーがログアウトした場合, the セッション認証 shall セッションを無効化し、以後の保護ページアクセスをログイン画面へ誘導する
3. The セッション認証 shall 期限切れセッションの定期クリーンアップを、ロック起因の失敗を起こさずに実行する

### Requirement 7: セキュリティ不変条件の維持

**Objective:** As a マルチユーザー環境の利用者, I want テナント分離・所有者認可・CSRF・Bearer 認証が移行後も無変更で機能すること, so that 他人のデータにアクセスされたり不正操作されたりしない

#### Acceptance Criteria

1. The システム shall 各ユーザーに対し、そのユーザーが所有するデバイス・データのみを参照・操作可能にする（テナント分離）
2. If あるユーザーが他ユーザー所有のデバイスに対する操作を試みた場合, then the システム shall 移行前と同じく所有者認可で拒否する（403 等）
3. The Web UI shall 状態変更操作に対する CSRF 保護を移行前と同様に有効に保つ
4. The デバイスデータ受信API shall Bearer トークン認証を移行前と同様に必須とする

### Requirement 8: 現場運用 CLI とシード投入 CLI の動作保証

**Objective:** As a 現場でデバイスを設定する運用者, I want トークン発行CLIが新ストア上で動作し、発行したトークンが実際にデータ受信APIで使えること, so that ESP32 のセットアップが滞らない

#### Acceptance Criteria

1. When 運用者がトークン発行CLIを実行した場合, the トークン発行CLI shall 新しいデータストア上で ESP32 用 Bearer トークンを発行する
2. When 発行されたトークンでデバイスデータ受信APIにセンサーデータを送信した場合, the デバイスデータ受信API shall 201 Created を返して計測を保存する
3. The 開発用シード投入CLI shall 新しいデータストア上で開発用データの投入・初期化を行える

### Requirement 9: 移行完了の品質ゲート

**Objective:** As a 本機能を引き継ぐ開発者, I want 移行後にビルドが通り全テストが green でカバレッジが維持され、スキーマ現状を参照する手段が残ること, so that 後続セッション（S10）へ安全に進める

#### Acceptance Criteria

1. The 移行成果物 shall ローカル開発環境（macOS/Linux）でビルドが成功する
2. The 移行成果物 shall ユニットテストと統合テストが新しいデータストア上で全て成功する
3. The 移行成果物 shall 80%以上のテストカバレッジを維持する
4. The システム shall 新しいデータストアに対する権威あるスキーマスナップショットを再生成する手段を保持する（初期は機能縮退を許容するが、最低限の再生成手段は確保する）
