# Requirements Document

## Introduction

農場運営者は、デバイス単位で設定したアラートルール（指標 × 比較演算子 × 閾値。例: 温度 > 35℃、湿度 < 30%）に基づいて、温湿度の異常を遅延なく検知したい。現状、ESP8266 デバイスからセンサーデータを受信する機械間 API（Bearer 認証）は計測値を保存しているが、受信時にアラートルールを評価する処理が接続されていないため、閾値超過が検知・記録されない。

本スペックは、センサーデータ受信時に対象デバイスの有効なアラートルールを同期的に評価し、条件にマッチしたルールについて発火履歴を記録する、バックエンド単独で完結する判定機能を定義する。判定結果の画面表示は後続スペック（ダッシュボードのアラートバナー、アラート履歴画面）が担い、本スペックの対象外とする。

判定で扱うデータ項目（指標 temperature/humidity、演算子 `>` `<` `>=` `<=`、閾値、実測値、発火日時、未通知フラグ）は、いずれも既存スキーマ上のアラートルール・アラート履歴・センサー計測値に実在する。

## Boundary Context

- **In scope（本スペックの責務）**:
  - センサーデータ受信時に、対象デバイスの有効なアラートルールを同期評価する
  - 比較演算子（`>` `<` `>=` `<=`）の境界値を正確に判定する
  - 条件にマッチしたルールごとに、発火時点のルール内容スナップショットと実測値を含むアラート履歴を作成する
  - 複数同時発火・No-op（0件・非該当）の扱い
  - 評価・記録の失敗が計測値の受信成功を妨げないベストエフォート挙動
- **Out of scope（本スペックが所有しない）**:
  - アラートの画面表示全般 — ダッシュボードの未対応アラートバナー、アラート履歴一覧画面
  - HTMX 等によるリアルタイム未通知件数更新エンドポイント
  - メール等の外部通知の実送信（履歴上の未通知フラグの設定のみ行う）
  - 原子的トランザクション／ロールバック、評価処理の非同期キュー化
- **Adjacent expectations（前提とする隣接の振る舞い）**:
  - センサーデータ受信 API（デバイス向け Bearer 認証の機械間 API）が、計測値の保存と**デバイス所有者検証**を本評価の前に完了していること。本評価は所有者検証を通過した計測値に対してのみ走るため、評価側で所有者の再確認は行わない。
  - 作成されたアラート履歴の閲覧・通知反映は、後続スペック（ダッシュボード／履歴画面）が担う。

## Requirements

### Requirement 1: センサーデータ受信時の同期アラート評価

**Objective:** As a 農場運営者, I want センサーデータ受信時に設定済みアラートルールが即座に評価されること, so that 閾値超過を遅延なく検知・記録できる

#### Acceptance Criteria

1. When センサーデータ受信 API が新しい計測値（温度・湿度）を正常に保存したとき, the アラート評価サービス shall その計測値に対するアラートルールの評価を実行する。
2. When アラート評価サービスが評価を実行するとき, the アラート評価サービス shall 計測値を受信した一連の処理の中で評価を同期的に完了する（後続のバッチ処理や遅延処理に委ねない）。
3. The アラート評価サービス shall 評価対象を、受信した計測値のデバイスに紐づくルールに限定し、他デバイス・他ユーザーのルールを評価しない。

### Requirement 2: 評価対象ルールの選定と実測値の対応付け

**Objective:** As a 農場運営者, I want 有効なルールだけが対象になり、各ルールの指標に対応する実測値で判定されること, so that 無効化・削除済みのルールで誤った履歴が作られず、温度・湿度が取り違えられない

#### Acceptance Criteria

1. When 評価を実行するとき, the アラート評価サービス shall 有効（is_enabled = true）かつ未削除のルールのみを評価対象とする。
2. While あるルールが無効化されている間, the アラート評価サービス shall そのルールについてアラート履歴を作成しない。
3. When 評価対象ルールの指標が temperature のとき, the アラート評価サービス shall 受信計測値の温度を実測値として判定に用いる。
4. When 評価対象ルールの指標が humidity のとき, the アラート評価サービス shall 受信計測値の湿度を実測値として判定に用いる。

### Requirement 3: 閾値判定セマンティクス（比較演算子・境界値）

**Objective:** As a 農場運営者, I want 比較演算子の境界が正確に判定されること, so that ちょうど閾値の値での誤発火や見逃しが起きない

#### Acceptance Criteria

1. When 演算子が `>` かつ 実測値が閾値を超えるとき, the アラート評価サービス shall アラート発火と判定する（例: 閾値 35.00 のとき 35.01 は発火、35.00 は非発火）。
2. When 演算子が `<` かつ 実測値が閾値を下回るとき, the アラート評価サービス shall アラート発火と判定する（例: 閾値 30.00 のとき 29.99 は発火、30.00 は非発火）。
3. When 演算子が `>=` かつ 実測値が閾値以上のとき, the アラート評価サービス shall アラート発火と判定する（閾値ちょうどの値も発火）。
4. When 演算子が `<=` かつ 実測値が閾値以下のとき, the アラート評価サービス shall アラート発火と判定する（閾値ちょうどの値も発火）。
5. If 実測値が当該ルールの演算子条件を満たさないとき, then the アラート評価サービス shall そのルールについてアラート履歴を作成しない。

### Requirement 4: アラート履歴の記録（発火時スナップショット）

**Objective:** As a 農場運営者, I want 発火時点のルール内容と実測値が履歴に残ること, so that 後からルールを変更・削除しても過去の発火事実を正確に追える

#### Acceptance Criteria

1. When あるルールが発火と判定されたとき, the アラート評価サービス shall そのルールに対応するアラート履歴を1件作成する。
2. When アラート履歴を作成するとき, the アラート評価サービス shall 発火時点の指標・演算子・閾値を履歴に保存し、その後ルール側が変更・削除されても履歴の値が変わらないようにする。
3. When アラート履歴を作成するとき, the アラート評価サービス shall 発火時の実測値と発火日時を履歴に保存する。
4. When アラート履歴を作成するとき, the アラート評価サービス shall その履歴を未通知状態（is_notified = false）として記録する。
5. The アラート評価サービス shall アラート履歴の作成までを責務とし、メール等の外部通知の実送信は行わない。

### Requirement 5: 複数同時発火と No-op

**Objective:** As a 農場運営者, I want 複数ルールが同時に成立すれば全て履歴化され、何も成立しなければ履歴が作られないこと, so that 検知漏れと無駄な履歴の双方を避けられる

#### Acceptance Criteria

1. When 同一の計測値に対し複数のルールが発火と判定されたとき, the アラート評価サービス shall 発火した各ルールにつき1件ずつアラート履歴を作成する。
2. If デバイスに評価対象ルールが1件も存在しないとき, then the アラート評価サービス shall アラート履歴を作成しない。
3. If 評価対象ルールは存在するがいずれの演算子条件も満たさないとき, then the アラート評価サービス shall アラート履歴を作成しない。
4. Where 受信レスポンスに発火件数を含める拡張が有効な場合, the センサーデータ受信 API shall 当該受信で作成されたアラート履歴の件数をレスポンスに含める。

### Requirement 6: ベストエフォートのエラー耐性

**Objective:** As a 農場運営者, I want アラート評価・記録の失敗が計測値の受信を妨げないこと, so that 異常検知側の不具合でセンサーデータの蓄積が止まらない

#### Acceptance Criteria

1. If アラート評価または履歴作成の途中でエラーが発生したとき, then the センサーデータ受信 API shall 計測値の受信成功レスポンス（201）を返し続け、評価失敗をもって受信を失敗扱いにしない。
2. If アラート評価中にエラーが発生したとき, then the アラート評価サービス shall そのエラーを記録（ログ出力）し、後から追跡可能にする。
3. If 複数ルールの記録途中で一部が失敗したとき, then the アラート評価サービス shall 既に作成済みのアラート履歴を取り消さない（原子的ロールバックを行わないベストエフォート）。
