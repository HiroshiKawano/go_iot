# 実装計画: alert-evaluation（アラート判定ロジック・バックエンド/非UI）

> 逐次（sequential）。上から1行ずつ `/tdd`（RED→GREEN→REFACTOR）で実装する前提。各サブタスクは1 TDD サイクルで完了する単一責務。
> 前提: 判定アルゴリズム（`domain.ComparisonOperator.Evaluate`）・sqlc クエリ（`ListEnabledAlertRulesByDevice` / `CreateAlertHistory`）・型変換（`internal/infra/pgconv`）は実装済み。DB スキーマ変更・コード生成・CSS 配信タスクは無し。テストは Querier 手書きモックで DB 非依存（テストガイダンス集 §39）。

- [x] 1. アラート判定サービス（AlertEvaluator）の実装

- [x] 1.1 単一ルールの発火判定と発火履歴スナップショットの作成
  - デバイスの有効ルールを取得し（有効・未削除のみ返すクエリを使用）、ルールの指標が temperature なら温度、humidity なら湿度を実測値として対応付ける
  - 比較演算子の条件が成立したルールについて発火履歴を1件作成する。履歴には発火時点の指標・演算子・閾値を非正規化保持し、実測値・発火日時（計測時刻 RecordedAt）を保存する
  - 実測値・閾値は型変換の往復を避けて元の数値型のまま保存し、比較のときだけ float に変換する
  - 外部通知の送信や通知済みフラグの更新は行わない（履歴作成のみ。フラグは既定の未通知のまま）
  - 観測可能な完了: 「温度 > 35.00」有効ルール1件＋温度 36.0 の計測で、履歴作成が1回呼ばれ、保存パラメータの指標/演算子/閾値/実測値/発火日時/未通知（既定）が期待値どおり（DB 非依存の Querier モックで検証）
  - _Requirements: 1.1, 1.3, 2.1, 2.2, 2.3, 2.4, 4.1, 4.2, 4.3, 4.4, 4.5_
  - _Boundary: AlertEvaluator_

- [x] 1.2 比較演算子の境界値網羅と非該当時の無発火
  - 4演算子（`>` `<` `>=` `<=`）の境界をテーブル駆動で検証する（`>` は閾値ちょうど非発火・超過発火、`<` は閾値ちょうど非発火・下回り発火、`>=`/`<=` は閾値ちょうど発火）
  - 演算子条件を満たさないルールは発火履歴を作成しない
  - 観測可能な完了: 閾値 35.00 で 35.00=非発火/35.01=発火、閾値 30.00 で 30.00=非発火/29.99=発火、`>=`/`<=` でちょうどの値=発火、非該当ケースで履歴作成が呼ばれないことをテーブル駆動テストで確認
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5_
  - _Boundary: AlertEvaluator_

- [x] 1.3 複数同時発火と No-op の返却契約
  - 1つの計測値に対し複数の有効ルールが成立した場合、各ルールにつき履歴を1件ずつ作成し、作成済み履歴を順に並べたスライスを返す
  - 有効ルールが0件、または全ルールが非該当の場合は履歴を作成せず、長さ0のスライス（nil）を返す
  - 観測可能な完了: 温度>・湿度< の2ルール＋両条件マッチの計測で履歴2件・戻り長2、有効ルール0件で履歴作成が一度も呼ばれず戻り長0
  - _Requirements: 5.1, 5.2, 5.3_
  - _Boundary: AlertEvaluator_

- [x] 1.4 ベストエフォートのエラー耐性と判定フローの観測ログ
  - 履歴作成が途中で失敗したらエラーを返して中断し、既に作成済みの履歴は取り消さない（トランザクション不使用）。ルール取得失敗時もエラーを返す
  - 失敗はエラーレベルでログ記録する。判定フローの入出力（評価ルール件数・発火件数）はデバッグレベルで出力する
  - 指標・演算子が値域外のルールは安全に読み飛ばして他ルールの評価を継続する（fail-safe）
  - 観測可能な完了: 履歴作成モックがエラーを返すと EvaluateAndNotify がエラーを返し既発火分のスライスが戻る、値域外ルールを混ぜても他ルールは評価される、注入ロガーのバッファにデバッグ/エラー出力が確認できる
  - _Requirements: 6.2, 6.3_
  - _Boundary: AlertEvaluator_

- [x] 2. 受信ハンドラへの同期接続と配線（統合）

- [x] 2.1 センサーデータ受信ハンドラへの評価接続とレスポンス拡張
  - 受信ハンドラの計測値保存（および最終通信時刻更新）の後に、判定サービスを同期的に呼び出す。判定は別途定義する最小の評価インターフェース経由で委譲し、受信側の DB ポートにはアラート用メソッドを足さない（関心事の分離）
  - 判定や履歴作成が失敗しても受信成功（201）を妨げない（エラーは握り潰す）。レスポンスに当該受信で作成された発火件数を含める
  - 既存の受信用 DB モックは無改変のまま、評価インターフェースのスパイ実装を注入して検証する。既存の受信・認可・保存系テスト（400/401/403/422/500・所有者検証）が引き続き緑であることを確認する
  - 観測可能な完了: 所有者一致の計測送信でスパイの評価呼び出しが1回発生し 201＋発火件数フィールドが戻り長と一致、スパイがエラーを返しても 201 を維持、既存ハンドラテストが全て緑（httptest+gin で検証）
  - _Requirements: 1.1, 1.2, 5.4, 6.1_
  - _Boundary: SensorAPI_
  - _Depends: 1.1_

- [x] 2.2 合成ルートでの判定サービス配線と疎通確認
  - 合成ルート（サーバ起動の組立箇所）で、受信ハンドラに実際の判定サービス（DB ポート実体を注入したもの）を配線する
  - 観測可能な完了: `go build ./...` が通り、開発DBへシードデータ投入後に温度 36℃・湿度 25% を送信すると `alert_histories` に新規行が作成されることを手動疎通で確認できる（受信→保存→判定→履歴化が実体で繋がる）
  - _Requirements: 1.1_
  - _Boundary: 合成ルート（cmd/server）_
  - _Depends: 1.4, 2.1_
