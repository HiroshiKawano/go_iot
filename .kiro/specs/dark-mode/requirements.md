# Requirements Document

## Introduction

本スペックは、現在ライト（白背景）一色である農業IoTシステムの Web UI 全体に**ダークモード（テーマ切替）**を追加する。実装計画の拡張スペック E2 であり、S（画面分割）/P（分析ロードマップ）系列とは独立の**全画面横断 UI/UX 拡張**である。新規画面はなく、対象は既存の全 9 認証後画面＋ログイン/新規登録（Guest）画面。データベーススキーマ・サーバの外形挙動（エンドポイント・レスポンス構造）は変更しない。

方式は「**既定 OS 追従＋手動トグルで上書き＋ブラウザローカル永続**」の両立型。以下 3 点は 2026-07-02 にユーザー確定済みの**製品判断（変更不可の前提）**である：①テーマ選択の保存先は**ブラウザローカル（localStorage）のみ**（DB 保存しない）、②初回（未選択）時の既定は **OS 外観設定への追従**、③**Guest 画面（login/register）にもトグルを置く**。

対象範囲は、全画面の UI 部品（カード・フォーム・テーブル・バッジ・モーダル・ページネーション・通知トースト・検索可能セレクトのドロップダウン・日付ピッカー等ネイティブコントロール）と、**グラフ 12 コンテナのクローム**（軸・凡例・ツールチップ・色スケール・期間ズームスライダー）の両テーマ判読性である。**データ意味色（温度橙/湿度青/VPD緑/露点青/GDD赤/トレンド紫/暑熱ピンク）は両テーマ据置**とし、「暖色=暑/乾・寒色=湿」の物理規約の向きはダークでも不変とする。付随して、ダーク値投入で顕在化する**既存のコントラスト不具合 2 件**（エラートースト・注意バッジの文字色）を同時修正する。

現状調査は `2cc_sdd/ダークモード調査.md`（v2.1・全項目裏取り済み）が一次情報であり、requirements〜design での再調査は差分のみでよい。ダークパレットの最終値・Guest トグルの配置・ツールチップ暗地化の要否などの HOW は design フェーズで確定する。本要件は WHAT（利用者・運用者に観測可能な振る舞いと境界）に限定する。

## Boundary Context

- **In scope**:
  - 全 9 認証後画面＋ログイン/新規登録画面のダークテーマ対応と、全画面へのテーマ切替トグル設置。
  - 初回 OS 追従・手動上書き・ブラウザローカル永続・再読込時のテーマちらつき（FOUC）防止。
  - 全 UI 部品（カード・フォーム・テーブル・バッジ・モーダル・ページネーション・通知トースト・警告バナー・検索可能セレクトのドロップダウン・日付ピッカー/チェックボックス/ラジオ/スクロールバー等ネイティブ UI）の両テーマ判読性。
  - グラフ 12 コンテナ（デバイス詳細 11＋トレンド分析 1）のクローム（軸ラベル・軸線・目盛線・凡例・ツールチップ・色スケール・期間ズームスライダー）の両テーマ判読性と、テーマ切替時の操作状態（ズーム位置・凡例トグル）保持。
  - 既存コントラスト不具合 2 件（エラートースト・注意バッジの文字色）の修正。
  - モック HTML・正本 CSS への同時反映（CSS 単一ソース運用・モックが正本の規約維持）。
- **Out of scope**:
  - テーマ選択の DB/アカウント保存・ユーザー設定画面（製品判断①で確定。将来の DB 昇格は非破壊で可能）。
  - 手動選択後に「OS 追従（自動）」へ戻す第 3 状態の UI。トグルはライト/ダークの 2 状態明示選択であり、ブラウザ保存の消去で OS 追従へ戻る。
  - データ意味色（系列色・暑熱スケール・バッジ信号色）のテーマ別変更。沈む色の微調整は実機スモーク後の個別判断とし、要件では両テーマ同一を原則とする。
  - グラフの業務仕様・URL・期間切替 UX の変更（無回帰対象）。
  - 新規エンドポイント・データベーススキーマ変更・マイグレーション（一切なし。本機能が扱う永続データ項目はゼロで、既存テーブル・カラムに非接触）。
  - 外部セレクト部品 CSS の self-host 化・CSP ヘッダ導入・フロントライブラリのバージョン変更。
  - グラフ描画基盤の組込ダークテーマ/テーマ登録機構の採用（調査で不採用が確定済み。方式詳細は design で確定）。
- **Adjacent expectations**:
  - グラフの期間切替（ページ全体を再読込しない部分更新）は既存機能（E1/P 系スペック所有）をそのまま消費し、本スペックはそのフロー自体を再設計しない。ダーク表示中でも同フローが従来どおり機能することを前提とする。
  - CSS 単一ソース運用（正本=モック CSS → 同期生成で本番配信）と「新規 UI 要素・クラスはモック先行反映」の規約（structure.md §31）に従う。グラフ等の動的描画はモック反映義務の例外（器のみ反映）。
  - 「暖色=暑い/乾きすぎ・寒色=多湿/湿りすぎ」の物理規約（project_vpd_physics_convention）は、テストが仕様前提を符号化するため向きの誤りを自動テストで捕捉できない。GO 判定後に両テーマ×全画面の実機スモークで目視確認する運用上の期待がある。
  - ダークパレット最終値（AA コントラスト実測）・Guest 画面のトグル配置・ツールチップ暗地化の要否・半透明帯の不透明度調整の要否は、`2cc_sdd/ダークモード調査.md`（§5/§9）を一次情報として design フェーズおよび実機スモークで確定する。

## Requirements

### Requirement 1: テーマの初期決定と適用（OS 追従・保存値復元・FOUC なし）

**Objective:** 夜間・暗所でモニタリング画面を確認する利用者として、初回から OS の外観設定どおりのテーマで表示され、選択済みならその選択が最初の描画から反映されてほしい。まぶしさや画面のちらつきなく環境データの確認を始めるため。

#### Acceptance Criteria

1. When 利用者がテーマを一度も手動選択していない状態で画面を表示した, the Web UI shall OS の外観設定（ライト/ダーク）に従ったテーマで表示する。
2. While テーマの手動選択が保存されていない, when OS の外観設定が変更された, the Web UI shall 表示中画面のテーマ（グラフのクローム着色を含む）をページ再読込なしで追従させる。
3. When テーマ選択が保存された状態で利用者が画面を表示・再読込した, the Web UI shall 保存されたテーマを最初の描画から適用し、別テーマの画面を一瞬も表示しない（FOUC なし）。
4. If 保存されたテーマ選択値が破損している・読み出せない場合, then the Web UI shall エラーを表示せず OS 追従の既定動作へフォールバックする。
5. The Web UI shall 認証前後・全対象画面・画面遷移をとおして同一のテーマ決定規則を一貫適用する（遷移先でテーマが変わらない）。

### Requirement 2: 手動テーマ切替（トグル）と永続化

**Objective:** 利用者として、OS 設定とは無関係に自分の好みでライト/ダークを明示的に選び、その選択が再訪時も維持されてほしい。作業環境（屋外の直射日光下・夜間のハウス内）に合わせて見やすい表示を固定するため。

#### Acceptance Criteria

1. The Web UI shall 全 9 認証後画面およびログイン/新規登録画面のすべてにテーマ切替トグルを提供する。
2. When 利用者がトグルを操作した, the Web UI shall ページを再読込せずに画面全体のテーマを即時切り替える。
3. When 利用者がテーマを切り替えた, the Web UI shall その選択を同一ブラウザ内に保持し、再読込・再訪問時にも選択したテーマを維持する。
4. While OS の外観設定がダーク, when 利用者がライトを明示選択した, the Web UI shall ライトテーマ固定で表示する（OS ライト設定時のダーク明示選択も同様）。
5. The Web UI shall テーマ選択をサーバへ送信せず、端末・ブラウザ単位でのみ保持する（アカウント・他端末には引き継がれない）。
6. The テーマ切替トグル shall キーボードで操作可能であり、支援技術（スクリーンリーダー）に操作名と現在の切替状態を伝える。
7. If テーマ選択の保存領域が利用できない場合（プライベートブラウジング等）, then the Web UI shall エラーを表示せず、少なくとも表示中ページ内でのテーマ切替を有効に保つ。

### Requirement 3: 全 UI 部品の両テーマ判読性

**Objective:** 利用者として、どの画面のどの部品もライト/ダーク両テーマで判読できてほしい。テーマによって読めない・見えない箇所があると環境データの確認や設定操作が阻害されるため。

#### Acceptance Criteria

1. The Web UI shall 全 9 認証後画面＋ログイン/新規登録画面の全 UI 部品（カード・フォーム・テーブル・バッジ・モーダル・ページネーション・フラッシュ/トースト通知・警告バナー）を両テーマで判読可能に表示する。
2. The Web UI shall 両テーマにおいて本文テキストのコントラストを WCAG AA（4.5:1）目安で確保する。
3. While ダークテーマ適用中, the Web UI shall ネイティブ UI コントロール（日付ピッカー・チェックボックス・ラジオボタン・数値入力・セレクト・スクロールバー）をダーク系の外観で表示する。
4. While ダークテーマ適用中, when 利用者が検索可能セレクトのドロップダウンを開いた, the Web UI shall ドロップダウン本体・候補行・選択中/フォーカス行をダーク配色で表示する（ライト用の白いドロップダウンを残さない）。
5. While ダークテーマ適用中, the Web UI shall 固定色部品（警告バナー・グラフ読込中プレースホルダ・グラフ十字線・セレクト矢印アイコン・カードの立体表現）も背景に対して判読・視認可能な表現で表示する。

### Requirement 4: 既存コントラスト不具合の修正

**Objective:** 利用者として、エラー通知や注意バッジの文字がどちらのテーマでも確実に読めてほしい。エラー・注意は見落とすと影響が大きい情報であり、テーマ導入によって読めなくなることは許容できないため。

#### Acceptance Criteria

1. When エラートースト（通信エラー等のグローバルエラー通知）が表示された, the Web UI shall 両テーマにおいて赤系の地の上に判読可能なコントラストの文字で表示する。
2. The Web UI shall 注意（caution）バッジの文字を、両テーマにおいて黄系の地の上に判読可能なコントラストで表示する。
3. The Web UI shall 彩色地の部品（主要ボタン・バッジ・ページネーションの選択中表示等）の上の文字を両テーマで判読可能に保つ。

### Requirement 5: グラフの両テーマ判読性と表示状態の保持

**Objective:** 環境データを解析する利用者として、ダークテーマでもグラフの軸・凡例・スケールが読め、テーマを切り替えても操作中の表示状態が失われないでほしい。長期間データの拡大表示や系列の取捨選択をやり直さずに解析を続けるため。

#### Acceptance Criteria

1. The グラフ表示 shall 全 12 グラフ（デバイス詳細 11＋トレンド分析 1）の軸ラベル・軸線・目盛線・凡例・ツールチップ・色スケール（visualMap）・期間ズームスライダー（dataZoom）を両テーマで判読可能に表示する。
2. When 利用者がテーマを切り替えた, the グラフ表示 shall 表示中の全グラフのクローム配色を即時追従させ、かつズーム位置・凡例のトグル状態を保持する。
3. While ダークテーマ適用中, when 利用者が期間切替（ページ全体を再読込しない部分更新）でグラフを再生成した, the グラフ表示 shall 新しいグラフを現在のテーマで正しく着色して表示する。
4. The グラフ表示 shall 半透明の帯表示（VPD 3 帯・結露帯・欠測帯・有意区間帯・正常帯）を両テーマで視認可能に保つ。

### Requirement 6: データ意味色の両テーマ一貫性

**Objective:** 沖縄の環境データを日常的に読む利用者として、色が持つ意味（どの色がどの指標か・暖色=暑/乾の向き）がテーマによって変わらないでほしい。テーマを切り替えるたびに色の読み方を学び直すことや、指標の誤読を防ぐため。

#### Acceptance Criteria

1. The Web UI shall データ意味色（温度=橙・湿度=青・VPD=緑・露点=青・GDD=赤・トレンド=紫・暑熱=ピンク系の系列色、暑熱スケールのグラデーション、バッジの信号色）を両テーマで同一に保つ。
2. The Web UI shall 「暖色=暑い/乾きすぎ・寒色=多湿/湿りすぎ」の意味の向きを両テーマで不変に保つ。

### Requirement 7: ライトテーマおよび既存機能の無回帰

**Objective:** 既存の利用者として、ダークモード導入後もライトテーマの見た目と全画面の操作性が従来どおりであってほしい。慣れた画面が意図せず変わらないため。

#### Acceptance Criteria

1. The Web UI shall ライトテーマの見た目を従来と同等に保つ（利用者が視覚差を認識しない）。
2. The Web UI shall 既存の画面機能（期間切替・フォーム送信・バリデーションエラー表示・モーダル・検索/フィルタ・ページネーション・グラフ操作）の挙動をテーマ機能追加の前後で変えない。

### Requirement 8: モック資産と配信スタイルの整合（開発運用）

**Objective:** 開発運用者として、モック HTML のブラウザ直開きで両テーマとトグル要素を確認でき、本番配信スタイルと乖離しないでほしい。モックを正本とするデザイン保守・デグレ防止の運用を維持するため。

#### Acceptance Criteria

1. The モック HTML（全画面・特にログイン/新規登録） shall ブラウザ直開きで両テーマの表示とテーマ切替要素を確認できる状態に保守される。
2. The 本番配信スタイル shall 同期手順の実行後に正本（モック側スタイル）と一致する。
3. Where 本機能で新しい UI 要素・スタイルが追加される, the モック資産 shall 同じ要素・スタイルを反映して保守される（動的なグラフ描画は器のみ反映の例外）。
