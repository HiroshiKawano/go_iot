# Requirements Document

## Project Description (Input)

アラートルール管理（インラインCRUD）。GET /alerts/rules 画面で、デバイスごとのアラートルール（指標 × 比較演算子 × 閾値）を追加・編集・削除・有効/無効切替できるインライン CRUD 画面を実装する。

- **誰が困っているか**: 農場運営者。デバイスごとにアラートルールを設定して、センサー異常値を自動検知できるようにしたい。
- **現状**: バックエンド（`alert_rules` テーブル、`GetAlertRule`/`ListAlertRulesByDevice`/`CreateAlertRule`/`UpdateAlertRule`/`ToggleAlertRule`/`SoftDeleteAlertRule` 等のクエリ、`domain.Metric`/`domain.ComparisonOperator` の `Label()`/`Unit()`）は実装済み。基盤（S1: Session 認証ガード・共通レイアウト・MethodOverride・CSRF・所有者認可 `internal/authz`）も実装済み。一方でアラートルール管理の Web UI 画面が存在しない。
- **何を変えるか**: GET /alerts/rules?device_id={N} 画面で、デバイス選択・追加フォーム・ルール一覧を表示し、追加・編集・更新・削除・有効/無効切替・デバイス切替を、画面全体を再読み込みしないインライン部分更新で実現する。これはテーブル内インライン CRUD パターンの最初の Web UI であり、後続セッションへ展開するプロトタイプとなる。

> 元プロンプト: `2cc_sdd/spec-init-prompts/session-07-alert-rules.md`

## Boundary Context

- **In scope（本機能が所有する振る舞い）**
  - 認証済みユーザーによる `/alerts/rules?device_id={N}` の初期表示（デバイス選択欄＋空の追加フォーム＋当該デバイスのルール一覧）。
  - デバイス選択の切替による、選択デバイスのルール管理表示への部分更新。
  - ルールの新規追加（POST `/alerts/rules`）と、成功時の一覧反映＋追加フォームの空状態への復帰。
  - 編集フォームの読込（GET `/alerts/rules/{rule}/edit`、既存値復元・送信先を更新へ切替）。
  - ルールの更新（PUT `/alerts/rules/{rule}`）と、成功時の一覧反映＋追加フォームの空状態への復帰。
  - ルールの有効/無効切替（PATCH `/alerts/rules/{rule}/toggle`）と、当該行のみの更新。
  - ルールの論理削除（DELETE `/alerts/rules/{rule}`、確認後）と、一覧からの除去。
  - 追加・更新時のバリデーション、エラー時のフォーム再描画・入力値復元・項目別エラー表示。
  - ルール一覧の表示（有効状態・日本語ラベル・閾値＋単位・空状態）。
  - 認証必須・本人所有デバイスのルールのみ操作可とするアクセス制御の適用。
- **Out of scope（本機能が所有しない振る舞い）**
  - アラート判定の実行（センサー受信時に閾値超過を判定し履歴化・通知する処理）。別セッション（alert-evaluation）が所有する。
  - アラート履歴の表示画面（`/alerts/history`）。別セッションが所有する。
  - デバイスの登録・編集・詳細・削除（device-create-edit / device-detail）、ダッシュボード（dashboard）。
  - Session 認証ガード・本人ユーザーID解決・共通レイアウト（ヘッダー／サイドバー）・HTML フォームのメソッドオーバーライド（POST→PUT/PATCH/DELETE）・CSRF 保護の仕組み本体。web-foundation-auth（S1）が所有し、本機能は前提として消費する。
  - 所有者認可（BOLA 防止）ロジック本体。`internal/authz` が所有し、本機能は適用するのみ。
- **Adjacent expectations（隣接仕様への期待）**
  - 未認証アクセスの `/login` リダイレクト・本人ユーザーID解決・共通レイアウト描画は web-foundation-auth（S1）が所有する。本機能はこれらを前提に描画し、再実装しない。
  - HTML フォームから送信される更新系リクエスト（更新＝PUT／有効切替＝PATCH／削除＝DELETE）を当該メソッドとして扱うメソッドオーバーライドは S1 が所有する。本機能は当該規約に従って送信する。
  - 本人が所有しないデバイス（およびそのルール）へのアクセスを拒否する所有者認可（BOLA 防止）は基盤の所有者認可機能が所有する。本機能はこれを適用してアクセス制御の振る舞いを実現する。
  - 全ミューテーションに付与する CSRF 保護は S1 が所有する。本機能は当該規約に従う。
- **データ対応（UI 概念 → 現状スキーマ。`docs/database_snapshot/` を正とする）**
  - 「指標」= `alert_rules.metric`（varchar(20)・必須・CHECK 許容値 `temperature` / `humidity`）。表示は domain.Metric の日本語ラベル（温度／湿度）。
  - 「条件」= `alert_rules.operator`（varchar(5)・必須・CHECK 許容値 `>` / `<` / `>=` / `<=`、DB には記号で格納）。表示は日本語ラベル（より大きい／より小さい／以上／以下）。
  - 「閾値」= `alert_rules.threshold`（numeric(5,2)・必須）。小数第2位精度で保存。表示は小数第2位＋指標の単位（℃／%）。
  - 「有効」= `alert_rules.is_enabled`（boolean・必須・既定 true）。新規追加は有効状態で作成。
  - 「対象デバイス」= `alert_rules.device_id`（→ `devices`）。デバイス選択欄の選択肢はログイン中ユーザー所有（`devices.deleted_at IS NULL`）に限る。
  - 論理削除済み（`alert_rules.deleted_at IS NOT NULL`）のルールは一覧・取得・更新・切替・削除の対象外。

## Requirements

### Requirement 1: アラートルール管理画面の初期表示
**Objective:** 農場運営者 として、選択したデバイスのアラートルール一覧と追加フォームを 1 画面で表示したい。そうすれば、現在の異常値検知設定を確認しながら管理を始められる。

#### Acceptance Criteria
1. When 認証済みユーザーが `/alerts/rules?device_id={N}` を GET し `{N}` が本人所有デバイスを指すとき、the アラートルール管理画面 shall ステータス 200 で、共通レイアウト内にデバイス選択欄・空のルール追加フォーム・当該デバイスのルール一覧を含むフルページ HTML を返す。
2. The アラートルール管理画面 shall デバイス選択欄の選択肢を、ログイン中ユーザーが所有する（論理削除されていない）デバイスのみとし、`{N}` のデバイスを選択状態で表示する。
3. The アラートルール管理画面 shall ルール追加フォームに、指標（温度／湿度）・条件（より大きい／より小さい／以上／以下）・閾値の入力欄と、対象デバイスを示す情報を配置し、送信先を `/alerts/rules`（新規追加）として構成する。
4. When `device_id` が指定されていないとき、the アラートルール管理画面 shall ログイン中ユーザーの最初の所有デバイスを選択中デバイスとして扱い、そのルール一覧を表示する。
5. If `device_id` で指定されたデバイスが存在しない、または論理削除済みのとき、then the アラートルール管理画面 shall 当該デバイスのルールを表示せず、ステータス 404 を返す。
6. While ログイン中ユーザーが所有デバイスを 1 件も持たないとき、the アラートルール管理画面 shall ルール追加・一覧の代わりに、ルールを設定できるデバイスがない旨の案内を表示する。

### Requirement 2: デバイス切替によるルール管理対象の切替
**Objective:** 農場運営者 として、画面上でデバイスを切り替えたい。そうすれば、別のデバイスのルール管理へ移動できる。

#### Acceptance Criteria
1. When ユーザーがデバイス選択欄で別の本人所有デバイスを選択したとき、the アラートルール管理機能 shall 画面全体を再読み込みせずに、選択したデバイスの空のルール追加フォームとルール一覧へ該当領域を更新する。
2. When デバイスを切り替えたとき、the アラートルール管理機能 shall 追加フォームの対象デバイスを、切り替えた後のデバイスに整合させる。
3. If ユーザーが本人所有でないデバイス、または存在しないデバイスのルール管理表示を要求したとき、then the アラートルール管理機能 shall 当該デバイスのルールを表示せず、アクセスを拒否する（本人非所有は 403／存在しないは 404）。

### Requirement 3: アラートルールの新規追加
**Objective:** 農場運営者 として、デバイスに新しいアラートルールを追加したい。そうすれば、指定した指標・条件・閾値での異常検知を有効化できる。

#### Acceptance Criteria
1. When 認証済みユーザーが本人所有デバイスに対し妥当な入力でルール追加（POST `/alerts/rules`）を送信したとき、the アラートルール追加機能 shall 当該デバイスに指標・条件・閾値を持つルールを有効状態で新規作成する。
2. When ルール追加が成功したとき、the アラートルール追加機能 shall 画面全体を再読み込みせずに、追加したルールを含む最新のルール一覧と、空に戻したルール追加フォームへ該当領域を更新する。
3. If ルール追加の入力にバリデーションエラーがあるとき、then the アラートルール追加機能 shall ルールを作成せず、入力値を復元したルール追加フォームを項目別エラー付きでステータス 422 で再描画する。
4. If 追加対象デバイスが本人所有でない、または存在しないとき、then the アラートルール追加機能 shall ルールを作成せず、アクセスを拒否する（本人非所有は 403／存在しないは 404）。
5. If ルール作成処理が内部エラーで失敗したとき、then the システム shall 機密情報を含まないステータス 500 のエラー応答を返す。

### Requirement 4: アラートルール編集フォームの読込
**Objective:** 農場運営者 として、既存ルールを編集フォームに呼び出したい。そうすれば、現在値を確認しながら修正できる。

#### Acceptance Criteria
1. When 認証済みユーザーが本人所有デバイスのルールの編集（GET `/alerts/rules/{rule}/edit`）を要求したとき、the アラートルール管理機能 shall 当該ルールの既存値（指標・条件・閾値）を埋めた編集フォームへ、画面全体を再読み込みせずに追加フォーム領域を差し替える。
2. The 編集フォーム shall 送信時に当該ルールの更新（PUT `/alerts/rules/{rule}`）となるよう構成し、新規追加モードと区別できる更新用の操作ラベル（例:「更新」）を表示する。
3. If 指定ルールが存在しない、または論理削除済みのとき、then the アラートルール管理機能 shall ルールの値を表示せず、ステータス 404 を返す。
4. If 指定ルールが本人所有でないデバイスに属するとき、then the アラートルール管理機能 shall 当該ルールの値を表示せず、アクセスを拒否する（403）。

### Requirement 5: アラートルールの更新
**Objective:** 農場運営者 として、既存ルールの指標・条件・閾値を更新したい。そうすれば、栽培状況の変化に合わせて検知設定を調整できる。

#### Acceptance Criteria
1. When 認証済みユーザーが妥当な入力で本人所有デバイスのルール更新（PUT `/alerts/rules/{rule}`）を送信したとき、the アラートルール更新機能 shall 当該ルールの指標・条件・閾値を更新する。
2. When ルール更新が成功したとき、the アラートルール更新機能 shall 画面全体を再読み込みせずに、更新を反映した最新のルール一覧と、空に戻したルール追加フォームへ該当領域を更新する。
3. If ルール更新の入力にバリデーションエラーがあるとき、then the アラートルール更新機能 shall ルールを更新せず、入力値を復元した編集フォームを項目別エラー付きでステータス 422 で再描画する。
4. If 指定ルールが存在しない、または論理削除済みのとき、then the アラートルール更新機能 shall ルールを更新せず、ステータス 404 を返す。
5. If 指定ルールが本人所有でないデバイスに属するとき、then the アラートルール更新機能 shall ルールを更新せず、アクセスを拒否する（403）。
6. If ルール更新処理が内部エラーで失敗したとき、then the システム shall 機密情報を含まないステータス 500 のエラー応答を返す。

### Requirement 6: アラートルールの有効/無効切替
**Objective:** 農場運営者 として、ルールを削除せずに一時的に有効/無効を切り替えたい。そうすれば、設定を残したまま検知のオン/オフを調整できる。

#### Acceptance Criteria
1. When ユーザーが一覧で本人所有デバイスのルールの有効状態を切り替えたとき（PATCH `/alerts/rules/{rule}/toggle`）、the アラートルール切替機能 shall 当該ルールの有効/無効状態を反転して保存する。
2. When 有効状態の切替が成功したとき、the アラートルール切替機能 shall 画面全体を再読み込みせずに、当該ルールの行のみを反転後の状態で更新し、他の行の表示には影響させない。
3. If 指定ルールが存在しない、または論理削除済みのとき、then the アラートルール切替機能 shall 状態を変更せず、ステータス 404 を返す。
4. If 指定ルールが本人所有でないデバイスに属するとき、then the アラートルール切替機能 shall 状態を変更せず、アクセスを拒否する（403）。

### Requirement 7: アラートルールの削除
**Objective:** 農場運営者 として、不要になったルールを削除したい。そうすれば、一覧を整理し管理対象のルールだけを残せる。

#### Acceptance Criteria
1. The アラートルール削除機能 shall 削除実行の前にユーザーへ確認を求め、同意した場合にのみ削除を実行する。
2. When ユーザーが確認に同意して本人所有デバイスのルール削除（DELETE `/alerts/rules/{rule}`）を実行したとき、the アラートルール削除機能 shall 当該ルールを論理削除する。
3. When ルール削除が成功したとき、the アラートルール削除機能 shall 画面全体を再読み込みせずに、削除したルールを除いた最新のルール一覧へ該当領域を更新する。
4. The アラートルール削除機能 shall 論理削除済みのルールを以後の一覧・取得・更新・切替の対象に含めない。
5. If 指定ルールが存在しない、または論理削除済みのとき、then the アラートルール削除機能 shall ステータス 404 を返す。
6. If 指定ルールが本人所有でないデバイスに属するとき、then the アラートルール削除機能 shall ルールを削除せず、アクセスを拒否する（403）。

### Requirement 8: 入力値のバリデーションと項目別エラー表示
**Objective:** 農場運営者 として、入力不備を項目ごとに明確なエラーで知りたい。そうすれば、修正すべき箇所をすぐ把握して再入力できる。

#### Acceptance Criteria
1. If 指標が未選択、または `temperature` / `humidity` 以外の値のとき、then the アラートルール追加・更新機能 shall ルールを保存せず、当該フォームを再描画し、指標欄に不正である旨の日本語エラーメッセージを表示する。
2. If 条件が未選択、または `>` / `<` / `>=` / `<=` 以外の値のとき、then the アラートルール追加・更新機能 shall ルールを保存せず、当該フォームを再描画し、条件欄に不正である旨の日本語エラーメッセージを表示する。
3. If 閾値が未入力、または数値として解釈できない値のとき、then the アラートルール追加・更新機能 shall ルールを保存せず、当該フォームを再描画し、閾値欄に不正である旨の日本語エラーメッセージを表示する。
4. When バリデーションエラーで再描画するとき、the アラートルール追加・更新機能 shall ユーザーが入力した全項目の値（指標・条件の選択状態、閾値の入力値）を復元して表示する。
5. When 複数項目で同時にエラーが発生したとき、the アラートルール追加・更新機能 shall 該当する各項目欄にそれぞれのエラーメッセージを同時に表示する。
6. The アラートルール追加・更新機能 shall エラーメッセージを日本語で提示する。

### Requirement 9: ルール一覧の表示仕様
**Objective:** 農場運営者 として、各ルールを人間が読みやすい形式で一覧したい。そうすれば、設定内容を直感的に把握できる。

#### Acceptance Criteria
1. The ルール一覧 shall 各ルールについて有効状態・指標・条件・閾値・操作（編集／削除）を表示する。
2. The ルール一覧 shall 指標と条件を日本語ラベル（例: 温度／より大きい）で、閾値を小数第2位かつ指標の単位（℃／%）付きで表示する。
3. While 選択中デバイスに（論理削除されていない）ルールが 1 件も存在しないとき、the ルール一覧 shall ルールが未設定である旨の空状態メッセージを表示する。
4. The ルール一覧 shall 論理削除済みのルールを表示しない。

### Requirement 10: 認証と所有者認可（BOLA 防止）
**Objective:** システム所有者 として、各ユーザーが自分のデバイスのルールのみを操作できるよう保証したい。そうすれば、他ユーザーのアラート設定の不正な閲覧・改変を防げる。

#### Acceptance Criteria
1. While 未認証であるとき、when ユーザーが `/alerts/rules` 配下（初期表示・追加・編集・更新・切替・削除）のいずれかへアクセスしたとき、the システム shall 当該操作を実行せず、ステータス 302 で `/login` へリダイレクトする（認証ガードは web-foundation-auth が所有する）。
2. When 認証済みユーザーが他ユーザー所有デバイス（およびそのルール）の表示・追加・編集・更新・有効切替・削除を試みたとき、the アラートルール管理機能 shall 当該操作を実行せず、ステータス 403 でアクセスを拒否する（所有者認可は基盤の所有者認可機能が担う）。
3. The アラートルール管理機能 shall ルールの追加・更新・有効切替・削除の各ミューテーション操作を、有効な CSRF 保護を伴うリクエストに対してのみ実行する（CSRF 機構は web-foundation-auth が所有する）。
4. The アラートルール管理機能 shall 画面表示・エラーメッセージ・確認メッセージ・操作ラベルを日本語で提示する。
