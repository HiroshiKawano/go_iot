# Requirements Document

## Introduction

S9（sqlite-migration）で永続化層を SQLite（ファイル型・組込）へ差し替えた農業 IoT アプリを、**UI（HTMX）・デバイス受信 API・データベースを 1 つに同梱した単一 Windows 実行ファイル（.exe）** として配布できるデスクトップアプリに仕立てる。目的は「サーバを使わず、畑でノートPC と ESP32 センサーだけで完結するデスクトップアプリ化」。利用者は .exe をダブルクリックするだけで、データベースファイルの自動作成・スキーマの自動適用・ブラウザでの UI 自動表示までが完了し、ESP32 は同一 LAN 上のノートPC へ Bearer 付きでセンサーデータを送れる。インターネット接続・コンテナ実行基盤・別途のデータベースサーバは一切不要とする。

本仕様は DB 層の移行（S9）の上に乗る **梱包（packaging）と起動 UX（bootstrap）の層**であり、SQL 方言の書き換え・クエリ生成のやり直し・型層の改修は扱わない（S9 完了済みを前提とする）。実装方針の権威ある根拠は `2cc_sdd/SQLite化・単一exe化_実現可能性調査.md` 観点D（判定 green）および §4.3 spec_breakdown ②。

> 注: 本機能は新しい業務画面・HTMX 部分更新の追加を含まない（packaging/bootstrap が中心）。唯一のユーザー向け画面追加は ESP32 接続先を現場で把握するための補助表示の検討余地のみで、宛先解決は mDNS（後述 Requirement 7）で行う方針。

## Boundary Context

- **In scope（本機能が担う）**:
  - 単一 .exe へのクロスビルド（cgo 不使用）と他 Windows 端末への配布性。
  - 環境変数・設定ファイルなしのゼロ設定起動と、データベースファイル/設定の保存場所の自動決定・自動作成。
  - セッション署名鍵の自動生成とローカル永続化（再起動後のセッション維持）。
  - 起動時のスキーマ自動適用（冪等）と失敗時の安全な中断。
  - 既定ブラウザの自動表示と、ポート競合時の空きポート自動採番。
  - インターネット遮断下・同一 LAN 内でのデバイスデータ受信とアラート同期判定。
  - DHCP による IP 変動に追従する mDNS による安定ホスト名公開（サーバ側応答）。
  - 同時アクセス（複数デバイス書込 × Web UI 読取 × 期限切れセッション削除）下での安定性。
  - GUI ビルドのコンソール窓非表示とログのファイル出力。
  - PostgreSQL / コンテナ実行基盤の残滓撤去と配布文書（README・設定例）の更新。
- **Out of scope（本機能は担わない）**:
  - SQLite 移行本体（SQL 方言書換・クエリ生成切替・型層改修・セッションストア差替・テスト書換）→ S9 sqlite-migration で完了済み前提。
  - 新しい業務 UI 画面・機能の追加。
  - ESP32 ファームウェア側の実装（公開ホスト名を宛先に設定する作業を含むファーム/運用設定）。
  - 自動更新（配布後アップデート）機構・コード署名・インストーラ作成。
  - 実 Windows + 実 ESP32 での通し検証（**残存リスクとして明記の上、別フェーズ**で実施。本機能の受け入れ基準はクロスビルド成功とロジック/結合テスト green までとする）。
- **Adjacent expectations（隣接領域への期待）**:
  - S9 完了状態（組込ファイル型 DB が `*sql.DB` 経由でリポジトリ/セッション管理に配線済み、セッションストアが当該 DB を使用、全テスト green、`db/migrations/*.sql` が当該 DB 方言）であること。本機能はこの前提に依存するが S9 の改修自体は所有しない。
  - 本機能が同梱するスキーマ適用ファイルは、S9 が用意した方言準拠のマイグレーション群であること。
  - ESP32 ファームウェアは、公開ホスト名（または LAN IP）と Bearer トークンを用いて受信 API へ送信するよう設定されていること（その設定作業は運用/ファーム側の責務）。

## Requirements

### Requirement 1: 単一実行ファイルのクロスビルドと配布性

**Objective:** 配布担当者（開発者）として、macOS/Linux 上で単一の Windows .exe をクロスビルドできるようにしたい。それにより、Windows 実機ビルド環境なしで圃場向けデスクトップアプリを配布できる。

#### Acceptance Criteria
1. When `make build-windows` を macOS または Linux 上で実行する, the ビルドプロセス shall 単一の実行ファイル `dist/go_iot.exe` を生成する。
2. The ビルドプロセス shall C 言語連携（cgo）を一切用いずに（`CGO_ENABLED=0` で）.exe を生成し、cgo 由来のリンクエラーを発生させない。
3. The 生成物 shall 追加の同梱ファイル群を必要としない単独の .exe である。
4. When 生成された .exe を別の Windows 端末へコピーして実行する, the デスクトップアプリ shall 追加ランタイム（別途のデータベースサーバ・コンテナ実行基盤・組込ブラウザランタイム）のインストールなしで起動する。

### Requirement 2: ゼロ設定起動とデータ保存場所

**Objective:** 圃場のノートPC 利用者として、環境変数や設定ファイルなしで .exe をダブルクリックするだけで起動できるようにしたい。それにより、サーバの知識がなくても利用を開始できる。

#### Acceptance Criteria
1. While 環境変数・設定ファイル（.env）・コンテナ実行基盤のいずれも存在しない, when 利用者が .exe を起動する, the 起動シーケンス shall 必須設定の欠如によるハードフェイルを起こさずにサーバ待受まで到達する。
2. When 初回起動時にデータベースファイルが存在しない, the デスクトップアプリ shall 既定の保存場所 `%LOCALAPPDATA%\go_iot\app.db` にデータベースファイルを自動作成する。
3. If 既定の保存先ディレクトリが存在しない, then the デスクトップアプリ shall 当該ディレクトリを自動作成する。
4. Where データベースの保存先を環境変数で指定している, the デスクトップアプリ shall 既定値より優先してその指定パスを使用する。
5. The デスクトップアプリ shall ユーザー書込が制限されうる領域（例: Program Files 配下）を既定の保存先にしない。

### Requirement 3: セッション署名鍵の自動生成と永続化

**Objective:** 圃場のノートPC 利用者として、セッション鍵を設定しなくてもログイン状態が再起動後も維持されるようにしたい。それにより、再起動のたびに再ログインせずに継続利用できる。

#### Acceptance Criteria
1. When 初回起動時にセッション署名鍵（SESSION_SECRET）が未設定である, the デスクトップアプリ shall ランダムな鍵を生成し、ローカルの保存場所に永続化する。
2. While 永続化済みのセッション鍵が存在する, when アプリを再起動する, the デスクトップアプリ shall 既存の鍵を再利用し、再起動前のログインセッションを有効なまま維持する。
3. Where セッション鍵を環境変数で明示指定している, the デスクトップアプリ shall 自動生成を行わず、その指定鍵を使用する。

> Boundary: 単一ユーザー・ローカル運用前提のため、生成鍵はローカルのファイルへ平文で保存することを許容する（保存形式の保護強度は design フェーズで確定）。

### Requirement 4: 起動時マイグレーションの自動適用（冪等）

**Objective:** 圃場のノートPC 利用者として、スキーマ適用をコマンド実行せずに済ませたい。それにより、データベース管理の知識がなくても常に最新スキーマで利用できる。

#### Acceptance Criteria
1. When 初回起動でデータベースが空である, the 起動シーケンス shall 必要な全テーブルを作成するスキーマ適用を自動実行してからサーバ待受を開始する。
2. When 起動時に未適用のスキーマ差分が存在する, the 起動シーケンス shall その差分のみを適用する。
3. When 2 回目以降の起動でデータベースが既に最新スキーマである, the 起動シーケンス shall 既存データを破壊せず、適用すべき差分がなければ何も変更しない（no-op）。
4. If スキーマの自動適用が失敗する, then the 起動シーケンス shall 利用者が原因を把握できるエラーを記録し、サーバ待受を開始せずに起動を中断する。

### Requirement 5: ブラウザ自動表示とポート自動採番

**Objective:** 圃場のノートPC 利用者として、起動後に自動でブラウザが開きログイン画面が表示されるようにしたい。それにより、URL を手入力せずに操作を始められる。

#### Acceptance Criteria
1. When サーバが待受を開始する, the デスクトップアプリ shall 既定ブラウザで自端末の UI（ログイン画面）を自動的に開く。
2. The デスクトップアプリ shall アドレスバーの見える通常のブラウザで UI を表示し、組込ブラウザランタイムを必要としない。
3. If 既定のポートが他プロセスで使用中である, then the デスクトップアプリ shall 自動的に空きポートを選択して待受を開始し、そのポートで UI をブラウザに開く。
4. When ポートを自動採番した, the デスクトップアプリ shall 実際に待受しているポートを利用者が確認できるよう記録または表示する。

### Requirement 6: オフライン・LAN 内でのデバイスデータ受信

**Objective:** 圃場の運用者として、インターネットなしで同一 LAN の ESP32 からデータを受信・保存できるようにしたい。それにより、通信圏外の畑でもモニタリングが成立する。

#### Acceptance Criteria
1. While 端末がインターネットに接続していない, when 同一 LAN 上のデバイスが有効な Bearer トークン付きで計測値を受信 API（`POST /api/sensor-data`）へ送信する, the デバイス受信 API shall 201 を返し、計測値をローカルのデータベースに保存する。
2. When 登録済みデバイスからの計測値を保存した, the デスクトップアプリ shall アラート判定を同期的に実行し、閾値超過があれば履歴化する。
3. If 不正なまたは未登録のトークンで受信 API にアクセスする, then the デバイス受信 API shall 401 を返し、計測値を保存しない。
4. The デバイス受信 API shall 受信・保存・アラート判定のいずれにおいても外部インターネットサービスへの依存を持たない。

### Requirement 7: mDNS による安定ホスト名公開

**Objective:** 圃場の運用者として、ノートPC の IP が DHCP で変わっても固定のホスト名で ESP32 が到達できるようにしたい。それにより、毎回 IP を調べてファームウェアに設定し直さずに済む。

#### Acceptance Criteria
1. While アプリがサーバ待受している, the デスクトップアプリ shall 同一 LAN 上で解決可能な安定したホスト名（例: `go-iot.local`）を mDNS で公開する。
2. When 同一 LAN 上のデバイスが当該ホスト名で名前解決を行う, the デスクトップアプリ shall 自端末の現在の LAN IP を応答し、DHCP による IP 変動後もその応答を追従させる。
3. The mDNS 公開機能 shall ローカルネットワーク内に限定され、インターネットへ情報を送出しない。
4. The mDNS 公開機能 shall cgo を用いずに（単一 .exe・追加ランタイム不要の前提を維持して）動作する。

> Boundary: ESP32 ファームウェア側で当該ホスト名を送信先として設定する作業は本機能のスコープ外（運用/ファーム側の責務）。

### Requirement 8: 並行アクセス時の安定性

**Objective:** 圃場の運用者として、複数デバイスの連続送信中に Web UI を操作しても処理が落ちないようにしたい。それにより、監視と入力を同時に安心して行える。

#### Acceptance Criteria
1. When 複数デバイス相当の連続的な計測値送信（書込）と Web UI の閲覧操作（読取）と期限切れセッションの定期削除（書込）が同時に発生する, the デスクトップアプリ shall データベースの一時的な競合（ビジー）に起因する 500 応答を発生させない。
2. If 書込時にデータベースが一時的にビジーである, then the デスクトップアプリ shall 規定の待機時間内で自動的にリトライし、処理を完了する。

### Requirement 9: コンソール窓非表示とログのファイル出力（GUI モード）

**Objective:** 圃場のノートPC 利用者として、.exe 起動時に黒いコンソール窓が表示されないようにしたい。それにより、デスクトップアプリらしい体験で利用できる。

#### Acceptance Criteria
1. When GUI ビルドの .exe を起動する, the デスクトップアプリ shall コンソール窓を表示しない。
2. While GUI ビルドで動作している, the デスクトップアプリ shall 動作ログを `%LOCALAPPDATA%\go_iot\app.log` に出力する。
3. Where ログ出力先を環境変数で指定している, the デスクトップアプリ shall 既定値より優先してその指定先へログを出力する。
4. When コンソール付きビルドで動作している, the デスクトップアプリ shall ログを標準出力に出す。

> Note: ログのファイル出力基盤と GUI/console の両ビルドターゲットは 2026-06-08 に先行実装・クロスビルド確認済み。実 Windows 上でコンソール窓が出ないこと・`app.log` へ書き出されることの実機確認は Requirement で示す残存リスク（別フェーズ検証）に含む。

### Requirement 10: PostgreSQL / コンテナ残滓の撤去と配布文書の更新

**Objective:** 開発者・配布担当者として、旧来の PostgreSQL / コンテナ前提の残滓が消えていることを保証したい。それにより、ビルド・起動・テストが単一 .exe 前提で一貫する。

#### Acceptance Criteria
1. The リポジトリ shall コンテナ構成ファイル（`docker-compose.yml`）を含まない。
2. When ビルド・起動・テストを実行する, the プロジェクト shall PostgreSQL またはコンテナ実行基盤への依存を要求しない。
3. The 配布文書（README・設定例ファイル）shall PostgreSQL 前提の記述を含まず、組込ファイル型 DB に基づくデスクトップ起動手順を記載する。

## Residual Risk（受け入れ基準に対する明示的な残存リスク）

- **実機未検証**: 実 Windows 端末 + 実 ESP32 での配布→起動→自動マイグレーション→ブラウザ自動表示→Bearer POST 到達の通し検証は本機能では実施せず、別フェーズで行う。本機能の完了判定はクロスビルド成功・ゼロ設定起動の自動検証・ロジック/結合テスト green を根拠とする。GUI モードのコンソール窓非表示と `app.log` 書出も同様に実機確認は別フェーズとする。
