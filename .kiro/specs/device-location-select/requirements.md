# Requirements Document

## Project Description (Input)

本仕様は [分析アイデアメモ.md](../../../2cc_sdd/分析アイデアメモ.md) 第1章 実装ロードマップの**フェーズ1（最初の spec）**。spec-init 本文は [phase-01-device-location-select.md](../../../2cc_sdd/spec-init-prompts/phase-01-device-location-select.md)。

> **2026-06-26 方針転換**: 当初の「市町村→地区の2段カスケード」から、**単一の検索可能な「地域」セレクトへ平坦化**した（決定経緯・確定データは [research.md](research.md)）。**理由＝沖縄の農家はほぼ60歳以上で、市町村合併前（旧町村）の呼び名で土地を認識している**（ユーザーの実地知見）。新しい合併名を先に選ばせる2段カスケードは当人の認識と逆順で、認識している名前を1つ選ぶ単一セレクトの方が当人に合い、実装も単純（カスケード/HTMX/Tom Select swap が不要）。

### (a) 誰の課題か
農場運営者（デバイス登録者・多くが60歳以上）と、蓄積データを地域横断で集計・比較したいベンダー/研究側。設置場所が自由入力で、地点を集計・比較できるキーになっていない。かつ農家は**合併前の旧町村名**で土地を認識する。

### (b) 現状
- `devices.location` は `VARCHAR(255)` nullable の自由入力1列。構造化された所在地キーは無い。
- 表示: device-show 情報パネル（DeviceInfoPanel.templ）、dashboard カード（DeviceCard.templ）。編集: 共有 DeviceForm.templ の text input。フォームはフルページ POST・`gorilla.csrf.Token` hidden・編集 `_method=put`。
- Tom Select は S1 で配線済み（CDN 2.3.1・`select.js-tom-select` を `initTomSelect()` で初回初期化）。
- sqlc: `CreateDevice`/`UpdateDevice` が `location` を受ける。

### (c) 変えること
設置場所を **沖縄県内の「地域」を選ぶ単一の検索可能セレクト**へ置き換える。
- **地域マスタ（53件）= 未合併36市町村（地域＝市町村名）＋ 旧町村17（5つの合併市町村の合併前町村）**を `internal/domain` の Go 定数として持つ（確定データは research.md）。各地域は**親市町村**（集計軸）に対応づく（合併＝旧町村→現市町村、未合併＝自身）。
- フォームは **1つの検索可能 select**（53地域・Tom Select）。農家が知っている名前（佐敷・石川・平良…／未合併は市町村名）で絞り込んで選べる。
- **同名区別**: 旧具志川市（うるま市）と旧具志川村（久米島町）等の同名を、表示「短縮名（現市町村）」で区別する。
- `devices` に構造化所在地キー `locality`（nullable VARCHAR）を追加。親市町村は `Locality.Municipality()` で導出（将来の SQL 集計で必要なら非破壊で市町村列を後付け可）。
- device-show 情報パネル・dashboard カードの所在地表示を**認識名（地域）**へ切り替え。
- 既存 `location` 値を非破壊でベストエフォート移行。
- **県は沖縄固定。字・番地・緯度経度は取らない。**
- これにより「地点（地域）」を集計・横並び比較可能なキーとして確立（設計ガードレール①）、後続フェーズ4（CSV集計）・13（圃場ベンチマーク）の前提キーを作る。

### 前提セッション
- **S1** web-foundation-auth（App レイアウト・CSRF・MethodOverride・Tom Select 初期化）
- **S4** device-create-edit（共有 DeviceForm・フルページ POST・入力値復元・バリデーション表示）

### スコープ外（別フェーズ）
- ハウス内センサ位置（内/外・東/西）= フェーズ10。圃場エンティティ・ベンチマーク・農家共有 = フェーズ13。作物マスタ = フェーズ3/6/7。
- 地域別の集計表示・グラフ・CSV 出力 = フェーズ4/13（本フェーズは集計**キー**のみ）。
- 緯度経度・字・番地 / 都道府県マスタ（沖縄固定）/ 自由記述補足欄の新設。
- 字・小字までの細粒度（本フェーズは旧町村/市町村レベルで確定。細粒度は将来）。
- sensor_readings スキーマ・受信 API・グラフ描画基盤（E1）の変更。

## Boundary Context

- **In scope（本機能が所有する振る舞い）**
  - デバイス登録・編集フォームの設置場所欄を、自由入力テキストから**単一の検索可能な「地域」select**へ置き換えること。
  - 沖縄の53地域（未合併36＋旧町村17）を、農家が認識する名前で提示・検索でき、同名を区別すること。
  - 選択した地域をデバイスに保存・更新し、編集表示・エラー再描画時に選択値を復元すること。
  - 存在しない地域を拒否すること。
  - デバイス詳細・ダッシュボードの所在地表示を、認識名（地域）へ切り替えること。
  - 既存の自由入力「設置場所」値を、新しい地域キーへ非破壊で移行すること。
- **Out of boundary（本機能が所有しない振る舞い）**
  - 2段カスケード・市町村別の連動更新（**平坦化により本フェーズでは作らない**）。
  - ハウス内位置（フェーズ10）／圃場・ベンチマーク（フェーズ13）／作物マスタ（フェーズ3/6/7）。
  - 地域別の集計表示・グラフ・CSV（フェーズ4/13）。集計の**キー**を用意するのみ。
  - 緯度経度・字・番地、都道府県の選択（沖縄固定）、自由記述補足欄。
  - 認証ガード・所有者認可・MethodOverride・CSRF の仕組み本体（基盤が所有）。
- **Adjacent expectations（隣接仕様への期待）**
  - 共通レイアウト・認証ガード・CSRF・メソッドオーバーライド・Tom Select 初期化は S1 が所有。本機能は前提として消費。
  - フォーム基本構造（フルページ送信・入力値復元・項目別エラー）と所有者認可（BOLA）の消費は S4 が所有。本機能は地域項目を追加・拡張する。
  - device-show（S5）・dashboard（S3）の画面本体は各 spec が所有。本機能は所在地の「表示」のみ差し替える。
- **データ対応（UI 概念 → 現状スキーマ。`docs/database_snapshot/` を正とする。新規列は migration 追加が前提）**
  - 「地域」= 新規マスタ `domain.Locality`（53・Go 定数）と、`devices` への `locality`（nullable VARCHAR）の追加。**現状スナップショットに未存在 → migration 追加と `make db-snapshot` 再生成を design/tasks の明示作業とする。**
  - 「親市町村（集計軸）」= `domain.Municipality`（41・Go 定数）。`Locality.Municipality()` で導出（本フェーズでは DB 列に持たない＝YAGNI。将来 SQL 集計で非破壊に追加可）。
  - 既存「設置場所（自由入力）」= `devices.location`（varchar(255)・nullable）。残置（移行元）。
  - 都道府県は沖縄県に固定し、保存も選択もしない。所在地は任意項目。

## Requirements

### Requirement 1: 地域セレクトのフォーム表示（登録・編集）
**Objective:** 農場運営者 として、設置場所を自分が知っている地域名（旧町村・市町村）で1つ選びたい。そうすれば、合併後の新しい市町村名を知らなくても迷わず入力でき、地点を比較できる形で記録できる。

#### Acceptance Criteria
1. When 認証済みユーザーがデバイス登録フォームまたは編集フォームを表示したとき、the 地域セレクト機能 shall 設置場所欄を、自由入力テキストではなく沖縄の地域を選ぶ単一の select として描画する。
2. The 地域セレクト機能 shall 沖縄の53地域（未合併36市町村＋5つの合併市町村の旧町村17）のみを選択肢として提示し、他都道府県を含めない。
3. The 地域セレクト機能 shall select の先頭に未選択を表す空の選択肢を置き、設置場所を任意項目（未選択のまま送信可能）として扱う。
4. The 地域セレクト機能 shall 地域 select を、ユーザーが文字を入力して候補を絞り込める検索可能な select として提供する。
5. The 地域セレクト機能 shall 都道府県を沖縄県に固定し、都道府県を選ぶ選択欄を設けない。

### Requirement 2: 認識名による提示と同名区別
**Objective:** 農場運営者 として、合併前の旧町村名で候補を見つけられ、同じ呼び名の別地域を取り違えたくない。そうすれば、自分の土地を正確に選べる。

#### Acceptance Criteria
1. The 地域セレクト機能 shall 合併した市町村の地域を、旧町村名を主とした表示（例「佐敷（南城市）」）で提示する。
2. The 地域セレクト機能 shall 未合併の市町村の地域を、その市町村名で提示する。
3. When 同名の旧町村が複数存在するとき（例 旧具志川市と旧具志川村）、the 地域セレクト機能 shall 各々を現市町村を併記して区別可能に提示する（例「具志川（うるま市）」「具志川（久米島町）」）。
4. The 地域セレクト機能 shall 旧町村名・現市町村名のいずれの入力でも該当地域を検索候補に表示する。

### Requirement 3: 所在地の保存（登録・更新）
**Objective:** 農場運営者 として、選んだ地域をデバイスに保存したい。そうすれば、以後その地点ごとに計測データを参照・比較できる。

#### Acceptance Criteria
1. When 認証済みユーザーが地域を選択して本人所有デバイスを登録または更新したとき、the デバイス所在地機能 shall 選択された地域を当該デバイスの所在地として保存する。
2. When 設置場所を未選択のまま他項目が妥当な状態で登録または更新したとき、the デバイス所在地機能 shall 所在地を未設定として保存し、登録・更新を成功させる。
3. The デバイス所在地機能 shall 所在地項目の追加後も、登録・編集フローの認証必須・本人所有デバイス限定・成功時のデバイス詳細画面へのリダイレクトという既存の観測可能な振る舞いを維持する。

### Requirement 4: 選択値の復元（編集表示・エラー再描画）
**Objective:** 農場運営者 として、編集時やエラー時に選んだ地域が保持されてほしい。そうすれば、選び直す手間なく作業を続けられる。

#### Acceptance Criteria
1. When 認証済みユーザーが本人所有デバイスの編集フォームを表示したとき、the 地域セレクト機能 shall 当該デバイスに保存済みの地域を選択状態にする。
2. When バリデーションエラーでフォームを再描画するとき、the 地域セレクト機能 shall ユーザーが選択していた地域を選択状態に復元する。
3. While 保存済み所在地が未設定であるとき、when 編集フォームを表示したとき、the 地域セレクト機能 shall 地域 select を未選択の状態で表示する。

### Requirement 5: 所在地入力の検証
**Objective:** システム所有者 として、不正な地域を弾きたい。そうすれば、地点ごとに集計できるきれいなデータだけを蓄積できる。

#### Acceptance Criteria
1. If 提示された選択肢に存在しない地域が送信されたとき、then the デバイス所在地機能 shall 当該デバイスを保存せず、フォームを再描画し、所在地欄に不正である旨の日本語エラーメッセージを表示する。
2. When 所在地と他項目に同時に不備があり再描画するとき、the デバイス所在地機能 shall デバイス名・MACアドレス等の既存項目と所在地の不備を同時に判定し、各不備項目の欄にそれぞれのエラーメッセージを表示する。

### Requirement 6: 構造化所在地の表示（デバイス詳細・ダッシュボード）
**Objective:** 農場運営者 として、一覧・詳細で所在地が自分の知っている地域名で表示されてほしい。そうすれば、どのデバイスがどこにあるかを一目で把握できる。

#### Acceptance Criteria
1. When 認証済みユーザーがデバイス詳細を表示したとき、the 所在地表示 shall 当該デバイスの所在地を認識名（合併地域は「旧町村（現市町村）」、未合併は市町村名）で情報パネルに表示する。
2. When 認証済みユーザーがダッシュボードのデバイス一覧を表示したとき、the 所在地表示 shall 各デバイスカードの所在地を同形式で表示する。
3. While デバイスの所在地が未設定であるとき、the 所在地表示 shall 従来同等の、未設定であることが分かる空表示にする。

### Requirement 7: 既存所在地データの非破壊移行
**Objective:** システム所有者 として、既存の自由入力の設置場所を新方式へ安全に移したい。そうすれば、これまで登録したデバイスや計測履歴を壊さずに引き継げる。

#### Acceptance Criteria
1. When 本機能を既存データへ適用するとき、the 所在地移行 shall 既存の自由入力「設置場所」値が沖縄の地域（旧町村名・現/旧市町村名）と対応付けられる場合に、当該デバイスへその地域を構造化所在地として設定する。
2. If 既存の自由入力値がいずれの地域にも対応付けられないとき、then the 所在地移行 shall 当該デバイスの構造化所在地を未設定のまま残す。
3. The 所在地移行 shall 移行処理の前後で既存デバイスおよび既存センサーデータ（計測履歴）の件数を減少させず、移行後も従来どおりの参照・編集を継続できるようにする。

### Requirement 8: 地域マスタの整備
**Objective:** システム所有者 として、沖縄の地域を共通の正規マスタとして用意したい。そうすれば、全デバイスの所在地を同じ語彙で記録・集計できる。

#### Acceptance Criteria
1. The 地域マスタ shall 沖縄の53地域（未合併36市町村＋旧町村17）を、所在地選択の正規マスタとして保持する。
2. The 地域マスタ shall 各地域を親市町村（集計軸・41市町村のいずれか）に対応づけ、合併地域は旧町村→現市町村、未合併地域は自身を親とする。
3. The 地域マスタ shall マスタを Go 定数として保持し、本番デバイス・計測データへの投入/初期化操作を伴わない（＝本質的に冪等・非破壊）。
