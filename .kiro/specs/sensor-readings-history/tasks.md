# Implementation Plan

> 逐次実装（番号昇順＝実装順）。各 executable サブタスクは `/tdd` の1サイクル（RED→GREEN→REFACTOR）で完了する単位。
> 前提（design の Out of Boundary）: CSS アセット配信（`make sync-css` / `static.go` / `CSSURL`）・ミドルウェア・認可・sqlc 3クエリは S1/既存で実装済み。マイグレーション不要。既存ヘルパ（`renderPage`/`renderComponent`/`renderError`/`formatActual`/`aggregateToFloat`/`var jst`/`renderDeviceReadError`）は同 `package handler` で再利用（新規作成しない）。

- [x] 1. Foundation: 表示用 ViewModel の型定義
- [x] 1.1 結果領域の ViewModel 型を定義する
  - `page` 側に `ReadingsView`（レイアウト・デバイスID・デバイス名・from/to の echo 値・結果領域 View を保持）を追記する
  - `component` 側に `DeviceReadingsListView`（集計・行・データ有無・ページャ・エラーマップ）、`SummaryView`（整形済み6項目）、`ReadingHistoryRow`（計測日時・温度・湿度・通信遅延の4項目。既存 `ReadingRow` とは別名）、`PaginationView`（現在/総ページ・前後有無・前後URL）を追記する
  - 観測完了: `make templ` 不要で `go build ./internal/view/...` が通り、各 struct がコンパイル可能な状態になる
  - _Requirements: 1.1, 1.5_
  - _Boundary: view ViewModel (page/views.go, component/views.go)_

- [x] 2. 純関数: ドメイン整形ロジック（DB 非依存・table-driven テスト）
- [x] 2.1 日付フィルタを BETWEEN 用の区間へ写すロジックを実装する
  - 開始日・終了日（YYYY-MM-DD・任意）を JST 暦日として解釈し、検索区間の下限・上限へ変換する
  - 未指定時は下限を遠い過去センチネル・上限を遠い未来センチネルとし、全期間検索を成立させる
  - 終了日指定時はその日を含めるため上限を当日の末尾（end-of-day）まで拡張する
  - 形式不正（`2026/04/20`・`abc` 等）は日本語エラーメッセージをエラーマップに積み、区間は使わせない
  - 観測完了: 未指定→遠過去/遠未来・終了日当日23:59の計測が含まれる区間・形式不正でエラーマップにキーが立つ、を検証する table-driven テストが緑
  - _Requirements: 2.2, 2.3, 2.4, 6.1, 6.2_
- [x] 2.2 ページ番号の正規化と総ページ数の算出を実装する
  - ページ番号を 1 以上へ正規化する（未指定・1未満・数値として解釈不可は 1 とする）
  - 総件数から総ページ数を算出する（1ページ20件。0件でも1ページ、21件で2ページ）
  - 観測完了: ""/"0"/"-1"/"abc"→1・"3"→3、総件数 0→1・20→1・21→2 を検証する table-driven テストが緑
  - _Requirements: 4.1, 4.4_
- [x] 2.3 通信遅延の整形を実装する
  - 計測時刻とサーバ受信時刻の差を四捨五入した整数秒の「N秒」表記にする
  - クロックずれによる負値は「0秒」にクランプする
  - 観測完了: 0.4秒→「0秒」・1.5秒→「2秒」・負値→「0秒」を検証するテストが緑
  - _Requirements: 5.1, 5.2_
- [x] 2.4 集計情報の表示用整形を実装する
  - 集計行を平均/最高/最低×温度/湿度の6項目（小数第2位＋単位 ℃/%）へ整形する
  - 該当データ0件（サンプル数0）のときは6項目すべてを「—」にする（0.00 と誤表示しない）
  - 既存の集計値安全変換ヘルパを用い、平均（numeric）と最高/最低（interface 型）の双方を扱う
  - 観測完了: 0件→全項目「—」、データ有→「28.30℃」「65.30%」を検証するテストが緑
  - _Requirements: 3.1, 3.2_
- [x] 2.5 ページャ用 URL 生成を実装する
  - 現在の開始日・終了日を保持したまま、指定ページ番号に差し替えた相対 URL を生成する
  - 観測完了: from/to を維持し page=N が反映された URL を返すテストが緑
  - _Requirements: 8.2_

- [x] 3. templ コンポーネント: 画面描画（Render 結果を文字列で検証）
- [x] 3.1 ページネーション部品を実装する
  - 「前へ（前ページ有時）/ 現在・総ページ表記 / 次へ（次ページ有時）」の簡易ページャを描画する
  - ナビゲーション要素に部分更新属性（結果領域をターゲットに内側差し替え・リンクの HTMX 化）を付与する
  - 既存の `.pagination` / `.btn` クラスのみ使用し独自クラスを新設しない
  - 観測完了: 前後ページ有無の切替で「前へ」「次へ」リンクの有無が変わることを Render 結果で検証、`make templ` で生成物が更新される
  - _Requirements: 4.2, 8.2_
- [x] 3.2 フィルタ結果領域フラグメントを実装する
  - 部分更新ターゲット id（`device-readings-list`）を持つラッパー内に、エラー表示→集計6項目→データ一覧テーブル（計測日時・温度・湿度・通信遅延）→ページャ、を内包する
  - 行が0件のときはテーブルを表示せず空状態メッセージ「指定期間の計測データはありません。」を表示する
  - エラーマップが非空のときはフラグメント内にインラインエラーを表示する（フルページのレイアウトバナーは部分応答に含まれないため）
  - モックを写経し独自クラスを新設しない（Alpine の表示制御ではなく条件分岐で描画）
  - 観測完了: 行ありでデータ一覧、行なしで空状態メッセージ、エラー有でエラー表示を含むことを Render 結果で検証、`make templ` で生成物が更新される
  - _Requirements: 1.1, 1.2, 1.3, 1.5, 3.1, 3.2, 5.1, 5.2, 6.2, 6.3, 7.1_
  - _Depends: 3.1_
- [x] 3.3 フルページ画面を実装する
  - 認証後レイアウトに、見出し（「センサーデータ履歴: デバイス名」）・フィルタフォーム（開始日/終了日、入力値の echo、検索ボタン）・結果領域フラグメントを内包する
  - フィルタフォームは GET 送信を部分更新へ昇華し、結果領域のみを差し替え、URL に検索条件を反映する（フォームは結果領域の外に配置して入力状態を保持する）
  - 観測完了: フルページにフィルタフォームと結果領域（`device-readings-list`）の双方を含むことを Render 結果で検証、`make templ` で生成物が更新される
  - _Requirements: 2.1, 8.2_
  - _Depends: 3.2_

- [x] 4. ハンドラ: HTTP 境界（httptest + 手書き Repo モックで DB 非依存検証）
- [x] 4.1 ハンドラと最小 DB ポートを定義し初期表示・所有者認可を実装する
  - 本ハンドラが必要とする最小メソッドのみの consumer interface を定義する（具象クエリ実装がそれを満たす）
  - デバイスID をパースし、所有者認可（不在/非所有は存在秘匿のため同一の未検出応答）を行い、ユーザー名を取得してフルページを描画する
  - 日付未指定時は全期間で最新20件＋集計を表示する
  - 観測完了: 部分更新ヘッダ無しで初期表示が成功し、集計・データ一覧・結果領域・デバイス名を含む。非数値IDは入力不正応答、不在/非所有は未検出応答、ユーザー取得のDBエラーはサーバエラー応答になる
  - _Requirements: 1.1, 1.4, 1.5, 3.3, 6.1, 8.3_
  - _Depends: 1.1, 2.1, 2.4, 3.3_
- [x] 4.2 期間フィルタ検索とページ送りを実装する
  - 部分更新リクエスト時は結果領域フラグメントのみを返し（レイアウトを含めない）、URL に検索条件・ページを反映する
  - 検索区間・ページ番号から取得位置（20件単位）を決め、総ページ数に収まるようページをクランプして一覧・集計・件数を取得する
  - 集計はフィルタ条件（期間全体）に連動し、同一条件内でのページ移動では変化しない
  - 観測完了: 部分更新ヘッダ有りでフラグメントのみ返却（レイアウト無）、終了日が末尾まで含めて検索に渡る、2ページ目で取得位置が20進む、過大なページは最終ページに収まる
  - _Requirements: 2.2, 2.3, 3.4, 3.5, 4.2, 4.3, 4.4, 8.1, 8.4_
  - _Depends: 2.2, 2.5, 3.1_
- [x] 4.3 日付形式エラー・0件・障害時のエラー写像を実装する
  - 日付形式エラー時は同一画面に留まり、インラインエラー＋空一覧で正常応答を返す（検索・集計クエリは実行しない）
  - 該当データ0件時は空状態メッセージを表示する（開始日が終了日より後で該当なしの場合も同様）
  - 件数・一覧・集計取得のDBエラーはサーバエラー応答へ写す
  - 観測完了: 不正な日付指定で正常応答かつインラインエラーを含み一覧が空・検索/集計クエリが未呼び出し、0件で空状態メッセージ、DBエラーでサーバエラー応答
  - _Requirements: 2.4, 6.2, 6.3, 7.1_

- [x] 5. Integration: ルーティング配線
- [x] 5.1 履歴画面のルートを Web グループへ配線する
  - 認証必須の GET ルート（`/devices/:device/readings`）を既存デバイス系経路の近傍に追加し、本ハンドラへ結ぶ
  - 観測完了: ルートが登録され、認証済みで初期表示が成功、未認証はログインへ誘導される。デバイス詳細画面の「もっと見る」導線が機能し、既存の `:device` 子経路と競合しない
  - _Requirements: 1.1, 8.1_
  - _Depends: 4.1_

  - 観測完了: `make templ`→`go build`→`go test ./...` が緑で、関連カバレッジ80%以上をレポートで確認できる
