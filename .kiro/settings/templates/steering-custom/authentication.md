# 認証・認可標準

[目的: 認証方式・トークン/セッションのライフサイクル・権限チェック・セキュリティの統一]

## 思想
- **認証 (who) と認可 (what) を明確に分離する**
- 既定で安全側に倒す: 最小権限、fail-closed、短命トークン
- UX を意識: リスクが高い箇所にだけ摩擦を置く

## 認証

### 方式（選択と根拠）
- 候補: JWT / セッション / OAuth2 / デバイストークン / ハイブリッド
- 採用: [方式] を採用。理由: [理由]
- IoT プロジェクト想定: ブラウザ UI はセッションまたは JWT、ESP32 などデバイスはデバイス固有トークン（`device_tokens` テーブル）

### フロー（概念）
```
1) ユーザー/デバイスが身元を提示（認証情報 または プロバイダ経由）
2) サーバーが検証しトークン/セッションを発行
3) クライアントはリクエストごとにトークンを送信
4) サーバーは検証後に処理を継続
```

### トークン/セッションのライフサイクル
- 格納: httpOnly Cookie または `Authorization` ヘッダ
- 有効期限: アクセストークンは短命、必要ならリフレッシュトークンで延長
- ローテーション: リフレッシュ時に再発行、失効リスト運用
- 失効: ログアウト/侵害疑い時に即時失効

### セキュリティパターン
- TLS 強制。JS からアクセスされる位置にトークンを置かない
- `aud` / `iss` を検証。クレームは最小限
- 機微操作はデバイス結合や IP/リスクチェックを検討

## 認可

### 権限モデル
- 候補: RBAC / ABAC / 所有権ベース / ハイブリッド
- ロール/属性は中央で定義し、コード各所にハードコードしない

### チェックの場所
- **ミドルウェア (Gin)**: 粗粒度のゲート（認証済みか、ロールを持つか）
- **サービス層**: 細粒度判断（この資源をこのユーザーが操作できるか）
- **UI/テンプレート**: 条件付き描画のみ。セキュリティを UI に依存させない

Go + Gin のパターン例:
```go
// ルートレベル: ミドルウェアで粗粒度チェック
r.GET("/api/devices", auth.RequirePermission("device:read"), handler.ListDevices)

// サービス層: 細粒度チェック
func (s *DeviceService) Update(ctx context.Context, user User, deviceID uuid.UUID, in UpdateInput) error {
    if !user.Can("device:update") {
        return ErrForbidden
    }
    // ...
}
```

### 所有権
- パターン: オーナー または 特権ロールのみ操作可
- エンティティ境界で、変更前に検証する

## パスワード & MFA
- パスワード: 強度ポリシーを適用。必ずハッシュ化（`golang.org/x/crypto/bcrypt` または `argon2`）。平文保存禁止
- リセット: 時間制限つき単発トークン。ユーザーに通知
- MFA: 機微操作ではステップアップを検討

## API 間認証 / デバイス認証
- API キーまたは OAuth クライアント認証情報
- デバイス (ESP32 等) は個別トークンを DB で発行・管理（失効・ローテ可能）
- スコープは最小限。定期ローテと監査ログ
- レート制限は識別子（ユーザー/キー/デバイス）ごと
