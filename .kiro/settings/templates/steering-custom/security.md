# セキュリティ標準

[目的: 検証・認可・シークレット・データ保護に関するセキュリティ姿勢をパターンで定義する]

## 思想
- 多層防御、最小権限、secure-by-default、fail-closed
- 境界で検証し、出力先に応じてサニタイズ。入力を信用しない
- 認証 (who) と認可 (what) を分離

## 入力 & 出力
- API/フォーム境界で検証（`go-playground/validator/v10` + 構造体タグ）
- 出力先に応じたエスケープ: templ は既定で HTML エスケープ、SQL は sqlc でパラメータ化、シェル実行は引数配列、ログは機微を除去
- 拒否リストより許可リストを優先。早期拒否、詳細は最小限

## 認証 & 認可
- 認証: 身元検証後、短命トークン/セッションを発行
- 認可: 操作前に権限チェック、既定で deny
- ポリシーは中央集約。コード各所に重複チェックを書かない

パターン例 (Go):
```go
if !user.HasPermission("resource:action") {
    c.AbortWithStatusJSON(http.StatusForbidden, gin.H{"message": "forbidden"})
    return
}
```

## シークレット & 設定
- シークレットをコミットしない。`.env` は `.gitignore`、本番はシークレットマネージャ
- 定期ローテ、アクセス監査、スコープ最小化
- 起動時に必須環境変数を検証し、欠けていたら fail fast（`log.Fatal` で終了）
- SQL 接続文字列・JWT 秘密鍵・外部 API キーは必ず環境変数経由

## 機微データ
- 収集を最小化。ログ内はマスク/編集。保存時・転送時に暗号化
- ロール/需要原則でアクセス制限。機微レコードへのアクセスを監査ログ化
- IoT センサーデータでも、個人を特定しうる位置情報などは要検討

## セッション/トークンのセキュリティ
- 可能なら httpOnly + Secure Cookie。TLS は全経路
- 有効期限は短く、リフレッシュでローテ、ログアウト/侵害疑いで失効
- JWT は `aud` / `iss` を検証、クレームは最小限
- デバイストークン（ESP8266 等）は個別発行し、DB で失効可能に

## ログ（セキュリティ観点）
- 記録: 認証試行、権限拒否、機微操作
- 非記録: パスワード、トークン、シークレット、PII 全量、機微ボディ
- `requestId` と文脈を付与し、イベントを相関付け

## ヘッダ & トランスポート
- TLS 強制、HSTS
- セキュリティヘッダを設定（CSP、X-Frame-Options、X-Content-Type-Options）
  - Gin なら `github.com/gin-contrib/secure` の `secure.New(...)` を入口に
- 最新の暗号を優先、弱い TLS/暗号は無効化

## 脆弱性対応
- 信頼できるライブラリを使い、依存を定期更新（`go list -m -u all` / `govulncheck ./...`）
- 静的/動的スキャンを CI に組込み、検出を追跡・修正
- `govulncheck`、`gosec` などのツールを活用
- よくある脆弱性クラスをチームで共有し、上記パターンに昇華する

---
_パターンと原則に集中。具体的な設定は運用ドキュメントに分離_
