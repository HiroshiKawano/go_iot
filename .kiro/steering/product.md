# プロダクト概要（product.md）

> 何を・誰のために作るか、その価値と判断基準。各 spec の requirements/design はこの目的に整合させる。

## プロダクト

**農業IoTシステム** — ESP32 + SHT31（高精度温湿度センサー）で計測した温湿度を Go バックエンドで受信・蓄積し、ブラウザで可視化・アラート管理する。ハウス・施設栽培など小規模農業の環境モニタリングを主対象とする。

データは一方向パイプライン **デバイス →（HTTP + Bearer）→ API → PostgreSQL → templ レンダリング → ブラウザ** で流れる。フロント用 JSON API は持たず、グラフ含む全表示ロジックをサーバサイドに集約する（→ [tech.md](./tech.md)）。

## コア能力（網羅列挙ではなく主能力）

1. **センサーデータ受信・蓄積** — ESP32 から Bearer 認証付き REST で温湿度を受信し、デバイス単位で時系列保存。
2. **可視化** — サーバサイド生成 SVG グラフ（24h / 7d / 30d 期間切替）・最新計測値・履歴一覧（フィルタ / 集計 / ページング）。
3. **アラート** — デバイス × 指標(temperature/humidity) × 比較演算子(>/</>=/<=) × 閾値 のルールを管理し、データ受信時に同期判定して履歴化・通知する。
4. **デバイス管理** — 登録・編集・論理削除。
5. **マルチユーザー** — ユーザーごとにデバイス/データを分離する（所有者認可を死守。実装方針は [structure.md](./structure.md)）。

## 対象ユースケース

- 農業ハウス/施設の **温湿度の常時モニタリング**。
- **閾値超過の検知**（例: 温度 > 35℃ で発火）と、未対応アラートの一覧把握。
- 過去データからの **傾向把握**（期間別グラフ・集計）。

## 価値・設計の指針（迷ったらここに立ち返る判断基準）

- **言語統一**: デバイス（TinyGo 可）とバックエンドを Go に揃え、スタックの分断と開発者の認知負荷を下げる。本プロジェクトを新規構築する主動機。
- **SSR 集約**: templ + HTMX + Alpine.js で JS ビルドとフロント用 API を最小化。フロントは HTML モック作成のみ、実装ロジックは Go 側に集約する。
- **小規模・スピード重視**: ほぼ個人開発・全9画面規模。過剰な抽象化より cc-sdd の開発回転数を優先する（厳格 Clean 不採用＝実務的 Layered-lite）。**ただしセキュリティの要（所有者認可 = BOLA 防止）は集約して死守する**（→ [structure.md](./structure.md) 依存方向ルール）。
- **運用も小さく**: PostgreSQL + Go バイナリを AWS Lightsail + docker-compose で運用（k8s は IoT 規模に対し過剰として不採用）。

## スコープ外（明示）

- フロント用 JSON API / SPA（templ が HTML を直接返すため不要）。
- デバイスファームウェア本体（本リポジトリはバックエンド + Web UI）。
- 大規模マルチテナント・スケールアウト前提の構成。
